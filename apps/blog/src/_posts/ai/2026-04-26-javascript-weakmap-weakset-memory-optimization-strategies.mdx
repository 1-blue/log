## 🤔 문제 제기: JavaScript의 메모리 누수

JavaScript는 가비지 컬렉션(Garbage Collection) 덕분에 개발자가 직접 메모리를 관리할 필요가 거의 없어요. 하지만 '가비지 컬렉션'이 만능은 아니라는 점을 아셔야 해요. 잘못된 참조가 남아있으면 메모리 누수가 발생할 수 있거든요.<br />

### 0️⃣ 강한 참조(Strong Reference)와 가비지 컬렉션의 한계

대부분의 JavaScript 객체는 '강한 참조'를 통해 서로 연결되어 있어요. 어떤 객체를 참조하는 변수가 하나라도 존재하면, 가비지 컬렉터는 해당 객체를 '사용 중인 메모리'로 간주하고 해제하지 않아요. 문제는 이 참조가 더 이상 필요 없는데도 끊어지지 않고 남아있을 때 발생해요. 이런 경우 메모리 누수가 발생하고, 애플리케이션 성능 저하로 이어질 수 있어요.

예를 들어, `Map`이나 `Set` 같은 컬렉션은 키나 값으로 객체를 저장할 때 '강한 참조'를 유지해요. 컬렉션에서 해당 객체를 명시적으로 제거하지 않으면, 객체에 대한 외부 참조가 모두 사라져도 컬렉션 내부의 참조 때문에 가비지 컬렉션 대상에서 제외될 수 있어요.

```tsx showLineNumbers
const cache = new Map();

function expensiveOperation(obj: object) {
  if (cache.has(obj)) {
    console.log('캐시 히트!', obj);
    return cache.get(obj);
  }

  // 비용이 많이 드는 연산이라고 가정해요.
  const result = { data: obj.id * 2 };
  cache.set(obj, result);
  console.log('캐시 미스! 연산 후 저장', obj);
  return result;
}

let user1 = { id: 1, name: '블루' };
let user2 = { id: 2, name: '레드' };

expensiveOperation(user1);
expensiveOperation(user2);
expensiveOperation(user1);

// user1 객체에 대한 외부 참조를 null로 만들어요.
// 하지만 cache Map 내에서는 여전히 user1 객체를 강하게 참조하고 있어요.
user1 = null;

// 가비지 컬렉터가 동작해도 user1 객체는 cache Map 때문에 해제되지 않을 수 있어요.
// 실제 메모리 해제는 보장되지 않으므로, 개발자 도구의 메모리 탭에서 확인해 보세요.
// console.log(cache); // Map에 user1 객체가 여전히 남아있는 것을 볼 수 있어요.
```

위 코드에서 `user1 = null;`을 해도 `cache` Map 내부에 `user1` 객체가 여전히 남아있어 가비지 컬렉터가 해당 객체를 해제하지 못할 가능성이 있어요. 이렇게 되면 필요 없는 객체가 메모리를 계속 점유하게 되는 메모리 누수가 발생해요.<br />

### 1️⃣ 실무에서 흔히 겪는 메모리 누수 시나리오

이러한 강한 참조로 인한 메모리 누수는 실무에서 다양한 형태로 나타날 수 있어요.

*   **DOM 요소 참조:** JavaScript에서 DOM 요소를 변수에 저장하고, 나중에 해당 DOM 요소가 DOM 트리에서 제거되어도 변수 참조가 남아있으면 메모리 누수가 발생해요.<br />
*   **캐싱:** 위 예시처럼 `Map` 등을 사용하여 객체를 캐싱할 때, 캐시된 객체가 더 이상 사용되지 않는데도 캐시에서 제거되지 않으면 메모리 누수가 생겨요.<br />
*   **이벤트 리스너:** 객체에 이벤트 리스너를 등록하고 해당 객체가 사라질 때 리스너를 해제하지 않으면, 리스너가 객체를 참조하여 메모리 누수를 유발할 수 있어요.

이러한 문제를 해결하기 위해 등장한 것이 바로 `WeakMap`과 `WeakSet`이에요.

## 💡 WeakMap과 WeakSet의 등장

`WeakMap`과 `WeakSet`은 이름에서 알 수 있듯이 '약한 참조(Weak Reference)'를 사용하는 특별한 컬렉션이에요. 이 약한 참조 덕분에 일반 `Map`이나 `Set`과는 다른 방식으로 메모리를 관리할 수 있어요.<br />

### 0️⃣ 약한 참조(Weak Reference)의 개념

약한 참조는 가비지 컬렉터가 객체를 수집하는 것을 막지 않는 참조 방식이에요. 즉, 어떤 객체에 대한 모든 '강한 참조'가 사라지면, 해당 객체는 가비지 컬렉션의 대상이 되고, `WeakMap`이나 `WeakSet` 내부에 남아있는 '약한 참조'는 자동으로 제거돼요. 이는 메모리 누수를 방지하는 데 매우 효과적이에요.<br />

### 1️⃣ WeakMap 깊이 알아보기

`WeakMap`은 `Map`과 유사하게 키-값 쌍을 저장하지만, 다음과 같은 중요한 차이점이 있어요.

*   **키는 반드시 객체여야 해요:** 원시 값(문자열, 숫자, 심볼 등)은 키로 사용할 수 없어요. `WeakMap`의 약한 참조는 오직 객체에만 적용되기 때문이에요.<br />
*   **키에 대한 약한 참조를 유지해요:** `WeakMap`의 키는 약한 참조로 저장돼요. 만약 키로 사용된 객체에 대한 모든 강한 참조가 사라지면, 해당 객체는 가비지 컬렉션의 대상이 되고, `WeakMap`에서도 해당 키-값 쌍이 자동으로 제거돼요.<br />
*   **이터러블하지 않아요:** `WeakMap`은 `Map`처럼 `size`, `clear()`, `keys()`, `values()`, `entries()` 메서드를 제공하지 않아요. 약한 참조로 인해 가비지 컬렉터가 언제 키를 제거할지 예측할 수 없기 때문에, `WeakMap`의 내용을 반복하거나 크기를 정확히 알 수 없어요. 이는 `WeakMap`이 주로 객체와 데이터를 연결하는 '메타데이터 저장소'로 사용되는 이유예요.

**주요 메서드:**

*   `weakMap.set(key, value)`: 키-값 쌍을 저장해요. `key`는 반드시 객체여야 해요.<br />
*   `weakMap.get(key)`: `key`에 해당하는 값을 반환해요. 키가 없으면 `undefined`를 반환해요.<br />
*   `weakMap.has(key)`: `key`가 `WeakMap`에 존재하는지 여부를 `boolean`으로 반환해요.<br />
*   `weakMap.delete(key)`: `key`에 해당하는 키-값 쌍을 제거해요.

### 2️⃣ WeakSet 깊이 알아보기

`WeakSet`은 `Set`과 유사하게 고유한 객체들을 저장하지만, 역시 중요한 차이점이 있어요.

*   **값은 반드시 객체여야 해요:** `WeakSet`은 오직 객체만 저장할 수 있어요. 원시 값은 허용되지 않아요.<br />
*   **값에 대한 약한 참조를 유지해요:** `WeakSet`의 값은 약한 참조로 저장돼요. 저장된 객체에 대한 모든 강한 참조가 사라지면, 해당 객체는 가비지 컬렉션 대상이 되고, `WeakSet`에서도 자동으로 제거돼요.<br />
*   **이터러블하지 않아요:** `WeakSet`도 `WeakMap`과 마찬가지로 `size`, `clear()`, `keys()`, `values()`, `entries()` 메서드를 제공하지 않아요. 이는 `WeakSet`의 내용이나 크기를 반복하거나 예측할 수 없다는 의미예요.

**주요 메서드:**

*   `weakSet.add(value)`: `value`(객체)를 `WeakSet`에 추가해요.<br />
*   `weakSet.has(value)`: `value`가 `WeakSet`에 존재하는지 여부를 `boolean`으로 반환해요.<br />
*   `weakSet.delete(value)`: `value`를 `WeakSet`에서 제거해요.

## ⚙️ 실전 활용 예시

`WeakMap`과 `WeakSet`은 특정 상황에서 메모리 효율적인 코드를 작성하는 데 매우 유용해요. 몇 가지 실용적인 예시를 살펴볼게요.<br />

### 0️⃣ DOM 요소에 데이터 연결하기 (WeakMap)

웹 애플리케이션에서 DOM 요소에 추가적인 데이터를 연결해야 할 때가 많아요. 이때 일반 객체나 `Map`을 사용하면 DOM 요소가 화면에서 제거되어도 메모리에서 해제되지 않는 문제가 발생할 수 있어요. `WeakMap`은 이 문제를 우아하게 해결해 줘요.

```tsx showLineNumbers
interface ElementData {
  clicks: number;
  lastAccessed: Date;
}

// DOM 요소와 데이터를 연결할 WeakMap을 생성해요.
const elementDataMap = new WeakMap<HTMLElement, ElementData>();

function attachDataToElement(element: HTMLElement) {
  const data: ElementData = {
    clicks: 0,
    lastAccessed: new Date(),
  };
  elementDataMap.set(element, data);
  console.log(`[${element.id}] 데이터 초기화:`, data);
}

function getElementClicks(element: HTMLElement): number {
  const data = elementDataMap.get(element);
  if (data) {
    data.clicks++;
    data.lastAccessed = new Date();
    console.log(`[${element.id}] 클릭 수 증가:`, data.clicks);
    return data.clicks;
  }
  return 0;
}

// 예시 DOM 요소 생성
const div1 = document.createElement('div');
div1.id = 'div1';
document.body.appendChild(div1);

const div2 = document.createElement('div');
div2.id = 'div2';
document.body.appendChild(div2);

attachDataToElement(div1);
attachDataToElement(div2);

getElementClicks(div1);
getElementClicks(div1);
getElementClicks(div2);

// div1을 DOM에서 제거하고, 해당 요소에 대한 강한 참조를 없애요.
document.body.removeChild(div1);
// div1 = null; // 외부 참조를 null로 만들면 WeakMap에서 자동으로 제거될 확률이 높아져요.

// 잠시 후 가비지 컬렉터가 동작하면 div1과 연결된 데이터는 WeakMap에서 자동으로 사라져요.
// WeakMap은 이터러블하지 않으므로 직접 확인할 방법은 없지만, 메모리 효율이 향상돼요.

// 여전히 참조되는 div2의 데이터는 유효해요.
console.log('div2의 현재 데이터:', elementDataMap.get(div2));
```

이 예시에서는 `elementDataMap`이 `div1`과 `div2` 객체에 대한 약한 참조를 유지해요. `div1`이 DOM에서 제거되고 더 이상 강한 참조가 없으면, `div1` 객체는 가비지 컬렉션의 대상이 되고 `elementDataMap`에서도 `div1`과 관련된 데이터가 자동으로 정리돼요. 이렇게 하면 개발자가 일일이 메모리를 해제할 필요 없이 메모리 누수를 방지할 수 있어요.<br />

### 1️⃣ 객체 캐싱 전략 (WeakMap)

비용이 많이 드는 계산 결과를 캐싱할 때, `WeakMap`을 사용하면 캐시된 객체가 더 이상 필요 없어질 때 자동으로 정리되도록 할 수 있어요. 이는 위에서 언급했던 `Map`을 사용한 캐싱 예시의 메모리 누수 문제를 해결해 줘요.

```tsx showLineNumbers {2-3, 10, 15}
// WeakMap을 사용하여 캐시를 구현해요.
const objectCache = new WeakMap<object, any>();

function calculateExpensiveValue(obj: object): any {
  if (objectCache.has(obj)) {
    console.log('✨ 캐시 히트!', obj);
    return objectCache.get(obj);
  }

  // 실제로는 복잡하고 비용이 많이 드는 계산이라고 가정해요.
  const result = { value: Math.random() * 100, timestamp: new Date() };
  objectCache.set(obj, result);
  console.log('🚀 캐시 미스! 새로운 값 계산 및 저장', obj, result);
  return result;
}

let config1 = { id: 'config-A', settings: { theme: 'dark' } };
let config2 = { id: 'config-B', settings: { theme: 'light' } };

calculateExpensiveValue(config1);
calculateExpensiveValue(config2);
calculateExpensiveValue(config1);

// config1 객체에 대한 외부 참조를 제거해요.
config1 = null;

// 잠시 후 가비지 컬렉터가 동작하면 config1 객체는 가비지 컬렉션 대상이 되고,
// objectCache에서 config1과 연결된 캐시 데이터도 자동으로 제거돼요.

// config2는 여전히 참조되므로 캐시가 유효해요.
calculateExpensiveValue(config2);

// WeakMap은 이터러블하지 않아 직접 내용 확인은 어렵지만,
// 메모리 효율이 Map보다 훨씬 우수해요.
```

이 예시에서 `objectCache`는 `config1`과 `config2` 객체에 대한 약한 참조를 유지해요. `config1 = null;`을 통해 `config1` 객체에 대한 강한 참조가 모두 사라지면, 가비지 컬렉터가 `config1`을 해제하고 `objectCache`에서도 관련 캐시 데이터가 자동으로 정리돼요. 이는 메모리 사용량을 효율적으로 관리하는 데 큰 도움이 돼요.<br />

### 2️⃣ 객체 존재 여부 추적 (WeakSet)

`WeakSet`은 특정 객체가 현재 '활성 상태'인지 또는 '처리된 상태'인지를 추적할 때 유용하게 사용될 수 있어요. 예를 들어, 특정 작업을 수행해야 하는 객체 목록을 관리하거나, 이미 처리된 객체를 표시하는 용도로 활용할 수 있어요.

```tsx showLineNumbers {2, 7, 13, 19, 23}
// 처리된 객체를 추적하는 WeakSet을 생성해요.
const processedObjects = new WeakSet<object>();

function processObject(obj: object) {
  if (processedObjects.has(obj)) {
    console.log('✅ 이미 처리된 객체예요:', obj);
    return;
  }

  console.log('🚀 객체를 처리 중...', obj);
  // 여기에 비용이 많이 드는 실제 처리 로직을 구현해요.
  processedObjects.add(obj);
  console.log('🎉 객체 처리 완료 및 WeakSet에 추가:', obj);
}

let item1 = { id: 'item-A' };
let item2 = { id: 'item-B' };
let item3 = { id: 'item-C' };

processObject(item1);
processObject(item2);
processObject(item1); // 다시 호출해도 이미 처리되었다고 나와요.

// item2에 대한 외부 참조를 제거해요.
item2 = null;

// 잠시 후 가비지 컬렉터가 동작하면 item2 객체는 가비지 컬렉션 대상이 되고,
// processedObjects에서 item2도 자동으로 제거돼요.
// WeakSet은 이터러블하지 않으므로 직접 확인은 어렵지만, 메모리 효율성이 좋아요.

processObject(item3);
```

이 예시에서 `processedObjects`는 `item1`, `item2`, `item3` 객체에 대한 약한 참조를 유지해요. `item2 = null;`을 통해 `item2`에 대한 강한 참조가 사라지면, `item2`는 가비지 컬렉션 대상이 되고 `processedObjects`에서도 자동으로 제거돼요. 이렇게 하면 불필요하게 메모리를 점유하는 객체를 자동으로 정리할 수 있어요.<br />

## 📝 WeakMap과 WeakSet 사용 시 주의사항

`WeakMap`과 `WeakSet`은 강력한 도구이지만, 몇 가지 제약사항과 주의할 점이 있어요.

*   **키/값은 반드시 객체여야 해요:** 앞에서 강조했듯이, 원시 값은 키(`WeakMap`)나 값(`WeakSet`)으로 사용할 수 없어요. 이는 약한 참조가 객체에만 적용되기 때문이에요.<br />
*   **이터러블하지 않아요:** `WeakMap`과 `WeakSet`은 `size` 속성이나 `keys()`, `values()`, `entries()` 같은 메서드를 제공하지 않아요. 따라서 컬렉션의 모든 요소를 반복하거나 현재 크기를 정확히 알 수 없어요. 이는 가비지 컬렉션이 언제 일어날지 예측할 수 없기 때문이에요. 이 때문에 `WeakMap`과 `WeakSet`은 주로 '메타데이터 저장소'나 '객체 존재 여부 트래킹'과 같이 특정 객체에 정보를 연결하거나 추적하는 용도로 사용돼요.<br />
*   **가비지 컬렉션 시점 예측 불가능:** 약한 참조가 언제 가비지 컬렉션될지는 정확히 예측할 수 없어요. 이는 JavaScript 엔진의 내부 구현과 현재 메모리 상태에 따라 달라져요. 따라서 `WeakMap`이나 `WeakSet`에 특정 키/값이 존재할 것이라고 맹목적으로 가정하면 안 돼요.<br />
*   **메모리 누수를 완전히 막아주지는 않아요:** `WeakMap`과 `WeakSet`은 '약한 참조'를 통해 특정 상황에서의 메모리 누수를 방지하지만, 모든 종류의 메모리 누수를 해결해 주지는 않아요. 예를 들어, 콜백 함수 내에서 외부 스코프의 변수를 강하게 참조하는 클로저로 인한 누수는 여전히 발생할 수 있어요. 항상 코드 전체의 참조 관계를 신중하게 고려해야 해요.<br />

<Blockquote type="tip">
**언제 `Map`/`Set`을 사용하고, 언제 `WeakMap`/`WeakSet`을 사용할까요?**<br />

*   **`Map`/`Set`:** 컬렉션에 저장된 모든 키/값 쌍을 명시적으로 관리하고, 반복하거나 크기를 알아야 할 때 사용해요. 키/값으로 원시 값을 포함한 모든 타입을 사용할 수 있어요.<br />
*   **`WeakMap`/`WeakSet`:** 특정 객체에 데이터를 연결하거나 객체의 존재 여부를 추적해야 하지만, 해당 객체가 더 이상 사용되지 않을 때 자동으로 메모리에서 해제되기를 원할 때 사용해요. 키/값은 반드시 객체여야 해요.
</Blockquote>

## 📝 정리

오늘은 JavaScript의 `WeakMap`과 `WeakSet`에 대해 자세히 알아보았어요. 이 특별한 컬렉션은 '약한 참조'라는 개념을 통해 메모리 누수를 방지하고 애플리케이션의 성능을 최적화하는 데 큰 도움을 줘요.<br />

*   **`WeakMap`**은 키가 객체인 키-값 쌍을 저장하며, 키 객체에 대한 강한 참조가 사라지면 자동으로 해당 엔트리가 제거돼요. 주로 DOM 요소에 데이터 연결, 객체 기반 캐싱 등에 활용돼요.<br />
*   **`WeakSet`**은 객체만을 값으로 저장하며, 값 객체에 대한 강한 참조가 사라지면 자동으로 해당 값이 제거돼요. 주로 객체의 존재 여부 추적, 이벤트 리스너 관리 등에 활용될 수 있어요.<br />

`WeakMap`과 `WeakSet`은 이터러블하지 않고 가비지 컬렉션 시점을 예측할 수 없다는 제약이 있지만, 이러한 특성을 이해하고 적절한 상황에 활용한다면 훨씬 더 효율적이고 안정적인 JavaScript 애플리케이션을 만들 수 있을 거예요.<br />

여러분의 프로젝트에서 메모리 관련 문제가 발생한다면, `WeakMap`과 `WeakSet`을 적극적으로 고려해 보시길 바라요! 다음에도 실무에 도움이 되는 유익한 정보로 찾아올게요.<br />
