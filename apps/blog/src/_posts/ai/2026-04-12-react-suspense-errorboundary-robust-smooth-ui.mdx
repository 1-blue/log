## 🤔 문제/배경: 로딩과 에러, 사용자 경험을 해치는 주범

우리가 만드는 웹 애플리케이션은 필연적으로 네트워크 요청과 비동기 데이터 처리를 수반해요. 그리고 그 과정에서 예기치 않은 에러가 발생하기도 하죠. 이런 상황들을 제대로 처리하지 못하면 사용자 경험은 크게 저하될 수 있어요.

### 0️⃣ 기존 로딩 처리의 한계: ‘스피너 지옥’과 복잡한 상태 관리

기존에는 데이터를 불러올 때 `isLoading`과 같은 상태 변수를 컴포넌트마다 관리하며 로딩 스피너를 보여주는 것이 일반적이었어요. 하지만 이 방식은 몇 가지 한계를 가지고 있어요.<br />

*   **스피너 지옥 (Spinner Hell):** 여러 컴포넌트에서 동시에 데이터를 불러올 때, 각기 다른 스피너가 덕지덕지 붙어 화면이 지저분해 보일 수 있어요.<br />
*   **로딩 상태 관리의 복잡성:** `isLoading` 상태를 부모 컴포넌트에서 자식 컴포넌트로 전달하거나, 전역 상태 관리 라이브러리를 통해 관리해야 하는 경우가 많아 코드가 복잡해지고 유지보수가 어려워져요.<br />
*   **데이터 페칭 로직 분산:** 데이터 페칭 로직이 `useEffect` 안에 있거나, 각 컴포넌트의 라이프사이클과 엮여 있어 재사용성과 응집성이 떨어지기 쉬워요.

### 1️⃣ 기존 에러 처리의 한계: 화면 전체 크래시와 일관성 없는 UI

JavaScript 애플리케이션에서 에러는 언제든 발생할 수 있는 일이에요. 특히 React 컴포넌트 렌더링 과정에서 발생하는 에러는 애플리케이션 전체를 멈추게 만들 수 있어요.<br />

*   **화면 전체 크래시:** 자식 컴포넌트에서 발생한 렌더링 에러가 부모 컴포넌트로 전파되어 애플리케이션 전체가 멈추고 빈 화면을 보여주는 경우가 흔했어요.<br />
*   **일관성 없는 에러 UI:** 각 컴포넌트에서 `try-catch` 등으로 에러를 개별적으로 처리하다 보면, 에러 메시지나 UI 스타일이 일관되지 않아 사용자에게 혼란을 줄 수 있어요.<br />
*   **에러 로깅의 어려움:** 에러가 발생했을 때 이를 중앙에서 수집하고 로깅하는 메커니즘이 없으면 문제 진단과 해결이 어려워져요.

## ⚙️ 해결 방법: Suspense와 ErrorBoundary의 시너지

React는 이러한 문제들을 해결하기 위해 **Suspense**와 **ErrorBoundary**라는 강력한 도구를 제공해요. 이 둘은 함께 사용될 때 진정한 시너지를 발휘하며, 애플리케이션의 견고성과 사용자 경험을 크게 향상시킬 수 있답니다.

### 0️⃣ React Suspense로 비동기 데이터 페칭과 로딩 UI 통합하기

**React Suspense**는 컴포넌트가 데이터를 불러오는 동안 렌더링을 일시 중지하고, 데이터 준비가 완료될 때까지 지정된 `fallback` UI를 보여주는 메커니즘이에요. React 18부터는 `React.lazy`와 함께 코드 스플리팅뿐만 아니라 데이터 페칭에도 정식으로 활용될 수 있게 되었어요.<br />

<Blockquote type="info">
**어떻게 동작하나요?**<br />
Suspense는 컴포넌트 내부에서 비동기 작업(데이터 페칭 Promise)이 발생하여 렌더링을 ‘일시 중단’(suspend)시키면, 가장 가까운 부모 Suspense 컴포넌트가 그 작업을 감지하고 `fallback`으로 지정된 UI를 대신 렌더링해요. Promise가 해결되면 다시 Suspense 내부의 컴포넌트 렌더링을 재개한답니다.
</Blockquote>

**주요 이점:**<br />
*   **선언적인 로딩 상태 관리:** `isLoading` 상태를 직접 관리할 필요 없이, `Suspense`로 감싸기만 하면 비동기 작업이 끝날 때까지 자동으로 `fallback` UI를 보여줘요.<br />
*   **로딩 UI의 중앙 집중화:** 여러 컴포넌트의 로딩 상태를 한 곳에서 관리할 수 있어 '스피너 지옥'을 방지하고 일관된 사용자 경험을 제공할 수 있어요.<br />
*   **데이터 페칭 로직 분리:** 컴포넌트 렌더링 로직과 데이터 페칭 로직을 더욱 명확하게 분리할 수 있게 도와줘요. `react-query`, `swr` 같은 라이브러리들이 Suspense를 지원하며 더욱 편리한 데이터 페칭 경험을 제공하고 있어요.

### 1️⃣ React ErrorBoundary로 컴포넌트 에러 격리하기

**ErrorBoundary**는 React 컴포넌트 트리에서 자식 컴포넌트들이 던진 JavaScript 에러를 잡아내고, 에러가 발생했을 때 지정된 `fallback` UI를 렌더링하는 React 컴포넌트예요. 마치 `try-catch` 블록이 JavaScript 코드에서 에러를 잡는 것처럼, ErrorBoundary는 React 컴포넌트 트리에서 에러를 잡는 역할을 해요.<br />

<Blockquote type="warning">
**주의할 점:** ErrorBoundary는 **렌더링 단계**에서 발생하는 에러(예: `render` 메서드 안에서 에러 발생)만 잡아낼 수 있어요. 이벤트 핸들러 내부나 비동기 코드(Promise `catch` 블록 외부)에서 발생하는 에러는 잡아내지 못해요. 이러한 에러는 일반적인 `try-catch`나 `window.onerror` 등의 전역 에러 핸들링으로 처리해야 한답니다.
</Blockquote>

**주요 이점:**<br />
*   **부분적인 에러 처리:** 자식 컴포넌트에서 에러가 발생해도 애플리케이션 전체가 멈추지 않고, 에러가 발생한 부분만 `fallback` UI로 대체할 수 있어요.<br />
*   **일관된 에러 UI:** 모든 에러를 ErrorBoundary를 통해 처리함으로써 애플리케이션 전반에 걸쳐 일관된 에러 메시지와 디자인을 유지할 수 있어요.<br />
*   **에러 로깅:** `componentDidCatch` 라이프사이클 메서드를 통해 에러 정보를 수집하고 Sentry 같은 에러 로깅 서비스로 전송하여 문제 진단에 활용할 수 있어요.

## 🧪 실전 예시: 데이터 페칭과 에러 처리의 우아한 구현

이제 Suspense와 ErrorBoundary를 실제로 어떻게 구현하고 활용하는지 코드 예시를 통해 살펴볼까요? 여기서는 간단한 데이터 페칭 상황과 에러 발생 상황을 가정해볼게요.

### 0️⃣ Suspense를 활용한 비동기 데이터 로딩

Suspense와 함께 사용하기 위해서는 Promise를 던질 수 있는 데이터 페칭 로직이 필요해요. `react-query`나 `swr` 같은 라이브러리를 사용하면 이 부분이 훨씬 편리하지만, 여기서는 개념 이해를 돕기 위해 간단한 헬퍼 함수를 만들어볼게요.

```tsx showLineNumbers
// utils/createSuspenseResource.ts
interface Resource<T> {
  read(): T;
}

function createSuspenseResource<T>(promise: Promise<T>): Resource<T> {
  let status = "pending";
  let result: T | Error;

  const suspender = promise.then(
    (r) => {
      status = "success";
      result = r;
    },
    (e) => {
      status = "error";
      result = e;
    }
  );

  return {
    read(): T {
      if (status === "pending") {
        throw suspender; // Promise를 던져 Suspense가 잡도록 함
      }
      if (status === "error") {
        throw result; // ErrorBoundary가 잡도록 에러를 던짐
      }
      return result as T;
    },
  };
}

// 가상의 데이터 페칭 함수
const fetchUserData = () =>
  new Promise<{
    id: number;
    name: string;
    email: string;
  }>((resolve) => {
    setTimeout(() => {
      resolve({
        id: 1,
        name: "블루",
        email: "blue@example.com",
      });
    }, 2000);
  });

// 데이터를 Suspense Resource로 래핑
export const userResource = createSuspenseResource(fetchUserData());

// components/UserProfile.tsx
import React from "react";
import { userResource } from "../utils/createSuspenseResource";

function UserProfile() {
  const user = userResource.read(); // 데이터가 준비될 때까지 Suspense가 기다림

  return (
    <div className="user-profile">
      <h3>✨ 사용자 프로필</h3>
      <p>이름: {user.name}</p>
      <p>이메일: {user.email}</p>
    </div>
  );
}

export default UserProfile;

// components/LoadingSpinner.tsx
import React from "react";

function LoadingSpinner() {
  return (
    <div className="loading-spinner">
      <p>⏳ 데이터를 불러오는 중이에요...</p>
    </div>
  );
}

export default LoadingSpinner;

// App.tsx 또는 부모 컴포넌트
import React, { Suspense } from "react";
import UserProfile from "./components/UserProfile";
import LoadingSpinner from "./components/LoadingSpinner";

function App() {
  return (
    <div className="app">
      <h1>🚀 Suspense 예시</h1>
      <Suspense fallback={<LoadingSpinner />}>
        <UserProfile />
      </Suspense>
    </div>
  );
}

export default App;
```

위 코드에서 `UserProfile` 컴포넌트는 `userResource.read()`를 호출하고, 데이터가 준비되지 않았다면 `createSuspenseResource` 내부에서 던져진 Promise를 가장 가까운 `Suspense` 컴포넌트가 잡아 `LoadingSpinner`를 보여주게 돼요. 데이터 로딩이 완료되면 `UserProfile`이 렌더링된답니다.

### 1️⃣ ErrorBoundary 컴포넌트 구현

ErrorBoundary는 클래스 컴포넌트로 구현해야 하며, `static getDerivedStateFromError`와 `componentDidCatch` 라이프사이클 메서드를 포함해야 해요.

```tsx showLineNumbers
// components/ErrorBoundary.tsx
import React, { Component, ErrorInfo, ReactNode } from "react";

interface Props {
  children: ReactNode;
  fallback: ReactNode;
}

interface State {
  hasError: boolean;
}

class ErrorBoundary extends Component<Props, State> {
  public state: State = {
    hasError: false,
  };

  // 에러가 발생했을 때 호출되며, 다음 렌더링에서 fallback UI를 보여주기 위해
  // 상태를 업데이트해요.
  public static getDerivedStateFromError(_: Error): State {
    return { hasError: true };
  }

  // 에러와 에러 정보를 로깅할 수 있어요.
  public componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    console.error("🚨 Error caught by ErrorBoundary:", error, errorInfo);
    // 실제 프로덕션에서는 Sentry와 같은 에러 로깅 서비스로 전송해요.
    // logErrorToService(error, errorInfo);
  }

  public render() {
    if (this.state.hasError) {
      // fallback UI를 렌더링해요.
      return this.props.fallback;
    }

    return this.props.children;
  }
}

export default ErrorBoundary;

// components/BuggyComponent.tsx
import React, { useState } from "react";

function BuggyComponent() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount((prev) => prev + 1);
  };

  if (count > 3) {
    // 렌더링 과정에서 에러를 강제로 발생시켜요.
    throw new Error("🔥 의도적으로 발생시킨 에러예요!");
  }

  return (
    <div className="buggy-component">
      <p>카운트: {count}</p>
      <button onClick={handleClick}>카운트 증가</button>
      <p>카운트가 3을 초과하면 에러가 발생해요.</p>
    </div>
  );
}

export default BuggyComponent;

// components/ErrorFallback.tsx
import React from "react";

function ErrorFallback({
  error,
  resetErrorBoundary,
}: {
  error?: Error;
  resetErrorBoundary?: () => void;
}) {
  return (
    <div role="alert" className="error-fallback">
      <h3>⚠️ 무언가 잘못되었어요!</h3>
      <p>에러 메시지: {error?.message}</p>
      {resetErrorBoundary && (
        <button onClick={resetErrorBoundary}>다시 시도</button>
      )}
    </div>
  );
}

export default ErrorFallback;

// App.tsx 또는 부모 컴포넌트
import React from "react";
import ErrorBoundary from "./components/ErrorBoundary";
import BuggyComponent from "./components/BuggyComponent";
import ErrorFallback from "./components/ErrorFallback";

function App() {
  return (
    <div className="app">
      <h1>💥 ErrorBoundary 예시</h1>
      <ErrorBoundary fallback={<ErrorFallback />}>
        <BuggyComponent />
      </ErrorBoundary>
    </div>
  );
}

export default App;
```

`BuggyComponent`는 `count`가 3을 초과하면 에러를 던지도록 만들었어요. 이 에러는 가장 가까운 `ErrorBoundary`가 잡아내어 `ErrorFallback` 컴포넌트가 대신 렌더링된답니다. 애플리케이션 전체가 멈추지 않고 에러가 발생한 부분만 격리되어 처리되는 것을 볼 수 있어요.

### 2️⃣ Suspense와 ErrorBoundary 함께 사용하기

이제 이 두 기능을 조합하여 더욱 견고한 UI를 만들어볼까요? 예를 들어, 데이터를 불러오는 과정에서 에러가 발생할 수도 있고, 데이터 로딩 중에는 로딩 스피너를 보여줘야 하는 상황이 흔해요.

```tsx showLineNumbers
// App.tsx (통합 예시)
import React, { Suspense } from "react";
import ErrorBoundary from "./components/ErrorBoundary";
import UserProfile from "./components/UserProfile";
import LoadingSpinner from "./components/LoadingSpinner";
import BuggyComponent from "./components/BuggyComponent";
import ErrorFallback from "./components/ErrorFallback";

// userResource는 앞서 정의한 createSuspenseResource를 사용한다고 가정해요.
// import { userResource } from "./utils/createSuspenseResource";

// 가상의 에러 발생 데이터 페칭 함수 (UserProfile에서 사용될 수 있음)
const fetchUserDataWithError = () =>
  new Promise<{
    id: number;
    name: string;
    email: string;
  }>((resolve, reject) => {
    setTimeout(() => {
      const shouldError = Math.random() > 0.5; // 50% 확률로 에러 발생
      if (shouldError) {
        reject(new Error("네트워크 요청 중 에러가 발생했어요!"));
      } else {
        resolve({
          id: 2,
          name: "블루2",
          email: "blue2@example.com",
        });
      }
    }, 2500);
  });

// userResource를 에러 발생 가능성이 있는 버전으로 교체 (실제 코드에서는 분리 관리)

function App() {
  return (
    <div className="app-container">
      <h1>✨ Suspense & ErrorBoundary 통합 예시 ✨</h1>

      <section className="section">
        <h2>📚 데이터 로딩과 에러 처리</h2>
        <ErrorBoundary fallback={<ErrorFallback error={new Error("데이터 로딩 실패!")} />}>
          <Suspense fallback={<LoadingSpinner />}>
            <UserProfile />
          </Suspense>
        </ErrorBoundary>
      </section>

      <section className="section">
        <h2>🐛 컴포넌트 에러 격리</h2>
        <ErrorBoundary fallback={<ErrorFallback error={new Error("컴포넌트 렌더링 실패!")} />}>
          <BuggyComponent />
        </ErrorBoundary>
      </section>

      <section className="section">
        <h2>✅ 정상적인 다른 컴포넌트</h2>
        <div>
          <p>이 부분은 에러나 로딩에 영향을 받지 않고 정상적으로 렌더링돼요.</p>
          <button onClick={() => console.log("정상 동작!")}>클릭</button>
        </div>
      </section>
    </div>
  );
}

export default App;
```

이 예시에서는 `UserProfile` 컴포넌트를 `Suspense`로 감싸 로딩 상태를 처리하고, 다시 이 `Suspense`와 `BuggyComponent`를 각각 `ErrorBoundary`로 감싸서 에러를 격리했어요. 이렇게 하면:<br />

*   `UserProfile`이 데이터를 불러오는 동안에는 `LoadingSpinner`가 보이고,<br />
*   데이터 페칭 중 에러가 발생하면 `UserProfile`을 감싼 `ErrorBoundary`의 `ErrorFallback`이 나타나고,<br />
*   `BuggyComponent`에서 렌더링 에러가 발생하면 해당 컴포넌트를 감싼 `ErrorBoundary`의 `ErrorFallback`이 나타나요.

각각의 에러와 로딩 상태가 서로에게 영향을 주지 않고 독립적으로 처리되면서, 사용자에게는 훨씬 더 안정적이고 부분적인 피드백을 제공하는 UI를 경험하게 된답니다.

## 📝 정리: 견고한 UI를 향한 첫걸음

React Suspense와 ErrorBoundary는 현대적인 React 애플리케이션에서 사용자 경험을 크게 향상시킬 수 있는 필수적인 도구예요. 이 둘을 적절히 활용하면 '스피너 지옥'과 '화면 전체 크래시'와 같은 고질적인 문제들을 우아하게 해결할 수 있어요.

### 0️⃣ 핵심 요약

*   **Suspense:** 비동기 데이터 로딩 중 렌더링을 일시 중단하고 `fallback` UI를 보여줌으로써, 선언적이고 중앙 집중화된 로딩 상태 관리를 가능하게 해요.<br />
*   **ErrorBoundary:** React 컴포넌트 트리 내에서 발생하는 렌더링 에러를 잡아내고 `fallback` UI를 렌더링하여, 애플리케이션 전체 크래시를 방지하고 에러를 격리하며 일관된 에러 UI를 제공해요.
*   **시너지 효과:** Suspense와 ErrorBoundary를 함께 사용하면 데이터 로딩 중 발생할 수 있는 에러까지 효과적으로 처리하여, 사용자에게 더욱 견고하고 부드러운 UI 경험을 제공할 수 있어요.

### 1️⃣ 다음 액션

기존 프로젝트에 Suspense와 ErrorBoundary를 한 번에 적용하기 어렵다면, 작은 단위부터 점진적으로 도입해 보세요. 예를 들어, 특정 대시보드 위젯이나 사용자 프로필 컴포넌트부터 적용해 보면서 그 효과를 직접 경험해 보는 것이 좋아요.<br />

또한, `react-query`, `swr` 같은 데이터 페칭 라이브러리들은 Suspense를 완벽하게 지원하고 ErrorBoundary와도 잘 연동되니, 이들을 함께 사용하면 훨씬 더 효율적으로 견고한 UI를 구축할 수 있답니다.

사용자에게 최고의 경험을 제공하기 위한 여정에 Suspense와 ErrorBoundary가 든든한 동반자가 되어줄 거예요. Happy Coding! 👋
