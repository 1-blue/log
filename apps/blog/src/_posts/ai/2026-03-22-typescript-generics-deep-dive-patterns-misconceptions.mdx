## 🧐 제네릭, 왜 필요할까요?

### 0️⃣ 제네릭의 본질과 등장 배경

우리가 함수나 클래스를 만들 때, 특정 타입에만 동작하도록 만들면 재사용성이 떨어져요. 그렇다고 `any` 타입을 남발하면 타입스크립트를 사용하는 의미가 없어지고요.<br />
이런 딜레마를 해결하기 위해 등장한 것이 바로 '제네릭'이에요. 제네릭은 함수, 클래스, 인터페이스 등의 정의 시점에 타입을 고정하지 않고, 사용하는 시점에 타입을 명시할 수 있도록 해주는 기능이에요. 마치 함수의 매개변수처럼 타입을 전달한다고 생각하시면 이해하기 쉬울 거예요.

<Blockquote type="info">
  제네릭은 '일반적인' 또는 '포괄적인'이라는 의미를 가지고 있어요. 특정 타입에
  얽매이지 않고 다양한 타입에서 동작할 수 있도록 코드를 유연하게 만들어주는
  역할을 하죠.
</Blockquote>

### 1️⃣ `any`와 제네릭의 차이점

`any` 타입을 사용하면 어떤 타입이든 받을 수 있지만, 타입 정보가 사라져서 컴파일 시점에 오류를 잡아낼 수 없게 돼요. 반면에 제네릭은 사용하는 시점에 타입을 전달받아 해당 타입을 '기억'하고, 그 타입에 대한 안정성을 보장해 줘요. 이것이 `any`와 제네릭의 가장 큰 차이점이며, 제네릭을 사용해야 하는 핵심 이유예요.

```tsx showLineNumbers
// any를 사용한 예시: 타입 안정성 상실
function logAny(arg: any): any {
  console.log(arg);
  return arg;
}

const anyResult = logAny("hello");
anyResult.length; // 런타임 에러 가능성 (anyResult가 문자열이 아닐 경우)

// 제네릭을 사용한 예시: 타입 안정성 유지
function logGeneric<T>(arg: T): T {
  console.log(arg);
  return arg;
}

const stringResult = logGeneric<string>("hello");
stringResult.length; // OK, string 타입임을 알고 있어요.

const numberResult = logGeneric<number>(123);
numberResult.toFixed(); // OK, number 타입임을 알고 있어요.
// numberResult.length; // 컴파일 에러: number 타입에는 length 속성이 없어요.
```

위 예시에서 `logAny`는 `anyResult`의 타입이 `any`이므로, `length` 속성에 접근해도 컴파일 에러가 발생하지 않아요. 하지만 `numberResult.length`는 `number` 타입에 `length` 속성이 없음을 타입스크립트가 인지하고 컴파일 에러를 발생시켜 안전한 코드를 작성하도록 도와주죠. 이것이 바로 제네릭의 힘이에요.

## 🛠️ 제네릭 기본 사용법 익히기

### 0️⃣ 함수에서 제네릭 사용하기

가장 기본적인 제네릭 사용법은 함수에서 타입을 추론하거나 명시하는 방식이에요. `<T>`와 같은 형태로 타입 변수를 선언하고, 이를 함수의 매개변수 타입이나 반환 타입으로 활용해요.

```tsx showLineNumbers
// 타입 변수 T는 함수의 인자로 들어오는 값의 타입을 의미해요.
function identity<T>(arg: T): T {
  return arg;
}

// 1. 타입 인자를 명시적으로 전달
let output1 = identity<string>("myString"); // output1의 타입은 string
console.log(output1); // "myString"

// 2. 타입 인자를 생략하고 타입 추론에 맡기기 (대부분의 경우 가능)
let output2 = identity(123); // output2의 타입은 number
console.log(output2); // 123

let output3 = identity(true); // output3의 타입은 boolean
console.log(output3); // true
```

함수에서 제네릭을 사용할 때는 보통 타입스크립트가 타입을 잘 추론해 주기 때문에 명시적으로 `<string>`과 같이 전달할 필요가 없는 경우가 많아요. 하지만 복잡한 상황이거나, 명확성을 위해 타입을 명시하는 것이 좋을 때도 있어요.

### 1️⃣ 인터페이스와 타입 별칭에서 제네릭 사용하기

데이터 구조를 정의하는 인터페이스나 타입 별칭에서도 제네릭을 활용하여 유연한 타입을 만들 수 있어요. 이는 다양한 타입의 데이터를 담을 수 있는 컨테이너 역할을 할 때 유용해요.

```tsx showLineNumbers
// 제네릭 인터페이스
interface Box<T> {
  value: T;
}

const stringBox: Box<string> = { value: "Hello Generics" };
console.log(stringBox.value); // "Hello Generics"

const numberBox: Box<number> = { value: 12345 };
console.log(numberBox.value); // 12345

// 제네릭 타입 별칭
type Pair<K, V> = {
  key: K;
  value: V;
};

const userSetting: Pair<string, boolean> = { key: "darkMode", value: true };
console.log(userSetting.key, userSetting.value); // "darkMode" true

const itemPrice: Pair<number, string> = { key: 1001, value: "$99.99" };
console.log(itemPrice.key, itemPrice.value); // 1001 "$99.99"
```

`Box<T>` 인터페이스는 어떤 타입이든 `value` 속성으로 가질 수 있도록 해주고요, `Pair<K, V>` 타입 별칭은 두 가지 다른 제네릭 타입을 받아서 key-value 쌍을 유연하게 정의할 수 있도록 해줘요. 이처럼 제네릭을 사용하면 타입의 재활용성을 극대화할 수 있어요.

### 2️⃣ 클래스에서 제네릭 사용하기

클래스에서도 제네릭을 사용하여 인스턴스화될 때 다양한 타입으로 동작하는 클래스를 만들 수 있어요. 특히 컬렉션(Collection)이나 자료 구조를 구현할 때 매우 유용하죠.

```tsx showLineNumbers
class GenericList<T> {
  private items: T[] = [];

  addItem(item: T): void {
    this.items.push(item);
  }

  getItem(index: number): T | undefined {
    return this.items[index];
  }

  getAllItems(): T[] {
    return this.items;
  }
}

const stringList = new GenericList<string>();
stringList.addItem("Apple");
stringList.addItem("Banana");
console.log(stringList.getAllItems()); // ["Apple", "Banana"]
// stringList.addItem(123); // 컴파일 에러: number 타입은 stringList에 추가할 수 없어요.

const numberList = new GenericList<number>();
numberList.addItem(10);
numberList.addItem(20);
console.log(numberList.getItem(0)); // 10
```

`GenericList<T>` 클래스는 어떤 타입이든 저장할 수 있는 리스트를 구현해요. `stringList`는 문자열만, `numberList`는 숫자만 저장할 수 있도록 타입 안정성을 보장하면서도 코드는 재사용할 수 있게 되죠.

## 💡 제네릭 제약 조건 (Constraints)

때로는 제네릭 타입이 특정 조건을 만족해야 할 때가 있어요. 예를 들어, 제네릭으로 받은 인자가 `length` 속성을 가지고 있다고 가정하고 싶을 때가 그렇죠. 이때 `extends` 키워드를 사용하여 제네릭 제약 조건을 추가할 수 있어요.

```tsx showLineNumbers {1, 4}
interface Lengthwise {
  length: number;
}

function logLength<T extends Lengthwise>(arg: T): T {
  console.log(arg.length); // 이제 T는 length 속성을 가지고 있음이 보장돼요.
  return arg;
}

logLength("hello"); // OK, 문자열은 length 속성을 가지고 있어요.
logLength([1, 2, 3]); // OK, 배열도 length 속성을 가지고 있어요.
// logLength(10); // 컴파일 에러: number 타입은 length 속성을 가지고 있지 않아요.
```

`T extends Lengthwise`는 `T`가 반드시 `Lengthwise` 인터페이스를 구현하거나, `Lengthwise` 인터페이스와 호환되는 형태여야 함을 의미해요. 이를 통해 제네릭 타입 `T`가 특정 속성이나 메서드를 가지고 있음을 보장하고, 더 안전하게 해당 속성에 접근할 수 있게 돼요.

## 🤯 제네릭과 관련된 흔한 오해와 해결책

### 0️⃣ 제네릭 기본값 설정 (Default Generic Types)

타입스크립트 2.3부터 제네릭 타입에 기본값을 설정할 수 있게 되었어요. 이는 제네릭 타입을 명시하지 않았을 때 특정 기본 타입이 사용되도록 하여, 유연성을 더 높여줘요.

```tsx showLineNumbers {1}
interface FetchResult<T = any> {
  data: T;
  status: number;
  message?: string;
}

const successResult: FetchResult = {
  data: { id: 1, name: "Test" },
  status: 200,
};
// T를 명시하지 않아도 기본값인 any가 적용되어 에러가 발생하지 않아요.
console.log(successResult.data.id);

const errorResult: FetchResult<string> = {
  data: "Error occurred",
  status: 500,
};
// string 타입을 명시하면 T는 string이 돼요.
console.log(errorResult.data.toUpperCase());
```

`FetchResult<T = any>`처럼 `T`에 `any`를 기본값으로 설정하면, 제네릭 타입을 명시하지 않았을 때 `any`로 동작해요. 이는 기존 코드와의 호환성을 유지하거나, 특정 상황에서 타입 추론이 어려운 경우 유용하게 사용될 수 있어요. 하지만 `any` 사용은 최소화하는 것이 좋다는 점을 항상 기억해 주세요.

### 1️⃣ 여러 개의 제네릭 타입과 제약 조건

하나의 함수나 인터페이스에 여러 개의 제네릭 타입을 사용할 수도 있어요. 각 제네릭 타입은 독립적인 제약 조건을 가질 수 있고요. 이는 복잡한 데이터 구조나 함수 시그니처를 정의할 때 유용해요.

```tsx showLineNumbers {1, 7}
// K는 string 또는 number를 확장해야 하고, V는 어떤 타입이든 가능해요.
function createKeyValue<K extends string | number, V>(
  key: K,
  value: V,
): { key: K; value: V } {
  return { key, value };
}

const myKeyValue = createKeyValue("name", "블루");
console.log(myKeyValue.key, myKeyValue.value); // "name" "블루"

// const invalidKeyValue = createKeyValue(true, 123); // 컴파일 에러: K는 string | number가 아니에요.

interface Dictionary<K extends string, V> {
  [key: string]: V;
}

const stringDict: Dictionary<string, number> = { a: 1, b: 2 };
// const numberDict: Dictionary<number, string> = { 1: "a" }; // 컴파일 에러: K는 string이어야 해요.
// 위 코드에서 K extends string은 인덱스 시그니처의 키가 string이어야 한다는 제약 조건과 일치해야 해요.
```

`createKeyValue` 함수는 `K`와 `V` 두 가지 제네릭 타입을 받아요. `K`는 `string` 또는 `number`로 제약되어 있어 키로 사용될 수 있는 타입을 제한하고 있죠. `Dictionary` 인터페이스 또한 키가 `string`이어야 한다는 제약 조건을 가지고 있어요. 이처럼 여러 제네릭 타입을 활용하면 더욱 정교한 타입 정의가 가능해요.

### 2️⃣ `keyof`와 `typeof` 연산자와 함께 사용하기

제네릭과 `keyof`, `typeof` 연산자를 조합하면 객체의 속성 이름을 타입으로 받거나, 기존 변수의 타입을 추론하여 더욱 유연하고 강력한 타입을 만들 수 있어요.

```tsx showLineNumbers {1, 4-5}
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { name: "블루", age: 10, email: "blue@example.com" };

const userName = getProperty(user, "name"); // userName의 타입은 string
console.log(userName); // "블루"

const userAge = getProperty(user, "age"); // userAge의 타입은 number
console.log(userAge); // 10

// const userAddress = getProperty(user, "address"); // 컴파일 에러: user 객체에는 'address' 속성이 없어요.

// typeof와 함께 사용한 예시
type UserType = typeof user;

function updateProperty<T extends UserType, K extends keyof T>(
  obj: T,
  key: K,
  value: T[K],
): T {
  return { ...obj, [key]: value };
}

const updatedUser = updateProperty(user, "age", 11);
console.log(updatedUser); // { name: "블루", age: 11, email: "blue@example.com" }
// updateProperty(user, "age", "열한살"); // 컴파일 에러: 'age' 속성은 number 타입이어야 해요.
```

`getProperty` 함수는 `keyof T`를 사용하여 `K`가 `T`의 유효한 속성 이름임을 보장해요. 그리고 `T[K]`를 통해 해당 속성의 타입을 정확하게 반환하죠. `updateProperty` 함수에서는 `typeof`를 사용하여 기존 객체의 타입을 추출하고, 이를 제네릭 제약 조건으로 활용하여 더욱 안전한 업데이트 함수를 만들었어요. 이 패턴은 객체의 속성에 동적으로 접근하거나 수정할 때 매우 유용해요.

## 📝 정리하며: 제네릭 마스터하기

### 0️⃣ 핵심 요약

오늘 우리는 타입스크립트 제네릭의 깊은 세계를 탐험해 보았어요. 제네릭은 `any` 타입의 단점을 보완하고, 코드의 재사용성과 타입 안정성을 동시에 높여주는 강력한 도구라는 것을 알게 되셨을 거예요.<br />
함수, 인터페이스, 클래스에서 제네릭을 활용하는 기본 방법부터 `extends`를 이용한 제약 조건 설정, 그리고 `keyof`, `typeof`와 같은 연산자들과의 조합까지 다양한 실용적인 패턴들을 살펴보았어요. 특히 제네릭 기본값 설정은 유연성을 더해주지만 `any` 남용은 주의해야 한다는 점도 강조 드렸죠.

### 1️⃣ 다음 액션: 직접 적용해 보기

제네릭은 한 번에 모든 것을 이해하기보다는, 직접 코드를 작성하면서 경험해 보는 것이 가장 중요해요. 여러분의 프로젝트에서 반복되는 코드 패턴이 있다면 제네릭을 사용하여 리팩토링해 보세요. 예를 들어, 공통적으로 데이터를 다루는 유틸리티 함수나 상태 관리 객체 등에 제네릭을 적용해 볼 수 있을 거예요.<br />
처음에는 어렵게 느껴질 수 있지만, 꾸준히 연습하다 보면 어느새 제네릭을 자유자재로 다루는 타입스크립트 고수가 되어 있을 거예요. 복잡한 타입을 다루는 데 있어 제네릭은 여러분의 가장 든든한 조력자가 될 것이라고 확신해요!
