> Next.js App Router에서 제공하는 다양한 캐싱 메커니즘을 이해하고, 이를 활용하여 애플리케이션의 성능을 극대화하면서도 데이터의 신선도를 유지하는 실질적인 방법을 제시합니다.

## 🤔 문제/배경

### 0️⃣ 왜 이 주제를 다루는가요?

Next.js App Router는 React Server Components(RSC)와 함께 새로운 데이터 가져오기(data fetching) 패러다임을 도입했어요. 이로 인해 기존 Pages Router와는 차원이 다른, 다층적이고 강력한 캐싱 메커니즘이 기본적으로 적용되는데요. 많은 초중급 개발자분들이 App Router의 캐싱 동작 방식을 정확히 이해하지 못해서 예상치 못한 데이터 문제나 성능 저하를 겪는 경우가 많아요. 데이터의 신선도(freshness)를 유지하면서도 애플리케이션의 성능을 최적화하는 균형점을 찾는 것이 매우 중요하기 때문에 이 주제를 다루게 되었어요.<br />

### 1️⃣ 기존 방식의 한계 (Pages Router와의 비교)

Pages Router에서는 `getServerSideProps`나 `getStaticProps` 등을 통해 서버 사이드 렌더링(SSR) 또는 정적 사이트 생성(SSG) 및 증분 정적 재생성(ISR)을 비교적 명확하게 제어할 수 있었어요. 캐싱 전략 또한 `revalidate` 옵션 하나로 관리하는 경우가 많았죠. 하지만 App Router는 기본적으로 모든 `fetch` 요청이 캐시될 수 있으며, 다양한 캐싱 계층이 존재해서 동작 방식을 직관적으로 이해하기 어렵다는 한계가 있어요. 이 복잡성 때문에 캐시가 예상대로 동작하지 않거나, 오래된 데이터가 노출되는 등의 문제가 발생하기도 해요.<br />

## ⚙️ 해결 방법

App Router의 캐싱 문제를 해결하고 효율적으로 관리하기 위해서는 Next.js가 제공하는 세 가지 주요 캐싱 계층과 `fetch` API의 확장 기능, 그리고 `revalidatePath`, `revalidateTag` 같은 캐시 무효화(invalidation) API를 정확히 이해하고 활용해야 해요.

### 0️⃣ 핵심 아이디어: 세 가지 캐싱 계층 이해하기

Next.js App Router의 캐싱은 크게 세 가지 계층으로 나눌 수 있어요.

<Blockquote type="info">**1. Request Memoization (요청 메모이제이션)**<br />
동일한 `fetch` 요청이 같은 React `render` 패스 내에서 여러 번 발생할 때, 한 번만 실행하고 결과를 재사용하는 방식이에요. 서버 컴포넌트 뿐만 아니라 클라이언트 컴포넌트에서도 동일하게 적용됩니다.</Blockquote>

<Blockquote type="info">**2. Data Cache (데이터 캐시)**<br />
`fetch` 요청의 결과를 서버에 저장하는 캐시에요. 기본적으로 `GET` 요청은 캐시되며, `POST` 등의 다른 HTTP 메서드는 캐시되지 않아요. 이 캐시는 Next.js 빌드 시 생성되는 `.next/cache/fetch-data` 디렉토리에 저장되고, 애플리케이션이 배포된 후에도 재사용될 수 있습니다. ISR과 유사하게 `revalidate` 옵션으로 유효 기간을 설정할 수 있어요.</Blockquote>

<Blockquote type="info">**3. Full Route Cache (전체 경로 캐시)**<br />
Next.js 서버 컴포넌트 트리와 모든 데이터 요청의 최종 HTML 및 RSC 페이로드를 캐시하는 방식이에요. 사용자가 특정 경로에 처음 접근할 때 생성되며, 이후 동일 경로 요청 시 캐시된 내용을 즉시 반환하여 매우 빠른 로딩 속도를 제공합니다. 이 캐시는 기본적으로 영구적으로 유지되며, `revalidatePath`나 `revalidateTag`를 통해서만 무효화할 수 있어요.</Blockquote>

이 세 가지 캐싱 계층이 계단식으로 동작하며, 개발자는 `fetch` API 옵션과 캐시 무효화 함수를 통해 이를 제어할 수 있어요.<br />

### 1️⃣ 적용 방법: `fetch` 옵션과 캐시 무효화

#### 1. `fetch` API 확장 옵션 활용

App Router에서는 기본 `fetch` API에 Next.js 전용 옵션을 추가하여 캐싱 동작을 세밀하게 제어할 수 있어요.

*   **`cache: 'no-store'`**: 해당 `fetch` 요청의 결과를 Data Cache에 저장하지 않고, 항상 최신 데이터를 서버에서 가져오도록 해요. SSR과 유사하게 동작합니다.
*   **`cache: 'force-cache'`**: Data Cache를 우선적으로 사용하고, 캐시된 데이터가 없으면 서버에서 가져와 캐시해요. SSG와 유사하게 동작합니다.
*   **`next: { revalidate: number }`**: Data Cache에 저장된 데이터의 유효 기간(초 단위)을 설정해요. 이 시간이 지나면 다음 요청 시 데이터를 다시 가져와 캐시를 갱신합니다. ISR과 유사하게 동작해요.
*   **`next: { tags: string[] }`**: Data Cache에 저장될 데이터에 태그를 부여해요. 이 태그를 사용하여 나중에 특정 데이터 캐시만 선택적으로 무효화할 수 있습니다.

<Blockquote type="tip">**팁:** 기본 `fetch` 동작은 `cache: 'force-cache'`와 `next: { revalidate: false }`가 결합된 것과 유사하게 동작해요. 즉, 한 번 캐시되면 영구적으로 유지되려고 합니다. 따라서 데이터 신선도가 중요한 경우에는 반드시 `no-store`나 `revalidate` 옵션을 명시적으로 설정해야 해요.</Blockquote>

#### 2. 캐시 무효화 (Cache Invalidation)

데이터가 변경되었을 때 캐시된 내용을 강제로 갱신해야 하는 경우가 있어요. Next.js는 이를 위해 두 가지 함수를 제공해요.

*   **`revalidatePath(path: string)`**: 특정 경로(path)와 관련된 모든 Full Route Cache 및 Data Cache를 무효화해요. 예를 들어, `/products` 경로의 게시물 목록이 업데이트되면 `revalidatePath('/products')`를 호출하여 캐시를 갱신할 수 있어요.
*   **`revalidateTag(tag: string)`**: `fetch` 요청 시 `next: { tags: [...] }` 옵션으로 설정했던 특정 태그와 관련된 Data Cache만 무효화해요. 이는 여러 경로에서 공유되는 데이터 캐시를 효율적으로 관리할 때 유용합니다.

이 두 함수는 **서버 액션(Server Actions)**이나 **Route Handlers** 내에서만 호출할 수 있어요. 클라이언트 컴포넌트에서는 직접 호출할 수 없다는 점을 유의해야 해요.<br />

## 🧪 예시

실제 코드 예시를 통해 App Router의 캐싱 전략을 어떻게 적용하는지 살펴보아요.

### 0️⃣ 코드/설정 예시

#### 1. 데이터 가져오기 (Data Fetching)와 캐싱 옵션

다음은 서버 컴포넌트에서 `fetch`를 사용하여 데이터를 가져오는 예시와 각 캐싱 옵션의 적용 방법이에요.

```tsx showLineNumbers
// app/products/page.tsx - 상품 목록 페이지 (SSR처럼 항상 최신 데이터)
import ProductList from './ProductList';

async function getProducts() {
  const res = await fetch('https://api.example.com/products', {
    cache: 'no-store', // 항상 최신 데이터를 가져옴
  });
  if (!res.ok) throw new Error('Failed to fetch products');
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts();
  return (
    <div>
      <h1>상품 목록</h1>
      <ProductList products={products} />
    </div>
  );
}
```

```tsx showLineNumbers
// app/posts/[id]/page.tsx - 블로그 게시물 상세 페이지 (ISR처럼 주기적으로 갱신)
import PostDetail from './PostDetail';

async function getPost(id: string) {
  const res = await fetch(`https://api.example.com/posts/${id}`, {
    next: { revalidate: 60 }, // 60초마다 데이터 갱신 시도
    // next: { tags: ['posts', `post-${id}`] } // 태그를 이용한 캐시 무효화 준비
  });
  if (!res.ok) throw new Error('Failed to fetch post');
  return res.json();
}

export default async function PostPage({ params }: { params: { id: string } }) {
  const post = await getPost(params.id);
  return (
    <div>
      <h1>{post.title}</h1>
      <PostDetail post={post} />
    </div>
  );
}

// generateStaticParams를 사용하여 빌드 시점에 특정 게시물들을 미리 생성할 수 있어요.
export async function generateStaticParams() {
  const res = await fetch('https://api.example.com/posts');
  const posts = await res.json();
  return posts.map((post: any) => ({ id: post.id.toString() }));
}
```

#### 2. 캐시 무효화 (Cache Invalidation) 예시

사용자가 새 게시물을 작성하거나 기존 게시물을 수정/삭제했을 때, 관련 캐시를 무효화하는 방법이에요. 이는 **서버 액션** 또는 **Route Handler** 내에서 이루어져야 합니다.

```tsx showLineNumbers {12,14}
// app/actions.ts - 서버 액션 예시
'use server';

import { revalidatePath, revalidateTag } from 'next/cache';
import { redirect } from 'next/navigation';

export async function createPost(formData: FormData) {
  const title = formData.get('title');
  const content = formData.get('content');
  // ... 데이터베이스에 새 게시물 저장 로직 ...

  revalidatePath('/posts'); // '/posts' 경로의 Full Route Cache 및 Data Cache 무효화
  // revalidateTag('posts'); // 'posts' 태그를 가진 Data Cache 무효화 (만약 fetch에 tags 옵션을 썼다면)
  redirect('/posts');
}

export async function updatePost(id: string, formData: FormData) {
  const title = formData.get('title');
  const content = formData.get('content');
  // ... 데이터베이스에 게시물 업데이트 로직 ...

  revalidatePath(`/posts/${id}`); // 특정 게시물 상세 페이지 캐시 무효화
  revalidatePath('/posts');       // 목록 페이지 캐시도 무효화하여 최신 변경사항 반영
  // revalidateTag(`post-${id}`); // 특정 게시물 태그 캐시 무효화
  // revalidateTag('posts');       // 모든 게시물 태그 캐시 무효화
  redirect(`/posts/${id}`);
}
```

```tsx showLineNumbers {13,15}
// app/api/posts/route.ts - Route Handler 예시 (POST 요청)
import { NextResponse } from 'next/server';
import { revalidatePath, revalidateTag } from 'next/cache';

export async function POST(request: Request) {
  const data = await request.json();
  // ... 데이터베이스에 새 게시물 저장 로직 ...

  // 데이터 저장 후 캐시 무효화
  revalidatePath('/posts'); // 전체 게시물 목록 페이지 캐시 무효화
  // revalidateTag('posts'); // 'posts' 태그를 가진 Data Cache 무효화

  return NextResponse.json({ message: 'Post created successfully' }, { status: 201 });
}
```

### 1️⃣ 적용 결과

위 예시들을 통해 캐싱 전략을 적용하면 다음과 같은 이점을 얻을 수 있어요.

*   **성능 향상**: `force-cache`나 `revalidate` 옵션을 통해 정적 콘텐츠를 빠르게 제공하고, `Request Memoization`으로 중복 요청을 줄여 서버 부담을 완화해요.
*   **데이터 신선도 유지**: `no-store` 옵션이나 `revalidate` 시간을 적절히 설정하여 중요한 데이터는 항상 최신 상태를 유지할 수 있어요. 또한 `revalidatePath`와 `revalidateTag`를 통해 데이터 변경 즉시 캐시를 갱신하여 사용자에게 최신 정보를 제공합니다.
*   **개발 편의성**: `fetch` API의 확장 옵션과 캐시 무효화 함수를 사용하여 복잡한 캐싱 로직을 비교적 선언적으로 관리할 수 있어요.

<Blockquote type="warning">**주의:** `revalidatePath`는 해당 경로의 모든 Full Route Cache와 Data Cache를 무효화하지만, `revalidateTag`는 Data Cache만 무효화해요. 따라서 `revalidateTag`를 사용한 후에도 해당 경로의 Full Route Cache는 여전히 오래된 데이터로 채워져 있을 수 있으므로, 경로 전체를 갱신해야 하는 경우에는 `revalidatePath`를 함께 사용해야 할 때가 많습니다.</Blockquote>

## 📝 정리

### 0️⃣ 핵심 요약

Next.js App Router의 캐싱은 Request Memoization, Data Cache, Full Route Cache의 세 가지 계층으로 이루어져 있어요. 각 계층은 `fetch` API의 `cache` 및 `next` 옵션, 그리고 `revalidatePath`, `revalidateTag` 함수를 통해 제어할 수 있습니다. 데이터의 중요도와 갱신 주기에 따라 적절한 캐싱 전략을 선택하고, 데이터 변경 시에는 반드시 캐시 무효화 함수를 호출하여 데이터 신선도를 유지하는 것이 핵심이에요.

*   **Request Memoization**: 같은 렌더링 패스 내 중복 `fetch` 요청 방지.
*   **Data Cache**: `fetch` 결과 서버 캐시 (SSG/ISR/SSR처럼 제어).
*   **Full Route Cache**: 전체 라우트의 HTML 및 RSC 페이로드 캐시 (가장 강력한 캐시).
*   **`cache: 'no-store'`**: 항상 최신 데이터 (SSR).
*   **`next: { revalidate: N }`**: N초 후 캐시 갱신 (ISR).
*   **`next: { tags: [...] }`**: 태그 기반 Data Cache 무효화.
*   **`revalidatePath('/path')`**: 특정 경로의 모든 캐시 무효화 (Full Route Cache 포함).
*   **`revalidateTag('tag')`**: 특정 태그를 가진 Data Cache 무효화.

### 1️⃣ 다음 액션

현재 운영 중인 Next.js App Router 기반 애플리케이션이나 새롭게 개발할 프로젝트에서 데이터 가져오기 로직을 검토해 보세요. 각 `fetch` 요청에 어떤 캐싱 전략이 가장 적합한지 고민하고, `revalidate` 옵션이나 `revalidatePath`, `revalidateTag`를 적극적으로 활용하여 성능과 데이터 신선도라는 두 마리 토끼를 모두 잡아보세요. 특히 서버 액션을 활용한 캐시 무효화 패턴에 익숙해지는 것이 중요해요.<br />
