## 🤔 낙관적 UI 업데이트, 왜 필요할까요?

웹 애플리케이션에서 사용자 경험은 성공의 핵심 요소 중 하나예요. 특히 사용자의 액션에 대한 즉각적인 피드백은 앱의 '반응성'을 결정하고, 이는 곧 사용자의 만족도로 이어지죠.<br />

### 0️⃣ 사용자 경험의 중요성

사용자가 버튼을 클릭하거나 폼을 제출했을 때, 우리는 서버로부터 응답이 올 때까지 기다리게 하는 경우가 많아요. 하지만 이 짧은 기다림조차도 사용자에게는 답답함으로 느껴질 수 있어요.<br />

예를 들어, 소셜 미디어에서 '좋아요' 버튼을 눌렀을 때, 바로 좋아요 수가 올라가거나 버튼 색이 바뀌지 않고 잠시 기다려야 한다면 어떨까요? 분명 긍정적인 경험은 아닐 거예요.<br />

### 1️⃣ 기존 비관적 UI의 한계

전통적인 방식은 '비관적(Pessimistic) UI'라고 부를 수 있어요. 사용자의 액션이 발생하면,<br />

1.  UI를 업데이트하지 않고 서버 응답을 기다려요.<br />
2.  서버 응답이 성공하면 UI를 업데이트하고,<br />
3.  실패하면 에러 메시지를 표시하고 UI를 원래 상태로 되돌리죠.<br />

이 방식은 데이터 일관성을 보장하지만, 네트워크 지연이나 서버 처리 시간 때문에 사용자에게 불필요한 대기 시간을 강요하게 돼요. 특히 모바일 환경이나 네트워크가 불안정한 환경에서는 이러한 문제가 더욱 두드러지게 나타나요.<br />

<Blockquote type="error">
비관적 UI는 항상 최신 서버 상태를 반영하지만, 사용자에게는 '느리다'는 인상을 줄 수 있어요.
</Blockquote>

## 🚀 `useOptimistic` 훅, 어떤 역할을 하나요?

React 18.3 (Canary)에 도입된 `useOptimistic` 훅은 이러한 비관적 UI의 한계를 극복하고, 사용자에게 즉각적인 피드백을 제공하는 '낙관적(Optimistic) UI'를 쉽게 구현할 수 있도록 도와줘요.<br />

### 0️⃣ `useOptimistic`의 핵심 원리

`useOptimistic`은 이름 그대로 '낙관적'으로 UI를 업데이트하는 것을 목표로 해요. 사용자가 특정 액션을 취하면, 서버 응답을 기다리지 않고 미리 UI를 업데이트하는 거예요. 그리고 백그라운드에서 실제 서버 요청을 보내죠.<br />

만약 서버 요청이 성공하면, 미리 업데이트된 UI 상태를 확정하고, 만약 실패하면 UI를 원래 상태로 되돌리거나 적절한 에러 처리를 하는 방식이에요.<br />

이를 통해 사용자는 자신의 액션이 즉시 반영되는 것처럼 느끼게 되어 애플리케이션이 훨씬 빠르고 반응성이 좋다고 인식하게 돼요.

### 1️⃣ `useOptimistic` 사용법

`useOptimistic` 훅은 다음과 같은 형태로 사용해요.

```tsx showLineNumbers
import { useOptimistic } from 'react';

function MyComponent({ items }) {
  const [optimisticItems, addOptimistic] = useOptimistic(
    items, // 초기 상태 (서버로부터 받은 실제 데이터)
    (currentItems, newItem) => { // 업데이트 함수 (낙관적으로 UI를 변경할 로직)
      return [...currentItems, newItem];
    }
  );

  // ... (이후 로직)
}
```

<br />

여기서 두 가지 중요한 인자를 볼 수 있어요.<br />

1.  **`state`**: `useOptimistic` 훅이 관리할 초기 상태예요. 일반적으로 서버로부터 받아온 실제 데이터가 되겠죠.<br />
2.  **`updater`**: 사용자가 낙관적으로 UI를 업데이트할 때 호출되는 함수예요. 이 함수는 현재 `state`와 `addOptimistic` 함수를 통해 전달받은 `payload`를 인자로 받아 새로운 낙관적 상태를 반환해요.<br />

`useOptimistic` 훅은 두 가지 값을 반환해요.<br />

1.  **`optimisticState`**: 현재 UI에 표시될 상태예요. 서버로부터 받은 실제 상태(committed state)이거나, `addOptimistic`을 통해 업데이트된 낙관적 상태(optimistic state)가 될 수 있어요.<br />
2.  **`addOptimistic`**: 낙관적 업데이트를 트리거하는 함수예요. 이 함수를 호출하면 `updater` 함수가 실행되어 `optimisticState`가 즉시 업데이트돼요.

## 🧪 `useOptimistic`과 Server Actions 연동 예시

이제 실제 코드를 통해 `useOptimistic` 훅이 Next.js Server Actions와 어떻게 아름답게 연동되는지 살펴볼게요. 댓글 추가 기능을 예시로 들어볼까요?

### 0️⃣ 시나리오: 댓글 추가 기능

사용자가 댓글을 입력하고 '작성' 버튼을 누르면,<br />

1.  댓글 목록에 즉시 새로운 댓글이 '등록 중...' 상태로 추가돼요.<br />
2.  백그라운드에서 Server Action이 호출되어 실제 댓글을 서버에 저장해요.<br />
3.  Server Action이 성공하면 '등록 중...' 상태의 댓글이 실제 댓글로 확정돼요.<br />
4.  Server Action이 실패하면 '등록 중...' 상태의 댓글이 사라지고, 에러 메시지가 표시돼요.

### 1️⃣ 코드 구현

먼저, Server Action을 정의해 볼게요. `app/actions.ts` 파일에 다음과 같이 작성할 수 있어요. (실제 DB 로직은 생략하고 `setTimeout`으로 비동기 작업을 흉내 냈어요.)

```ts showLineNumbers
// app/actions.ts
'use server';

interface Comment {
  id: string;
  text: string;
  status: 'pending' | 'committed' | 'failed';
}

let comments: Comment[] = [
  { id: '1', text: '첫 번째 댓글이에요!', status: 'committed' },
  { id: '2', text: '블루님 글 정말 유익해요!', status: 'committed' },
];

export async function addComment(commentText: string): Promise<Comment[]> {
  console.log('Server Action: 댓글 추가 요청 시작', commentText);
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (Math.random() > 0.8) { // 20% 확률로 실패 시뮬레이션
        console.error('Server Action: 댓글 추가 실패!');
        reject(new Error('댓글 추가에 실패했어요. 다시 시도해 주세요.'));
      } else {
        const newComment: Comment = {
          id: String(comments.length + 1),
          text: commentText,
          status: 'committed',
        };
        comments.push(newComment);
        console.log('Server Action: 댓글 추가 성공', newComment);
        resolve(comments);
      }
    }, 1500); // 1.5초 지연 시뮬레이션
  });
}

export async function getComments(): Promise<Comment[]> {
  console.log('Server Action: 댓글 불러오기 요청');
  // 실제 DB에서 불러오는 로직 대신 현재 comments 배열 반환
  return comments;
}
```

<br />

이제 이 Server Action을 사용하는 클라이언트 컴포넌트를 만들어 볼게요. `app/comments-page.tsx`에 다음과 같이 작성할 수 있어요.<br />

```tsx showLineNumbers {13-19, 23, 27-28, 32-34, 43-46}
// app/comments-page.tsx
'use client';

import { useState, useRef } from 'react';
import { useOptimistic } from 'react';
import { addComment } from './actions';

interface Comment {
  id: string;
  text: string;
  status: 'pending' | 'committed' | 'failed';
}

export default function CommentsPage({ initialComments }: { initialComments: Comment[] }) {
  const [optimisticComments, addOptimisticComment] = useOptimistic(
    initialComments, // 초기 상태는 서버에서 받은 댓글 목록
    (currentComments, newCommentText: string) => [
      ...currentComments,
      { id: 'temp-' + Date.now(), text: newCommentText, status: 'pending' }, // 낙관적 댓글 추가
    ]
  );
  const formRef = useRef<HTMLFormElement>(null);
  const [error, setError] = useState<string | null>(null);

  async function handleSubmit(formData: FormData) {
    const commentText = formData.get('comment') as string;
    if (!commentText.trim()) return;

    formRef.current?.reset(); // 폼 초기화
    setError(null);

    // 1. 낙관적으로 UI 업데이트
    addOptimisticComment(commentText);

    try {
      // 2. Server Action 호출 (실제 데이터 저장)
      await addComment(commentText);
      // 성공 시, useOptimistic은 자동으로 initialComments(현재 서버 상태)로 동기화됨
    } catch (e: any) {
      setError(e.message || '댓글 추가 중 알 수 없는 오류가 발생했어요.');
      // 실패 시, 실제 데이터로 되돌아가지 않고, 낙관적 상태가 유지될 수 있으므로
      // 이 예시에서는 에러 메시지 표시 후 사용자가 새로고침하도록 유도하거나,
      // 더 복잡한 로직으로 낙관적 상태를 제거해야 할 수 있어요.
      // 현재 예시에서는 `optimisticComments`에 'pending' 상태로 남아있게 됩니다.
      // 실제 프로덕션에서는 실패한 낙관적 아이템을 필터링하거나, 'failed' 상태로 변경하는 로직이 필요해요.
    }
  }

  return (
    <div className="max-w-xl mx-auto p-4">
      <h1 className="text-2xl font-bold mb-4">💬 댓글 목록</h1>

      <form ref={formRef} action={handleSubmit} className="mb-6 flex gap-2">
        <input
          type="text"
          name="comment"
          placeholder="댓글을 입력하세요..."
          className="flex-grow p-2 border border-gray-300 rounded-md focus:outline-none focus:ring-2 focus:ring-blue-500"
          required
        />
        <button
          type="submit"
          className="px-4 py-2 bg-blue-600 text-white rounded-md hover:bg-blue-700 focus:outline-none focus:ring-2 focus:ring-blue-500 focus:ring-offset-2"
        >
          작성
        </button>
      </form>

      {error && (
        <Blockquote type="error">
          {error}
        </Blockquote>
      )}

      <ul className="space-y-3">
        {optimisticComments.map((comment, index) => (
          <li
            key={comment.id || `optimistic-${index}`}
            className={`p-3 border rounded-md ${comment.status === 'pending' ? 'bg-yellow-50 text-yellow-800 border-yellow-200 animate-pulse' : comment.status === 'failed' ? 'bg-red-50 text-red-800 border-red-200' : 'bg-gray-50'}`}
          >
            {comment.text}
            {comment.status === 'pending' && <span className="ml-2 text-sm"> (등록 중...)</span>}
            {comment.status === 'failed' && <span className="ml-2 text-sm"> (등록 실패!)</span>}
          </li>
        ))}
      </ul>
    </div>
  );
}
```

<Blockquote type="info">
이 예시에서는 `CommentsPage` 컴포넌트를 클라이언트 컴포넌트로 만들고 `initialComments`를 props로 받았어요. Next.js 13 이상의 App Router 환경에서는 다음과 같이 서버 컴포넌트에서 초기 데이터를 가져와 클라이언트 컴포넌트에 전달하는 것이 일반적이에요.<br />

```tsx showLineNumbers
// app/page.tsx (서버 컴포넌트)
import { getComments } from './actions';
import CommentsPage from './comments-page'; // 위에서 정의한 클라이언트 컴포넌트

export default async function Page() {
  const initialComments = await getComments();
  return <CommentsPage initialComments={initialComments} />;
}
```
</Blockquote>

### 2️⃣ 동작 원리 분석

위 코드의 핵심은 `handleSubmit` 함수 내부에 있어요.<br />

1.  **`addOptimisticComment(commentText);`**: 사용자가 댓글을 제출하자마자 `addOptimisticComment`를 호출하여 `optimisticComments` 배열에 `status: 'pending'`인 임시 댓글을 추가해요. 이로 인해 UI는 즉시 업데이트되어 사용자는 댓글이 추가된 것처럼 느끼게 돼요.<br />
2.  **`await addComment(commentText);`**: 이와 동시에 실제 서버 액션인 `addComment`가 호출돼요. 이 함수는 1.5초의 지연과 20%의 실패 확률을 시뮬레이션하고 있죠.<br />
3.  **성공 시**: `addComment`가 성공적으로 완료되면, `useOptimistic` 훅은 자동으로 `optimisticComments`를 초기 `initialComments` (또는 `useOptimistic`의 첫 번째 인자로 전달된 최신 상태)로 재동기화해요. 이 과정에서 'pending' 상태였던 임시 댓글은 사라지고, 서버에서 실제 저장된 댓글 목록이 다시 렌더링되면서 확정된 상태가 돼요. (이 예시에서는 Server Action이 새로운 comments 배열을 반환하므로, React는 이 새로운 배열을 `initialComments`로 간주하고 `optimisticComments`를 업데이트합니다. 실제로는 `revalidatePath` 등을 통해 캐시를 무효화하여 최신 데이터를 가져오는 것이 일반적이에요.)<br />
4.  **실패 시**: `addComment`가 실패하면 `catch` 블록이 실행돼요. 이 예시에서는 에러 메시지를 표시하지만, `useOptimistic`은 기본적으로 낙관적 상태를 자동으로 되돌리지 않아요. 따라서 실패한 낙관적 아이템을 UI에서 제거하거나, 'failed' 상태로 명확히 표시하여 사용자에게 알리는 추가적인 로직이 필요해요. 위의 코드에서는 'failed' 상태를 사용하여 시각적으로 구분하도록 했어요.

<Blockquote type="tip">
`useOptimistic`은 Server Actions와 함께 사용할 때 특히 강력해요. Server Actions는 폼 제출과 같은 사용자 인터랙션을 서버에서 직접 처리하고, React의 캐시 무효화 및 데이터 재검증(revalidation) 메커니즘과 잘 통합되기 때문이에요.
</Blockquote>

## 💡 `useOptimistic` 활용 팁 및 고려사항

`useOptimistic` 훅은 매우 유용하지만, 몇 가지 고려사항을 염두에 두어야 해요.

### 0️⃣ 사용자 경험 최적화

*   **명확한 시각적 피드백**: 낙관적 상태의 아이템은 일반 아이템과 시각적으로 구분하는 것이 좋아요. (예: 투명도 조절, 회색 처리, '등록 중...' 텍스트 추가, 로딩 스피너 등). 사용자가 어떤 항목이 아직 확정되지 않았는지 명확히 알 수 있도록 해주세요.<br />
*   **짧은 지연 시간 예상**: `useOptimistic`은 서버 응답이 비교적 빠르게 올 것으로 예상되는 액션에 적합해요. 아주 긴 시간이 소요되는 작업에는 적합하지 않을 수 있어요.

### 1️⃣ 에러 핸들링 전략

*   **롤백 로직**: 서버 액션이 실패했을 때 낙관적으로 추가했던 UI를 어떻게 처리할지 명확히 정의해야 해요. 단순히 UI에서 제거할 수도 있고, 에러 상태를 표시하며 재시도 버튼을 제공할 수도 있어요. 이 예시에서는 `status: 'failed'`로 변경하여 시각적으로 구분했어요.<br />
*   **에러 메시지**: 실패 시 사용자에게 친절하고 명확한 에러 메시지를 제공하여 혼란을 줄여야 해요.

### 2️⃣ 다른 상태 관리와의 조합

*   `useOptimistic`은 특정 액션에 대한 임시 UI 상태 관리에 특화되어 있어요. 전역 상태 관리 라이브러리(Zustand, Recoil 등)와 함께 사용할 수도 있지만, `useOptimistic`은 React 훅이므로 컴포넌트 내부에서 로컬 상태처럼 관리하는 것이 일반적이에요. Server Actions와 함께 사용하면 별도의 전역 상태 관리 없이도 충분히 강력한 기능을 구현할 수 있어요.

## 📝 정리하며

오늘은 React의 `useOptimistic` 훅을 활용하여 낙관적 UI 업데이트를 구현하는 방법을 자세히 살펴보았어요. 특히 Next.js Server Actions와의 시너지를 통해 사용자 경험을 크게 향상시킬 수 있음을 확인했죠.<br />

### 0️⃣ 핵심 요약

*   `useOptimistic`은 서버 응답을 기다리지 않고 즉시 UI를 업데이트하여 사용자에게 더 빠르고 반응적인 경험을 제공해요.<br />
*   `[optimisticState, addOptimistic]` 형태로 사용하며, `addOptimistic`을 호출하여 낙관적 상태를 변경해요.<br />
*   Server Actions와 함께 사용하면 클라이언트-서버 간의 데이터 동기화 및 UI 업데이트를 매우 효율적으로 관리할 수 있어요.<br />
*   에러 처리 및 시각적 피드백을 통해 사용자에게 혼란을 주지 않도록 주의해야 해요.

### 1️⃣ 다음 스텝

`useOptimistic` 훅은 댓글, 좋아요, 할 일 목록 추가 등 다양한 상황에서 사용자 경험을 개선하는 데 활용될 수 있어요. 여러분의 프로젝트에 직접 적용해 보면서 이 훅의 강력함을 경험해 보시길 권해 드려요. 더 나아가, 실패 시 낙관적 UI를 자동으로 롤백하는 커스텀 훅을 만들어 보는 것도 좋은 학습 경험이 될 거예요.<br />
