## 🤔 왜 ESLint와 Prettier가 필요한가요?

현대 웹 개발에서 코드 품질과 일관성은 프로젝트의 성공을 좌우하는 중요한 요소예요. 특히 여러 개발자가 함께 작업하는 팀 프로젝트에서는 이러한 중요성이 더욱 커지죠.

### 0️⃣ 코드 품질과 일관성의 중요성

코드 품질은 버그를 줄이고 유지보수성을 높이는 데 결정적인 역할을 해요. 또한, 일관된 코딩 스타일은 가독성을 향상시키고, 새로운 팀원이 프로젝트에 합류했을 때 적응 시간을 단축시켜 주기도 합니다.<br />
만약 각자 다른 스타일로 코드를 작성한다면, 코드 리뷰 과정에서 불필요한 스타일 논쟁이 발생하거나, 코드 병합 시 충돌이 자주 발생할 수 있어요. 이는 개발 생산성을 저해하는 주요 원인이 되죠.

### 1️⃣ 개발 생산성 향상

ESLint와 Prettier는 이러한 문제를 해결하고 개발 생산성을 크게 향상시키는 데 도움을 줍니다.<br />
ESLint는 잠재적인 버그, 안티 패턴, 코드 품질 문제를 정적으로 분석하여 미리 경고하거나 수정해 줍니다. 예를 들어, 사용되지 않는 변수, 접근 불가능한 코드, 잠재적인 메모리 누수 등을 찾아내죠.<br />
Prettier는 코드 포맷팅에 특화된 도구로, 정해진 규칙에 따라 코드를 자동으로 재정렬해 줍니다. 들여쓰기, 따옴표 스타일, 세미콜론 사용 여부 등 사소하지만 논쟁이 될 수 있는 스타일 문제를 깔끔하게 해결해 줘요. 덕분에 개발자들은 코드 스타일 대신 비즈니스 로직 구현에 더 집중할 수 있게 된답니다.

## ⚙️ ESLint와 Prettier, 어떻게 함께 사용하나요?

ESLint와 Prettier는 각각 다른 목적을 가지고 있지만, 함께 사용할 때 시너지를 발휘해요.

### 0️⃣ 두 도구의 역할 분담

ESLint는 코드의 "품질"과 "잠재적 문제"를 검사하는 데 중점을 둡니다. 즉, 코드의 논리적 오류나 비효율적인 패턴을 찾아내죠. <LinkPreviewCard href="https://eslint.org/docs/latest/" /> 반면, Prettier는 코드의 "스타일"과 "형식"을 통일하는 데 집중해요. 예를 들어, 줄 바꿈, 공백, 따옴표 사용 방식 등을 정해진 규칙에 따라 자동으로 맞춰줍니다.<br />
이 두 도구를 함께 사용하면, ESLint가 코드의 의미론적 문제를 검사하고, Prettier가 코드의 시각적 일관성을 책임지게 되어, 깔끔하고 오류 없는 코드를 작성할 수 있게 돼요.

### 1️⃣ 설치 및 기본 설정

새로운 Next.js 프로젝트를 생성할 때 ESLint는 기본적으로 포함되어 있어요. 하지만 Prettier와 함께 사용하기 위해서는 몇 가지 추가 설정이 필요합니다.<br />
먼저 필요한 패키지들을 설치해 볼까요?

```bash showLineNumbers
npm install --save-dev prettier eslint-config-prettier eslint-plugin-prettier
# 또는
yarn add --dev prettier eslint-config-prettier eslint-plugin-prettier
```

<Blockquote type="info">
`prettier`: 코드 포맷팅을 담당하는 핵심 라이브러리예요.<br />
`eslint-config-prettier`: ESLint의 포맷팅 규칙 중 Prettier와 충돌하는 부분을 비활성화하여, Prettier가 단독으로 포맷팅을 처리하도록 해줘요.<br />
`eslint-plugin-prettier`: Prettier를 ESLint 규칙으로 실행하여, Prettier 규칙에 위배되는 코드를 ESLint 오류로 보고하게 해줍니다.
</Blockquote>

## 🛠️ Next.js 프로젝트에 적용하기

이제 설치한 패키지들을 활용하여 Next.js 프로젝트의 ESLint와 Prettier 설정을 최적화해볼게요.

### 0️⃣ 기본 ESLint 설정 확장

Next.js 프로젝트는 기본적으로 `eslint-config-next`를 사용하고 있어요. 여기에 `eslint-plugin-prettier`와 `eslint-config-prettier`를 추가하여 Prettier와 잘 연동되도록 설정해 봅시다.<br />
프로젝트 루트에 있는 `.eslintrc.json` 파일을 다음과 같이 수정해 주세요.

```json showLineNumbers {5-6}
{
  "extends": [
    "next/core-web-vitals",
    "plugin:prettier/recommended" // Prettier 관련 설정을 마지막에 추가해요.
  ],
  "rules": {
    // 여기에 프로젝트 특화 ESLint 규칙을 추가할 수 있어요.
    // 예시: "react/react-in-jsx-scope": "off"
  }
}
```

<Blockquote type="tip">
`"plugin:prettier/recommended"`는 `eslint-plugin-prettier`와 `eslint-config-prettier`를 한 번에 적용하는 편리한 방법이에요.<br />
이 설정을 `extends` 배열의 가장 마지막에 두는 것이 중요합니다. 그래야 다른 모든 ESLint 규칙을 덮어쓰지 않고, Prettier와 충돌하는 규칙만 비활성화할 수 있어요.
</Blockquote>

### 1️⃣ Prettier 설정 추가

Prettier는 별도의 설정 파일(`.prettierrc.json` 또는 `.prettierrc.js`)을 통해 작동해요. 프로젝트 루트에 `.prettierrc.json` 파일을 생성하고 원하는 포맷팅 규칙을 추가해 보세요.

```json showLineNumbers
// .prettierrc.json
{
  "semi": true,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "all",
  "printWidth": 80
}
```

<Blockquote type="info">
`semi`: 문장 끝에 세미콜론을 추가할지 여부 (`true`면 추가)<br />
`singleQuote`: 따옴표를 작은따옴표로 사용할지 여부 (`true`면 작은따옴표)<br />
`tabWidth`: 들여쓰기 시 공백 수<br />
`trailingComma`: 객체나 배열의 마지막 요소 뒤에 쉼표를 추가할지 여부 (`all`은 가능한 모든 곳에 추가)<br />
`printWidth`: 한 줄의 최대 길이 (이 길이를 넘으면 자동으로 줄 바꿈)
</Blockquote>

또한, Prettier가 특정 파일을 무시하도록 `.prettierignore` 파일을 추가할 수 있어요. 일반적으로 빌드 결과물, 캐시 파일 등을 제외합니다.

```
# .prettierignore
.next/
node_modules/
public/
dist/
build/
```

### 2️⃣ 충돌 방지 설정

`eslint-config-prettier`를 통해 ESLint의 포맷팅 규칙과 Prettier의 규칙이 충돌하는 것을 방지했어요.<br />
하지만, ESLint가 Prettier 규칙에 따라 코드를 포맷팅하도록 하려면 `package.json`에 스크립트를 추가하는 것이 좋아요.

```json showLineNumbers {6-7}
// package.json
{
  "name": "my-next-app",
  "version": "0.1.0",
  "private": true,
  "scripts": {
    "lint": "next lint",
    "lint:fix": "eslint --fix .",
    "format": "prettier --write ."
  },
  "dependencies": {
    // ...
  },
  "devDependencies": {
    // ...
  }
}
```

<Blockquote type="tip">
`npm run lint`: ESLint 규칙에 따라 코드 품질을 검사해요.<br />
`npm run lint:fix`: ESLint 규칙 위반 중 자동으로 수정 가능한 부분을 수정해 줘요.<br />
`npm run format`: Prettier 규칙에 따라 전체 프로젝트의 코드 스타일을 일관되게 포맷팅합니다.
</Blockquote>

### 3️⃣ VS Code 연동

개발 환경에서 코드를 저장할 때마다 자동으로 포맷팅되도록 VS Code 설정을 해두면 매우 편리해요.

1.  **Prettier 확장 설치**: VS Code 마켓플레이스에서 "Prettier - Code formatter" 확장을 설치해 주세요.
2.  **VS Code 설정 (`settings.json`)**:
    `Ctrl + Shift + P` (또는 `Cmd + Shift + P`)를 눌러 "Preferences: Open User Settings (JSON)"을 검색하여 `settings.json` 파일을 열고 다음 설정을 추가합니다.

```json showLineNumbers {3-6}
// settings.json
{
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.formatOnSave": true,
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  }
}
```

<Blockquote type="info">
`"editor.defaultFormatter": "esbenp.prettier-vscode"`: 기본 포매터로 Prettier 확장을 지정해요.<br />
`"editor.formatOnSave": true`: 파일을 저장할 때마다 자동으로 코드를 포맷팅하도록 설정합니다.<br />
`"editor.codeActionsOnSave": { "source.fixAll.eslint": "explicit" }`: 파일을 저장할 때 ESLint 규칙 위반 중 자동으로 수정 가능한 부분을 수정하도록 설정해요.
</Blockquote>

이제 파일을 저장할 때마다 ESLint와 Prettier가 자동으로 작동하여 코드 품질과 스타일을 관리해 줄 거예요.

## 🧪 실전 예제: 코드 스타일 자동 교정

실제로 어떻게 작동하는지 간단한 예제를 통해 확인해 볼까요?

### 0️⃣ 잘못된 코드 작성

아래와 같이 ESLint 및 Prettier 규칙에 위배되는 코드를 작성해 보세요.<br />
`pages/index.tsx` (Next.js Pages Router 기준) 또는 `app/page.tsx` (Next.js App Router 기준) 파일에 작성해 봅니다.

```tsx showLineNumbers {2-3, 5, 8, 11-12}
// src/app/page.tsx (App Router 예시)
const MyComponent = () => {
  const unusedVariable = "Hello";
  let greeting = 'world';

  if(true) {
    console.log(greeting);
  }

  return (
    <div>
      <p>    Test Component   </p>
    </div>
  );
};

export default MyComponent;
```

위 코드에는 다음과 같은 문제점들이 있어요.<br />
- `unusedVariable`: 사용되지 않는 변수 (ESLint 경고)<br />
- `greeting = 'world'`: 작은따옴표 대신 큰따옴표 사용 (Prettier 규칙 위반, `singleQuote: true`일 경우)<br />
- `if(true)`: 중괄호와 조건문 사이의 공백 부족 (Prettier 규칙 위반)<br />
- `<p>    Test Component   </p>`: 불필요한 공백 (Prettier 규칙 위반)

### 1️⃣ 자동 교정 확인

위 파일을 저장해 보세요. VS Code 설정이 올바르게 되어 있다면, 파일 저장 시 자동으로 코드가 다음과 같이 수정될 거예요.

```diff-tsx showLineNumbers {2-3, 5, 8, 11-12}
--- a/src/app/page.tsx
+++ b/src/app/page.tsx
 // src/app/page.tsx (App Router 예시)
 const MyComponent = () => {
-  const unusedVariable = "Hello";
-  let greeting = 'world';
+  // const unusedVariable = "Hello"; // ESLint가 사용되지 않는 변수를 주석 처리하거나 삭제하도록 권고할 수 있어요.
+  let greeting = 'world'; // singleQuote 규칙에 따라 작은따옴표로 변경될 거예요。
 
-  if(true) {
+  if (true) { // 공백이 추가되고, Prettier 규칙에 따라 포맷팅될 거예요。
     console.log(greeting);
   }
 
   return (
     <div>
-      <p>    Test Component   </p>
+      <p>Test Component</p> {/* 불필요한 공백이 제거될 거예요。 */}
     </div>
   );
 };
 
 export default MyComponent;
```

<Blockquote type="success">
ESLint는 `unusedVariable` 같은 잠재적 오류를 경고하고, `eslint --fix`를 실행하면 해당 라인을 삭제하거나 주석 처리하는 식으로 자동 수정해요.<br />
Prettier는 `greeting` 변수의 따옴표, `if` 문의 공백, `<p>` 태그 내부의 불필요한 공백 등을 `.prettierrc.json` 설정에 따라 자동으로 교정해 줍니다.
</Blockquote>

이처럼 ESLint와 Prettier를 함께 사용하면 개발자가 수동으로 스타일을 맞추거나 오류를 찾는 데 드는 시간을 절약하고, 더 중요한 로직 개발에 집중할 수 있게 된답니다.

처음에는 설정이 다소 복잡하게 느껴질 수도 있지만, 한 번 올바르게 설정해두면 장기적으로 엄청난 생산성 향상을 가져다줄 거예요.

### 0️⃣ 핵심 요약

-   **ESLint**: 코드 품질을 검사하고 잠재적 오류를 찾아내요.
-   **Prettier**: 코드 스타일을 일관되게 포맷팅해요.
-   **함께 사용**: `eslint-config-prettier`로 충돌을 방지하고, `eslint-plugin-prettier`로 Prettier 규칙을 ESLint 오류로 보고해요.
-   **VS Code 연동**: `formatOnSave` 및 `codeActionsOnSave` 설정을 통해 저장 시 자동 포맷팅 및 오류 수정을 활성화하여 개발 경험을 극대화할 수 있어요.
