> Next.js 15와 React 19의 등장으로 웹 애플리케이션의 캐싱 전략은 더욱 중요해지고 복잡해졌어요. 이 글에서는 Next.js 15의 새로운 캐싱 메커니즘과 React Server Components(RSC)와의 연계를 통해 어떻게 효율적으로 데이터를 관리하고 성능을 극대화할 수 있는지 심층적으로 알아보아요.<br />

## 🤔 문제/배경
### 0️⃣ 왜 이 주제를 다루는가
Next.js는 App Router 도입 이후 데이터 가져오기(Data Fetching)와 캐싱 전략에 있어 큰 변화를 겪었어요. 특히 Next.js 15와 함께 발표된 React 19는 React Compiler를 비롯한 다양한 개선 사항을 포함하고 있으며, 이는 기존의 캐싱 접근 방식에도 영향을 미쳐요. 많은 개발자들이 App Router 환경에서 데이터가 언제, 어디서, 어떻게 캐시되는지에 대해 혼란을 겪고 있으며, 실무에서 예측 불가능한 데이터 상태로 인해 어려움을 느끼는 경우가 많아요.<br />
이러한 배경 속에서 Next.js 15의 강화된 캐싱 전략을 정확히 이해하고 적용하는 것은 웹 애플리케이션의 성능을 극대화하고 사용자 경험을 향상시키는 데 필수적인 요소가 되었어요.

### 1️⃣ 기존 방식의 한계
App Router 이전, 또는 App Router 초기에는 캐싱 전략이 다소 파편화되어 있었어요. `getServerSideProps`나 `getStaticProps`를 통해 데이터를 가져오고 캐싱을 제어했지만, 이는 페이지 단위에 국한되었어요.<br />
`fetch` API를 직접 사용할 때도 기본 캐싱 동작이 있었지만, Server Components와 Client Components 간의 데이터 흐름이 복잡해지면서 일관된 캐싱 전략을 적용하기 어려웠어요. 예를 들어, 한 번 가져온 데이터가 의도치 않게 오래된 상태로 유지되거나, 반대로 매 요청마다 데이터를 다시 가져와 불필요한 네트워크 트래픽과 서버 부하를 유발하는 경우가 발생했어요. <br />
또한, React Server Components가 도입되면서 서버에서 데이터를 가져오는 방식이 더욱 강력해졌지만, 이 데이터가 클라이언트에 어떻게 전달되고 재사용되는지에 대한 명확한 이해 없이는 오히려 성능 저하를 초래할 수도 있었죠.

<Blockquote type="warning">
  **주의:** App Router에서 `fetch`는 브라우저의 `fetch`와 다르게 동작하며, Next.js 자체의 캐싱 레이어를 거치게 돼요. 이 점을 간과하면 예상치 못한 캐싱 이슈가 발생할 수 있어요.
</Blockquote>

## ⚙️ 해결 방법
### 0️⃣ 핵심 아이디어
Next.js 15는 `fetch` API를 확장하여 더욱 강력하고 세밀한 캐싱 제어 기능을 제공해요. 핵심 아이디어는 React Server Components의 **데이터 캐시(Data Cache)**와 **Full Route Cache**, 그리고 **`fetch` 요청에 대한 캐시 동작 제어**를 통합하여, 개발자가 명시적으로 캐싱 전략을 정의할 수 있도록 하는 것이에요.<br />
이러한 캐싱 메커니즘은 정적(static) 데이터와 동적(dynamic) 데이터 모두에 적용될 수 있으며, 빌드 시점, 요청 시점, 그리고 일정 시간 간격으로 데이터를 재검증(revalidation)할 수 있는 유연성을 제공해요.

<Blockquote type="tip">
  **팁:** Next.js 15의 캐싱 전략은 React 19의 `React.cache`와 같은 새로운 API들을 활용하여 Server Components 내부에서 데이터 중복 페칭을 방지하고 캐시 일관성을 유지하는 데 큰 도움이 돼요.
</Blockquote>

### 1️⃣ 적용 방법
Next.js 15에서는 `fetch` 옵션을 통해 캐싱 동작을 명확하게 제어할 수 있어요. 주요 옵션은 다음과 같아요.<br />

*   **`cache: 'force-cache'` (기본값):** fetch 결과를 Next.js Data Cache에 캐시하고, 동일한 요청이 들어오면 캐시된 데이터를 사용해요. 빌드 시점 또는 최초 요청 시점에 캐시되며, 명시적으로 재검증하지 않으면 계속 유지돼요.
*   **`cache: 'no-store'`:** fetch 결과를 캐시하지 않고, 항상 최신 데이터를 서버에서 가져와요.
*   **`next: { revalidate: number | false }`:** 캐시된 데이터의 유효 기간을 초 단위로 설정해요. `0`은 `cache: 'no-store'`와 유사하게 동작하여 항상 최신 데이터를 가져와요. `false`는 캐시된 데이터가 절대 재검증되지 않도록 해요 (`cache: 'force-cache'`와 함께 사용 시 무기한 캐시로 동작). `number`는 지정된 시간 간격으로 캐시를 재검증하도록 해요.
*   **`next: { tags: string[] }`:** 특정 캐시 항목에 태그를 부여하여, 나중에 `revalidateTag()` 함수를 통해 해당 태그를 가진 모든 캐시를 수동으로 재검증할 수 있도록 해요.

또한, `React.cache` API를 사용하여 서버 컴포넌트 내부에서 동일한 데이터 호출에 대한 중복 실행을 방지할 수 있어요.

<br />

<Blockquote type="info">
  **정보:** `fetch` 외에도, `route segment config` (예: `export const dynamic = 'force-dynamic'`)를 통해 전체 경로 세그먼트의 캐싱 동작을 설정할 수 있어요. 이는 해당 세그먼트 내의 모든 `fetch` 요청에 영향을 줘요.
</Blockquote>

## 🧪 예시
### 0️⃣ 코드/설정 예시

가장 일반적인 캐싱 시나리오를 코드로 살펴보아요.<br />

**1. 정적 데이터 캐싱 (기본 동작)**<br />
별도의 옵션이 없다면 `fetch`는 기본적으로 `cache: 'force-cache'`로 동작하며, 빌드 시 또는 최초 요청 시 캐시돼요. 이는 정적인 블로그 포스트 목록과 같이 자주 변경되지 않는 데이터에 적합해요.<br />

```tsx showLineNumbers {5-7}
// app/posts/page.tsx (Server Component)
import PostList from './_components/post-list';

async function getPosts() {
  const res = await fetch('https://api.example.com/posts');
  // 기본적으로 { cache: 'force-cache' }와 동일하게 동작해요.
  // 이 데이터는 빌드 시 또는 최초 요청 시 캐시되며,
  // 명시적 revalidate 없이는 계속 캐시된 데이터를 사용해요.
  if (!res.ok) {
    throw new Error('Failed to fetch posts');
  }
  return res.json();
}

export default async function PostsPage() {
  const posts = await getPosts();
  return (
    <div>
      <h1>블로그 포스트</h1>
      <PostList posts={posts} />
    </div>
  );
}
```
<br />

**2. 동적 데이터 재검증 (ISR과 유사)**<br />
특정 주기마다 데이터를 갱신해야 하는 경우 `next: { revalidate: number }` 옵션을 사용해요. 이는 Incremental Static Regeneration(ISR)과 유사하게 동작하여, 지정된 시간(초)이 지나면 백그라운드에서 데이터를 다시 가져와 캐시를 갱신해요.<br />

```tsx showLineNumbers {3, 6-8}
// app/products/[id]/page.tsx (Server Component)
interface ProductPageProps {
  params: { id: string };
}

async function getProduct(id: string) {
  const res = await fetch(`https://api.example.com/products/${id}`, {
    next: { revalidate: 60 }, // 60초마다 데이터 재검증 (ISR과 유사)
  });
  if (!res.ok) {
    throw new Error('Failed to fetch product');
  }
  return res.json();
}

export default async function ProductPage({ params }: ProductPageProps) {
  const product = await getProduct(params.id);
  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <span>가격: {product.price}원</span>
    </div>
  );
}
```
<br />

**3. 항상 최신 데이터 가져오기 (`no-store`)**<br />
매번 최신 데이터가 필요한 경우 (예: 사용자 장바구니, 인증 정보) `cache: 'no-store'` 옵션을 사용해요. 이 경우 데이터는 캐시되지 않고, 매 요청마다 서버에서 새로 가져오게 돼요.<br />

```tsx showLineNumbers {5-7}
// app/cart/page.tsx (Server Component)
import CartDisplay from './_components/cart-display';

async function getUserCart(userId: string) {
  const res = await fetch(`https://api.example.com/users/${userId}/cart`, {
    cache: 'no-store', // 항상 최신 데이터를 가져와요.
  });
  if (!res.ok) {
    throw new Error('Failed to fetch user cart');
  }
  return res.json();
}

export default async function CartPage() {
  // 실제 userId는 인증 컨텍스트에서 가져와야 해요.
  const userId = 'some-user-id'; 
  const cart = await getUserCart(userId);
  return (
    <div>
      <h1>내 장바구니</h1>
      <CartDisplay cart={cart} />
    </div>
  );
}
```
<br />

**4. 수동 재검증 (Tags)**<br />
특정 태그를 부여하고 `revalidateTag()` 함수를 통해 수동으로 캐시를 갱신할 수 있어요. 이는 CMS 업데이트나 관리자 페이지에서 데이터가 변경되었을 때 유용해요.<br />

```tsx showLineNumbers {5-7, 10-15}
// app/admin/products/page.tsx (Server Component)
import { revalidateTag } from 'next/cache';

async function getAdminProducts() {
  const res = await fetch('https://api.example.com/admin/products', {
    next: { tags: ['admin-products'] }, // 'admin-products' 태그 부여
  });
  if (!res.ok) {
    throw new Error('Failed to fetch admin products');
  }
  return res.json();
}

// 이 함수는 API Route (Route Handler)나 Server Action에서 호출될 수 있어요.
export async function revalidateAdminProducts() {
  'use server'; // Server Action 예시
  revalidateTag('admin-products'); // 'admin-products' 태그를 가진 캐시를 재검증해요.
  console.log('Admin products revalidated!');
}

export default async function AdminProductsPage() {
  const products = await getAdminProducts();
  return (
    <div>
      <h1>관리자용 상품 목록</h1>
      {/* ... 상품 목록 표시 및 revalidate 버튼 등 ... */}
    </div>
  );
}
```

**5. `React.cache`를 활용한 중복 호출 방지**<br />
하나의 요청-응답 주기 내에서 동일한 데이터를 여러 Server Component에서 호출할 때 `React.cache`를 사용하여 실제 네트워크 요청을 한 번만 보내도록 최적화할 수 있어요.<br />

```tsx showLineNumbers {5-10, 13-17, 20-24}
// lib/data.ts
import { cache } from 'react';

// `cache`로 감싸서, 동일한 함수 호출 시 캐시된 결과를 반환해요.
export const getCachedUserData = cache(async (userId: string) => {
  console.log(`Fetching user data for ${userId}...`); // 이 로그는 한 번만 찍혀요.
  const res = await fetch(`https://api.example.com/users/${userId}`);
  if (!res.ok) throw new Error('Failed to fetch user');
  return res.json();
});

// app/dashboard/page.tsx
import { getCachedUserData } from '@/lib/data';
import UserProfile from './_components/UserProfile';
import UserSettings from './_components/UserSettings';

export default async function DashboardPage() {
  const userId = 'user-123';
  const user = await getCachedUserData(userId); // 첫 번째 호출
  
  return (
    <div>
      <h1>대시보드</h1>
      <UserProfile user={user} />
      <UserSettings user={user} /> {/* 두 번째 호출이지만, 실제 fetch는 일어나지 않아요. */}
    </div>
  );
}
```

### 1️⃣ 적용 결과
이러한 캐싱 전략을 적용하면 다음과 같은 효과를 얻을 수 있어요.<br />

*   **성능 향상:** 불필요한 네트워크 요청을 줄여 페이지 로딩 속도를 단축시키고 서버 부하를 경감시켜요.<br />
*   **비용 절감:** API 호출 횟수를 줄여 클라우드 서비스 비용을 절약할 수 있어요.<br />
*   **향상된 사용자 경험:** 빠르게 데이터를 제공하여 사용자가 더 쾌적하게 애플리케이션을 이용할 수 있도록 해요.<br />
*   **일관된 데이터:** 명시적인 재검증 전략을 통해 데이터의 신선도와 일관성을 유지할 수 있어요.<br />

  **성공:** Next.js 15의 강화된 캐싱 전략을 잘 활용하면, Server Components와 Client Components가 조화롭게 동작하는 고성능 웹 애플리케이션을 구축할 수 있어요!

## 📝 정리
### 0️⃣ 핵심 요약
Next.js 15는 `fetch` API의 확장된 캐싱 옵션 (`cache`, `next: { revalidate, tags }`)과 React 19의 `React.cache`를 통해 데이터 캐싱을 더욱 강력하고 유연하게 만들었어요. 개발자는 이제 정적 및 동적 데이터에 대한 캐싱 정책을 명시적으로 제어하여, 빌드 시점 캐싱, 요청 시점 캐싱, 주기적 재검증, 수동 재검증 등 다양한 시나리오에 대응할 수 있게 되었어요. 이는 App Router 환경에서 웹 애플리케이션의 성능과 데이터 일관성을 크게 향상시키는 핵심적인 요소예요.
