## 🤔 Next.js 미들웨어, 왜 필요한가요?

### 0️⃣ 미들웨어란 무엇인가요? 🧐

미들웨어는 클라이언트의 요청(Request)이 서버에 도달하기 전이나, 서버의 응답(Response)이 클라이언트에 전달되기 전에 중간에서 특정 작업을 처리하는 함수를 의미해요. 마치 공항의 보안 검사대나 세관처럼, 요청과 응답 흐름 사이에 끼어들어 필요한 검증, 변환, 라우팅 등의 작업을 수행하는 것이죠.<br />

Next.js 미들웨어는 이러한 역할을 웹 애플리케이션의 엣지(Edge) 환경에서 수행하여, 매우 빠르고 효율적인 요청 처리를 가능하게 해요. 덕분에 사용자 경험을 개선하고, 서버 부하를 줄이는 데 크게 기여할 수 있어요.

### 1️⃣ App Router 시대의 미들웨어 🏞️

Next.js 13부터 도입된 App Router는 서버 컴포넌트(Server Components)와 클라이언트 컴포넌트(Client Components)의 구분을 통해 새로운 패러다임을 제시했죠. 이 환경에서 미들웨어는 서버 컴포넌트보다 먼저 실행되어, 요청에 대한 초기 판단과 처리를 담당해요.<br />

예를 들어, 특정 페이지에 접근하기 전에 사용자가 로그인했는지 확인하거나, 사용자의 언어 설정에 따라 다른 페이지로 리다이렉션해야 할 때 미들웨어가 빛을 발해요. 모든 요청에 대해 반복적으로 수행되어야 하는 공통 로직을 한 곳에 모아 관리함으로써 코드의 응집도를 높이고 유지보수성을 향상시킬 수 있답니다.

## 🛠️ 미들웨어 구성 및 기본 활용법

### 0️⃣ 미들웨어 파일 생성 📁

Next.js에서 미들웨어를 사용하려면 프로젝트의 루트 디렉터리에 `middleware.ts` (또는 `middleware.js`) 파일을 생성해야 해요. 이 파일은 Next.js가 자동으로 감지하고 미들웨어로 등록한답니다.<br />

기본적인 미들웨어의 형태는 다음과 같아요.

```tsx showLineNumbers
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  // 요청 URL을 콘솔에 출력해요.
  console.log("Request URL:", request.url);

  // 모든 요청을 그대로 통과시켜요.
  return NextResponse.next();
}

// 미들웨어를 특정 경로에만 적용하려면 matcher 설정을 사용해요.
// 여기에 설정된 경로에만 미들웨어가 실행된답니다.
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - api (API routes)
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico (favicon file)
     */
    "/((?!api|_next/static|_next/image|favicon.ico).*)",
  ],
};
```

<Blockquote type="tip">
  `NextRequest`와 `NextResponse`는 Next.js가 제공하는 확장된 Request/Response
  객체로, HTTP 헤더, 쿠키, URL 조작 등 미들웨어에서 필요한 다양한 기능을
  제공해요.
  <br />
  일반적인 웹 표준 `Request`와 `Response` 객체보다 더 많은 편의 기능을 가지고
  있어 미들웨어 개발에 매우 유용하답니다.
</Blockquote>

### 1️⃣ 기본 미들웨어 설정: Matcher 🎯

모든 요청에 미들웨어를 적용하는 것은 비효율적일 수 있어요. `config.matcher`를 사용하면 미들웨어를 특정 경로에만 적용하여 불필요한 실행을 방지할 수 있답니다.<br />
위 예시의 `matcher`는 API 라우트, Next.js 내부 파일, 파비콘 등을 제외한 모든 경로에 미들웨어를 적용하도록 설정되어 있어요. 정규식을 사용하여 매우 유연하게 경로를 지정할 수 있어요.

<Blockquote type="info">
  `matcher`는 정적(static)으로 분석되기 때문에 런타임에 동적으로 변경될 수
  없어요. 즉, 조건부 로직을 이용해 `matcher` 배열을 변경하는 것은 불가능하다는
  점을 기억해 주세요.
</Blockquote>

### 2️⃣ Next.js 14.1+에서 NextResponse 활용 ✨

Next.js 14.1 버전부터는 `NextResponse` 객체를 통한 다양한 응답 조작 기능이 더욱 강화되었어요. 리다이렉션, 리라이트, 헤더 설정, 쿠키 설정 등을 미들웨어에서 효율적으로 처리할 수 있답니다.

```tsx showLineNumbers {11-17, 20-22}
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const url = request.nextUrl;

  // 특정 경로로 리다이렉션하는 예시
  if (url.pathname === "/old-page") {
    // /old-page로 접근하면 /new-page로 이동시켜요.
    return NextResponse.redirect(new URL("/new-page", request.url));
  }

  // 특정 경로로 리라이트(rewrite)하는 예시
  // URL은 변경되지 않지만, 실제 보여주는 콘텐츠는 /another-page의 것이 돼요.
  if (url.pathname === "/some-internal-path") {
    return NextResponse.rewrite(new URL("/another-page", request.url));
  }

  // 요청 헤더를 조작하는 예시
  const response = NextResponse.next();
  response.headers.set("X-Custom-Header", "Hello from Middleware");
  return response;
}

export const config = {
  matcher: ["/old-page", "/some-internal-path", "/any-other-path"],
};
```

## 🚀 실전 미들웨어 활용 예제

이제 실제 시나리오에서 미들웨어가 어떻게 활용될 수 있는지 구체적인 예제들을 살펴볼까요?

### 0️⃣ 사용자 인증/인가 처리 🔐

가장 흔하게 사용되는 미들웨어의 역할 중 하나는 사용자 인증(Authentication)과 인가(Authorization) 처리예요. 로그인하지 않은 사용자가 특정 관리자 페이지에 접근하는 것을 막고 로그인 페이지로 리다이렉션하는 로직을 미들웨어에서 구현할 수 있어요.

```tsx showLineNumbers {13-22, 25-27}
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const isAuthenticated = request.cookies.has("session_token"); // 세션 토큰 유무로 인증 여부 판단
  const url = request.nextUrl;

  // 보호된 경로 목록을 정의해요.
  const protectedPaths = ["/dashboard", "/admin", "/profile"];

  // 현재 경로가 보호된 경로 중 하나인지 확인해요.
  if (protectedPaths.some((path) => url.pathname.startsWith(path))) {
    // 인증되지 않은 사용자라면 로그인 페이지로 리다이렉션해요.
    if (!isAuthenticated) {
      const loginUrl = new URL("/login", url.origin);
      loginUrl.searchParams.set("redirect", url.pathname + url.search);
      return NextResponse.redirect(loginUrl);
    }
    // 인증된 사용자라면 다음 단계로 진행해요.
    return NextResponse.next();
  }

  // 그 외의 경로는 모두 통과시켜요.
  return NextResponse.next();
}

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico|login).*)"], // 로그인 페이지는 미들웨어에서 제외해요.
};
```

<Blockquote type="warning">
  실제 프로덕션 환경에서는 세션 토큰의 유효성을 서버에서 검증하는 과정이
  필요해요. 미들웨어에서는 간단한 토큰 유무 확인으로 예시를 들었지만, 보안
  강화를 위해 JWT 검증 등의 로직을 추가해야 한답니다.
</Blockquote>

### 1️⃣ 국제화(i18n) 라우팅 🌍

다국어 웹사이트를 구축할 때, 사용자의 선호 언어나 브라우저 설정에 따라 적절한 언어 버전의 페이지로 라우팅하는 것도 미들웨어의 강력한 활용 사례예요. Next.js 13+ App Router에서는 `i18n` 설정을 `next.config.js`에서 제거하고 미들웨어에서 직접 처리하는 방식을 권장하고 있어요.

```tsx showLineNumbers {10-18, 20-30, 32-34}
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

const PUBLIC_FILE_REGEX = /\.(.*)$/;
const i18n = {
  defaultLocale: "en",
  locales: ["en", "ko", "ja"],
};

function getLocale(request: NextRequest): string | undefined {
  // 1. 쿠키에서 언어 설정을 확인해요.
  const cookieLocale = request.cookies.get("NEXT_LOCALE")?.value;
  if (cookieLocale && i18n.locales.includes(cookieLocale)) {
    return cookieLocale;
  }

  // 2. Accept-Language 헤더에서 언어 설정을 확인해요.
  const acceptLanguageHeader = request.headers.get("Accept-Language");
  if (acceptLanguageHeader) {
    const preferredLocale = acceptLanguageHeader
      .split(",")
      .map((lang) => lang.split(";")[0].trim())
      .find((lang) => i18n.locales.includes(lang));
    if (preferredLocale) return preferredLocale;
  }

  // 3. 기본 언어를 반환해요.
  return i18n.defaultLocale;
}

export function middleware(request: NextRequest) {
  const pathname = request.nextUrl.pathname;

  // 정적 파일, Next.js 내부 경로, API 경로 등은 미들웨어에서 제외해요.
  if (
    pathname.startsWith("/_next") ||
    pathname.startsWith("/api") ||
    pathname.startsWith("/static") ||
    PUBLIC_FILE_REGEX.test(pathname)
  ) {
    return NextResponse.next();
  }

  // 현재 경로에 언어 프리픽스가 없는 경우
  const pathnameHasLocale = i18n.locales.some(
    (locale) => pathname.startsWith(`/${locale}/`) || pathname === `/${locale}`,
  );

  if (!pathnameHasLocale) {
    const locale = getLocale(request);
    // 사용자의 선호 언어에 따라 리다이렉션해요.
    const newUrl = new URL(
      `/${locale}${pathname === "/" ? "" : pathname}`,
      request.url,
    );
    return NextResponse.redirect(newUrl);
  }

  // 이미 언어 프리픽스가 있는 경우, 그대로 통과시켜요.
  return NextResponse.next();
}

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
};
```

<Blockquote type="info">
  이 예제는 URL에 `/en`, `/ko`와 같은 언어 프리픽스를 추가하는 방식이에요. 실제
  구현 시에는 `i18next`나 `react-i18next` 같은 라이브러리와 함께 사용하여 번역된
  콘텐츠를 제공해야 한답니다.
</Blockquote>

### 2️⃣ A/B 테스트 및 Feature Toggling 🧪

미들웨어는 특정 사용자 그룹에게만 새로운 기능을 노출하거나, 다른 버전의 UI를 보여주는 A/B 테스트나 기능 토글(Feature Toggling)에도 활용될 수 있어요. 사용자 ID, 쿠키, 또는 요청 헤더 등의 정보를 기반으로 동적으로 라우팅을 변경할 수 있답니다.

```tsx showLineNumbers {10-21, 24-27}
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const url = request.nextUrl;

  // A/B 테스트를 위한 쿠키가 있는지 확인해요.
  let abTestGroup = request.cookies.get("ab-test-group")?.value;

  if (!abTestGroup) {
    // 쿠키가 없다면, 랜덤으로 A 또는 B 그룹을 할당해요.
    abTestGroup = Math.random() < 0.5 ? "A" : "B";
    const response = NextResponse.next();
    // 할당된 그룹을 쿠키에 저장하여 다음 요청에도 유지되도록 해요.
    response.cookies.set("ab-test-group", abTestGroup, { path: "/" });
    // 중요한 점: 쿠키를 설정한 후에는 응답 객체를 반환해야 해요.
    return response;
  }

  // 'feature-x' 경로에 대한 A/B 테스트를 수행해요.
  if (url.pathname === "/feature-x") {
    if (abTestGroup === "B") {
      // B 그룹 사용자에게는 다른 버전의 페이지를 보여줘요.
      return NextResponse.rewrite(new URL("/feature-x-variant-b", request.url));
    }
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/feature-x"],
};
```

## 💡 미들웨어 개발 팁과 주의사항

### 0️⃣ 미들웨어 실행 환경 이해 🧠

Next.js 미들웨어는 Node.js 런타임이 아닌, Vercel Edge Runtime과 같은 **엣지 런타임(Edge Runtime)**에서 실행돼요. 이는 Node.js API (예: `fs`, `process.env`의 모든 변수)를 직접 사용할 수 없다는 것을 의미해요. 환경 변수(`process.env.MY_VARIABLE`)는 사용 가능하지만, 파일 시스템 접근 같은 Node.js 특정 기능은 제한돼요.<br />
따라서 미들웨어 로직을 작성할 때는 이러한 제약을 인지하고, 가벼우면서도 효율적인 코드를 작성하는 것이 중요해요.

<LinkPreviewCard href="https://nextjs.org/docs/app/building-your-application/routing/middleware#edge-runtime" />

### 1️⃣ 성능 최적화와 캐싱 ⚡

미들웨어는 모든 요청에 대해 실행될 수 있으므로 성능에 매우 큰 영향을 줄 수 있어요. 불필요한 연산을 최소화하고, 가능하다면 캐싱 전략을 고려해야 해요.<br />
예를 들어, 사용자 인증 정보가 자주 변경되지 않는다면, 세션 토큰을 쿠키에 저장하고 미들웨어에서는 쿠키 유무만 빠르게 확인하는 것이 효율적이에요. 복잡한 데이터베이스 조회나 외부 API 호출은 미들웨어에서 지양하고, 서버 컴포넌트나 API 라우트에서 처리하는 것이 좋답니다.

### 2️⃣ 에러 처리 전략 🚨

미들웨어에서 에러가 발생하면 사용자에게 빈 페이지나 에러 메시지가 노출될 수 있어요. 미들웨어 내부에서 발생할 수 있는 잠재적인 에러를 예측하고, 적절한 에러 처리 로직을 구현해야 해요.<br />
예를 들어, 외부 서비스 호출에 실패했을 때 대체 로직을 수행하거나, 특정 에러 발생 시 사용자에게 친화적인 에러 페이지로 리다이렉션하는 등의 전략을 사용할 수 있어요.

```tsx showLineNumbers {5-12}
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  try {
    // 미들웨어 로직
    // ...
    return NextResponse.next();
  } catch (error) {
    console.error("Middleware error:", error);
    return NextResponse.redirect(new URL("/error", request.url)); // 에러 발생 시 에러 페이지로 리다이렉션
  }
}

export const config = {
  matcher: ["/protected-path"],
};
```

### 0️⃣ 핵심 요약 🎯

오늘은 Next.js App Router 환경에서 미들웨어를 활용하는 다양한 방법을 알아보았어요. 미들웨어는 요청 처리 흐름의 '관문' 역할을 하며, 사용자 인증, 국제화, A/B 테스트 등 애플리케이션 전반에 걸쳐 공통적으로 필요한 로직을 효과적으로 중앙 관리할 수 있게 해준답니다.<br />

- **미들웨어의 역할:** 요청이 서버에 도달하기 전/후에 로직을 실행하여 요청 흐름을 제어해요.<br />
- **주요 기능:** `NextResponse.redirect()`, `NextResponse.rewrite()`, 헤더/쿠키 조작 등 다양한 기능을 제공해요.<br />
- **활용 예시:** 사용자 인증/인가, 국제화 라우팅, A/B 테스트 및 기능 토글 등에 유용해요.<br />
- **주의사항:** 엣지 런타임 환경의 제약사항을 이해하고, 성능과 에러 처리에 유의해야 해요.
