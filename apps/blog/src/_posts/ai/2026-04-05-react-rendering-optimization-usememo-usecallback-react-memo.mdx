## 🤔 왜 React 렌더링 최적화가 중요할까요?

### 0️⃣ 불필요한 렌더링은 성능 저하의 주범이에요

React는 선언적인 UI를 구축하는 데 탁월한 라이브러리지만, 컴포넌트의 상태(state)나 속성(props)이 변경되면 해당 컴포넌트는 물론, 그 하위 컴포넌트들까지 재렌더링될 수 있어요.<br />
작은 애플리케이션에서는 큰 문제가 되지 않지만, 복잡하고 규모가 큰 애플리케이션에서는 불필요한 재렌더링이 반복될 경우 사용자 경험에 직접적인 영향을 미치는 성능 저하로 이어질 수 있답니다.<br />
특히 데이터가 많거나 복잡한 연산을 수행하는 컴포넌트가 빈번하게 재렌더링된다면, 애플리케이션이 느려지고 버벅거리는 현상을 쉽게 목격할 수 있을 거예요.

### 1️⃣ React의 렌더링 원리를 이해해야 해요

React는 Virtual DOM을 사용하여 효율적으로 UI를 업데이트해요.<br />
어떤 컴포넌트가 렌더링되면, React는 새로운 Virtual DOM 트리를 생성하고 이전 Virtual DOM 트리와 비교(Diffing)해서 실제 DOM에 필요한 최소한의 변경 사항만 적용(Reconciliation)하죠.<br />
이 과정 자체는 매우 효율적이지만, Diffing 과정 자체도 비용이 들고, 무엇보다 컴포넌트 함수 자체가 다시 호출되는 것(렌더링)은 피할 수 없어요.<br />
따라서 불필요한 컴포넌트 함수의 호출 횟수를 줄이는 것이 렌더링 최적화의 핵심 목표가 된답니다.

## ⚙️ 렌더링 최적화의 핵심 도구: useMemo, useCallback, React.memo

React에서 불필요한 렌더링을 방지하고 성능을 향상시키기 위한 세 가지 강력한 도구가 있어요. 바로 `useMemo`, `useCallback`, 그리고 `React.memo`입니다.<br />
이 세 가지는 모두 "메모이제이션(Memoization)"이라는 개념을 기반으로 해요. 메모이제이션은 이전에 계산한 값을 저장해두었다가 동일한 입력이 들어오면 다시 계산하지 않고 저장된 값을 반환하는 최적화 기법을 말해요.

### 0️⃣ `useMemo`: 값(Value)의 메모이제이션 💡

`useMemo` 훅은 함수형 컴포넌트 내부에서 특정 "값"의 연산 결과를 메모이제이션할 때 사용해요.<br />
복잡한 계산이나 큰 객체 생성이 필요한 경우, 의존성 배열에 있는 값이 변경되지 않는 한 이전에 계산된 값을 재사용하여 불필요한 연산을 방지할 수 있어요.

#### 📝 `useMemo` 사용법

```tsx showLineNumbers
import React, { useState, useMemo } from 'react';

interface Item {
  id: number;
  name: string;
  price: number;
  quantity: number;
}

function ShoppingCart() {
  const [items, setItems] = useState<Item[]>([
    { id: 1, name: 'Laptop', price: 1200, quantity: 1 },
    { id: 2, name: 'Mouse', price: 25, quantity: 2 },
    { id: 3, name: 'Keyboard', price: 75, quantity: 1 },
  ]);
  const [shippingFee, setShippingFee] = useState(50);

  // useMemo를 사용하여 총 가격을 메모이제이션해요.
  // items 배열이나 shippingFee가 변경될 때만 다시 계산해요.
  const totalCost = useMemo(() => {
    console.log('총 가격을 다시 계산해요...');
    const subtotal = items.reduce((acc, item) => acc + item.price * item.quantity, 0);
    return subtotal + shippingFee;
  }, [items, shippingFee]); // 의존성 배열

  const addItem = () => {
    setItems(prevItems => [
      ...prevItems,
      { id: prevItems.length + 1, name: `New Item ${prevItems.length + 1}`, price: 100, quantity: 1 },
    ]);
  };

  const changeShippingFee = () => {
    setShippingFee(prevFee => prevFee + 10);
  };

  const handleTestRender = () => {
    console.log('렌더링 테스트 버튼 클릭됨');
    // 이 버튼은 상태를 변경하지 않으므로, totalCost는 다시 계산되지 않아요.
  };

  return (
    <div>
      <h1>장바구니</h1>
      <ul>
        {items.map(item => (
          <li key={item.id}>
            {item.name} - ${item.price} x {item.quantity}
          </li>
        ))}
      </ul>
      <p>배송비: ${shippingFee}</p>
      <p>총 결제 금액: ${totalCost}</p> {/* totalCost는 useMemo로 메모이제이션된 값이에요 */}
      <button onClick={addItem}>아이템 추가</button>
      <button onClick={changeShippingFee}>배송비 변경</button>
      <button onClick={handleTestRender}>렌더링 테스트</button>
    </div>
  );
}

export default ShoppingCart;
```

위 코드에서 `totalCost`는 `items` 배열이나 `shippingFee`가 변경될 때만 다시 계산돼요.<br />
만약 `addItem` 버튼을 눌러 `items`가 변경되면 `totalCost`는 다시 계산되지만, `changeShippingFee` 버튼을 눌러 `shippingFee`만 변경되거나, `렌더링 테스트` 버튼을 눌러 컴포넌트가 불필요하게 재렌더링되더라도 `items`와 `shippingFee`가 변경되지 않았다면 `totalCost`는 다시 계산되지 않고 이전에 메모이제이션된 값을 사용해요.<br />
콘솔 로그를 통해 이를 확인할 수 있답니다.

<Blockquote type="warning">
**주의사항:** `useMemo`는 남용하면 오히려 성능을 저하시킬 수 있어요.<br />
메모이제이션 자체에도 비용(메모리 사용, 의존성 비교)이 들기 때문에, 계산 비용이 적은 값에 `useMemo`를 사용하는 것은 오히려 손해일 수 있어요.<br />
정말로 복잡하고 비용이 많이 드는 계산에만 적용하는 것이 현명해요.
</Blockquote>

### 1️⃣ `useCallback`: 함수(Function)의 메모이제이션 🚀

`useCallback` 훅은 함수형 컴포넌트 내부에서 특정 "함수" 자체를 메모이제이션할 때 사용해요.<br />
React 컴포넌트가 렌더링될 때마다 컴포넌트 내부의 함수들은 매번 새로 생성돼요. 이는 특히 자식 컴포넌트에 props로 함수를 넘겨줄 때 문제가 될 수 있어요.

#### 📝 `useCallback` 사용법

```tsx showLineNumbers
import React, { useState, useCallback, memo } from 'react';

// 자식 컴포넌트
// React.memo를 사용하여 props가 변경되지 않으면 재렌더링을 방지해요.
const ChildComponent = memo(({ onClick }: { onClick: () => void }) => {
  console.log('ChildComponent가 렌더링되었어요.');
  return (
    <button onClick={onClick}>
      자식 버튼 클릭
    </button>
  );
});

function ParentComponent() {
  const [count, setCount] = useState(0);
  const [text, setText] = useState('');

  // handleClick 함수를 useCallback으로 메모이제이션해요.
  // count가 변경될 때만 함수가 새로 생성돼요.
  const handleClick = useCallback(() => {
    console.log(`버튼이 클릭되었어요. 현재 카운트: ${count}`);
    setCount(prevCount => prevCount + 1);
  }, [count]); // 의존성 배열에 count를 넣어주세요.

  const handleChange = useCallback((e: React.ChangeEvent<HTMLInputElement>) => {
    setText(e.target.value);
  }, []); // 의존성 배열이 비어있으면 컴포넌트가 마운트될 때 한 번만 생성돼요.

  return (
    <div>
      <h1>부모 컴포넌트</h1>
      <p>카운트: {count}</p>
      <input type="text" value={text} onChange={handleChange} placeholder="텍스트 입력" />
      <br /><br />
      <ChildComponent onClick={handleClick} /> {/* 메모이제이션된 함수를 전달해요 */}
      <button onClick={() => setCount(prevCount => prevCount + 1)}>카운트 증가</button>
      <button onClick={() => setText('새로운 텍스트')}>텍스트 변경</button>
    </div>
  );
}

export default ParentComponent;
```

`ParentComponent`가 렌더링될 때마다 `handleClick` 함수는 매번 새로 생성돼요.<br />
하지만 `useCallback`을 사용하면 `count` 값이 변경될 때만 `handleClick` 함수가 새로 생성되고, 그 외의 경우에는 이전에 생성된 함수를 재사용해요.<br />
`ChildComponent`는 `React.memo`로 감싸져 있기 때문에, `onClick` prop으로 전달되는 `handleClick` 함수가 변경되지 않는 한 재렌더링되지 않는답니다.<br />
만약 `useCallback`을 사용하지 않았다면, `ParentComponent`가 렌더링될 때마다 `handleClick` 함수가 새로 생성되고, 이는 `ChildComponent`의 `onClick` prop이 변경된 것으로 간주되어 `ChildComponent`도 불필요하게 재렌더링될 거예요.<br />
`text` 입력 필드의 값을 변경해도 `handleClick`은 새로 생성되지 않으므로 `ChildComponent`는 재렌더링되지 않음을 확인할 수 있어요.

<Blockquote type="tip">
**언제 `useCallback`을 사용해야 할까요?**<br />
주로 다음과 같은 경우에 `useCallback`을 고려해 볼 수 있어요.<br />
1. `React.memo`로 감싸진 자식 컴포넌트에 props로 함수를 전달할 때<br />
2. `useEffect`의 의존성 배열에 함수를 넣어야 할 때 (무한 루프 방지)<br />
3. `useContext`를 통해 함수를 전달할 때
</Blockquote>

### 2️⃣ `React.memo`: 컴포넌트(Component)의 메모이제이션 ♻️

`React.memo`는 고차 컴포넌트(Higher-Order Component, HOC)로, 래핑된 함수형 컴포넌트의 props가 변경되지 않았다면 해당 컴포넌트를 재렌더링하지 않도록 해줘요.<br />
이는 `shouldComponentUpdate` 라이프사이클 메서드를 함수형 컴포넌트에서 사용하는 것과 유사해요.

#### 📝 `React.memo` 사용법

위 `useCallback` 예시에서 `ChildComponent`를 `React.memo`로 감쌌던 것을 기억하시죠?<br />
`React.memo`의 기본 동작은 props의 "얕은 비교(shallow comparison)"를 수행해요.<br />
즉, 객체나 배열 같은 참조 타입의 prop은 내부 값이 변경되었더라도 참조 자체가 동일하면 변경되지 않은 것으로 간주한답니다.<br />
만약 깊은 비교가 필요하다면, `React.memo`의 두 번째 인자로 비교 함수를 전달할 수 있어요.

```tsx showLineNumbers
import React, { memo } from 'react';

interface User {
  id: number;
  name: string;
  email: string;
}

interface UserProfileProps {
  user: User;
  onEdit: () => void;
}

// React.memo로 UserProfile 컴포넌트를 감싸요.
// user 객체나 onEdit 함수가 변경되지 않으면 재렌더링되지 않아요.
const UserProfile = memo(({ user, onEdit }: UserProfileProps) => {
  console.log(`UserProfile 컴포넌트 렌더링: ${user.name}`);
  return (
    <div>
      <h2>{user.name}</h2>
      <p>이메일: {user.email}</p>
      <button onClick={onEdit}>수정</button>
    </div>
  );
});

// 커스텀 비교 함수를 사용하는 예시 (선택 사항)
interface ComplexData {
  value: number;
  config: {
    theme: string;
    level: number;
  };
}

interface ComplexComponentProps {
  data: ComplexData;
}

// data prop의 config 객체 내부까지 비교하고 싶다면 custom comparator를 사용해요.
const ComplexComponent = memo(({ data }: ComplexComponentProps) => {
  console.log('ComplexComponent 렌더링');
  return (
    <div>
      <p>Value: {data.value}</p>
      <p>Theme: {data.config.theme}</p>
    </div>
  );
}, (prevProps, nextProps) => {
  // 이전 props와 다음 props를 비교하여 true를 반환하면 렌더링 안 함, false를 반환하면 렌더링 함
  return (
    prevProps.data.value === nextProps.data.value &&
    prevProps.data.config.theme === nextProps.data.config.theme &&
    prevProps.data.config.level === nextProps.data.config.level
  );
});

export default function App() {
  const [currentUser, setCurrentUser] = React.useState<User>({
    id: 1, name: '블루', email: 'blue@example.com'
  });
  const [otherState, setOtherState] = React.useState(0);

  // onEdit 함수는 useCallback으로 메모이제이션하는 것이 좋아요.
  const handleEdit = React.useCallback(() => {
    console.log(`${currentUser.name}님 정보 수정`);
    // 수정 로직...
  }, [currentUser]); // currentUser가 변경될 때만 함수가 새로 생성돼요.

  const [complexData, setComplexData] = React.useState<ComplexData>({
    value: 10,
    config: { theme: 'dark', level: 1 }
  });

  return (
    <div>
      <h1>메인 애플리케이션</h1>
      <UserProfile user={currentUser} onEdit={handleEdit} /> {/* currentUser가 변경되지 않으면 UserProfile은 재렌더링되지 않아요 */}
      <button onClick={() => setOtherState(prev => prev + 1)}>다른 상태 변경 ({otherState})</button>
      <button onClick={() => setCurrentUser(prev => ({ ...prev, name: '새로운 블루' }))}>사용자 이름 변경</button>
      
      <hr />

      <ComplexComponent data={complexData} />
      <button onClick={() => setComplexData(prev => ({ ...prev, value: prev.value + 1 }))}>복잡 데이터 값 변경</button>
      <button onClick={() => setComplexData(prev => ({ ...prev, config: { ...prev.config, level: prev.config.level + 1 } }))}>복잡 데이터 레벨 변경</button>
    </div>
  );
}
```

`UserProfile` 컴포넌트는 `React.memo`로 감싸져 있어요.<br />
`otherState`를 변경하는 버튼을 눌러도 `UserProfile`의 `user` prop과 `onEdit` prop이 변경되지 않으므로, `UserProfile`은 재렌더링되지 않는 것을 콘솔 로그를 통해 확인할 수 있답니다.<br />
하지만 `사용자 이름 변경` 버튼을 누르면 `currentUser` 객체 자체가 새로 생성되어 `user` prop이 변경된 것으로 간주되고, `UserProfile`은 재렌더링돼요.<br />
`ComplexComponent`의 경우, `data` prop의 `config.level`만 변경해도 재렌더링이 일어나지 않는 것을 볼 수 있어요. 이는 커스텀 비교 함수를 통해 `level` 값까지 비교하기 때문이에요.

<Blockquote type="warning">
**주의사항:** `React.memo`를 사용할 때, props로 함수를 전달한다면 반드시 `useCallback`으로 해당 함수를 메모이제이션해야 해요.<br />
그렇지 않으면 부모 컴포넌트가 렌더링될 때마다 새로운 함수가 생성되고, `React.memo`는 prop이 변경되었다고 판단하여 자식 컴포넌트를 불필요하게 재렌더링하게 된답니다.<br />
또한, `React.memo`도 얕은 비교 비용이 발생하므로 모든 컴포넌트에 무분별하게 적용하기보다는 렌더링 비용이 큰 컴포넌트에 선별적으로 적용하는 것이 좋아요.
</Blockquote>

## 🧪 실전 렌더링 최적화 전략과 흔한 실수 🤦‍♀️

### 0️⃣ 언제, 어떻게 적용해야 할까요?

*   **컴포넌트 프로파일링:** React DevTools의 Profiler 탭을 사용하여 어떤 컴포넌트가 불필요하게 자주 렌더링되는지, 렌더링 비용이 높은지 파악하는 것이 중요해요.<br />
    병목 지점을 정확히 찾아낸 후 최적화를 시작해야 가장 큰 효과를 볼 수 있답니다.
*   **하위 컴포넌트에 대한 Props 안정화:** 부모 컴포넌트의 렌더링으로 인해 자식 컴포넌트가 불필요하게 렌더링되는 경우, 자식 컴포넌트를 `React.memo`로 감싸고, 자식에게 전달되는 props (특히 객체, 배열, 함수)를 `useMemo`나 `useCallback`으로 메모이제이션하여 안정화해야 해요.
*   **복잡한 계산 결과 메모이제이션:** 컴포넌트 내부에서 배열 필터링, 정렬, 큰 데이터 구조 변환 등 비용이 많이 드는 계산이 있다면 `useMemo`를 사용하여 결과를 캐싱하세요.

### 1️⃣ 흔히 저지르는 실수들

*   **불필요한 메모이제이션:** 모든 값이나 함수를 메모이제이션하는 것은 오히려 오버헤드만 증가시켜요.<br />
    메모이제이션 자체에도 비용이 들기 때문에, 성능 저하가 명확한 곳에만 선별적으로 적용해야 해요.
*   **잘못된 의존성 배열:** `useMemo`나 `useCallback`의 의존성 배열을 비워두거나, 필요한 의존성을 누락하면 오래된 클로저 값을 참조하거나 예상치 못한 버그를 유발할 수 있어요.<br />
    반대로 너무 많은 의존성을 넣으면 메모이제이션 효과가 사라질 수 있으니 주의해야 해요.
*   **객체/배열 prop의 불변성 유지 실패:** `React.memo`를 사용하더라도, props로 전달되는 객체나 배열이 매번 새로운 참조로 생성된다면 `React.memo`는 무용지물이 돼요.<br />
    항상 불변성(immutability)을 유지하며 새로운 객체나 배열을 생성하는 습관을 들여야 한답니다.
    ```tsx
    // 잘못된 예시: 매번 새로운 객체 생성
    <ChildComponent user={{ name: '블루', age: 30 }} />

    // 올바른 예시: useMemo로 객체 메모이제이션
    const user = useMemo(() => ({ name: '블루', age: 30 }), []);
    <ChildComponent user={user} />
    ```

## 📝 정리하며: 현명한 최적화가 중요해요

오늘은 React 렌더링 최적화의 핵심 도구인 `useMemo`, `useCallback`, `React.memo`에 대해 자세히 알아보았어요.<br />
이 세 가지 훅과 HOC는 React 애플리케이션의 성능을 크게 향상시킬 수 있는 강력한 기능들이에요.<br />
하지만 무분별한 사용은 오히려 코드를 복잡하게 만들고 디버깅을 어렵게 하며, 때로는 성능을 저하시킬 수도 있다는 점을 항상 기억해야 해요.<br />
성능 저하가 의심되는 지점을 정확히 파악하고, 필요한 곳에만 현명하게 적용하는 것이 진정한 최적화의 길이라고 말씀드리고 싶어요.<br />
