오늘은 TypeScript의 강력한 기능 중 하나인 '템플릿 리터럴 타입(Template Literal Types)'에 대해 깊이 있게 다뤄보려고 해요.<br />
이름만 들으면 어렵게 느껴질 수도 있지만, 실무에서 마주하는 동적인 문자열 패턴들을 훨씬 안전하고 유연하게 다룰 수 있도록 도와주는 마법 같은 기능이랍니다.

## 🤔 문제/배경: 왜 문자열 타입에 마법이 필요할까요?

### 0️⃣ 동적인 문자열, 불안정한 타입

프런트엔드 개발을 하다 보면, 특정 패턴을 가지는 문자열들을 자주 다루게 돼요.<br />
예를 들어, CSS 유틸리티 클래스 이름(`text-red-500`, `bg-blue-200`), 이벤트 리스너 이름(`onClick`, `onMouseEnter`), 또는 API 엔드포인트 경로(`GET /users/123`, `POST /products/new`) 등이 그렇죠.<br />
이런 문자열들은 단순히 `string` 타입으로만 선언하면 오타나 잘못된 패턴으로 인해 런타임 에러가 발생하기 쉬워요.<br />
타입스크립트를 사용하는 이유가 런타임 에러를 줄이고 코드의 안정성을 높이는 것인데, 문자열 타입에서는 그 효과가 반감될 때가 많아요.

### 1️⃣ 기존 방식의 한계점: `string`과 단순 유니온 타입의 아쉬움

기존에는 이런 동적인 문자열들을 어떻게 다뤘을까요?<br />
가장 흔한 방법은 `string` 타입을 사용하거나, 가능한 모든 문자열을 직접 유니온 타입으로 나열하는 것이었어요.

```ts showLineNumbers
type CssColor = 'red' | 'blue' | 'green';
type TextSize = 'sm' | 'md' | 'lg';

// 문제점: 'text-red'는 허용되지만, 'text-red-sm'과 같은 조합은 불가능해요.
// 모든 조합을 수동으로 나열해야 해요.
type CssClass = `text-${CssColor}` | `font-${TextSize}` | string; // 'string'을 넣으면 타입 안전성이 사라져요.

const myClass: CssClass = 'text-red'; // OK
const anotherClass: CssClass = 'font-lg'; // OK
// const invalidClass: CssClass = 'text-red-sm'; // 에러가 발생하지 않아요. (string으로 인해)
```

위 코드처럼 `string`을 포함하면 모든 문자열이 허용되어 타입 안전성이 무너지고, `string` 없이 모든 조합을 유니온으로 나열하는 것은 매우 번거롭고 실수할 가능성이 높아요.<br />
여기서 템플릿 리터럴 타입이 빛을 발한답니다.

## ✨ 템플릿 리터럴 타입이란?

### 0️⃣ 기본적인 문법과 동작 원리

템플릿 리터럴 타입은 ES2015에 도입된 템플릿 리터럴 문자열 문법(` `` ` 백틱)과 유사하게 작동해요.<br />
타입 변수를 사용하여 새로운 문자열 리터럴 타입을 생성하는 방식인데요, 마치 JavaScript에서 변수를 삽입해 문자열을 만드는 것처럼 타입에서도 동일하게 할 수 있어요.

기본 문법은 다음과 같아요.

```ts showLineNumbers
type Greeting = `Hello, ${string}!`;

const message1: Greeting = 'Hello, TypeScript!'; // OK
const message2: Greeting = 'Hello, World!'; // OK
// const message3: Greeting = 'Hi, TypeScript!'; // Type '"Hi, TypeScript!"' is not assignable to type '"Hello, ${string}!"'. 에러 발생!
```

위 예시에서 `string`은 어떤 문자열이든 올 수 있다는 의미예요.<br />
`Hello,` 와 `!` 사이에 어떤 문자열이 오더라도 `Greeting` 타입으로 인정하겠다는 것이죠.<br />
하지만 `Hello,` 접두사나 `!` 접미사가 없으면 에러가 발생해서 타입 안전성이 확보돼요.

### 1️⃣ 유니온 타입과의 시너지

템플릿 리터럴 타입의 진정한 힘은 유니온 타입과 결합했을 때 발휘돼요.<br />
유니온 타입의 각 멤버가 템플릿 리터럴 타입의 변수에 삽입되면서 가능한 모든 조합의 문자열 리터럴 타입을 생성해 준답니다.

```ts showLineNumbers
type Color = 'red' | 'blue' | 'green';
type Size = 'sm' | 'md' | 'lg';

// Color와 Size의 모든 조합을 자동으로 생성해줘요.
type CssUtilityClass = `text-${Color}-${Size}` | `bg-${Color}`; 

const class1: CssUtilityClass = 'text-red-sm'; // OK
const class2: CssUtilityClass = 'bg-blue'; // OK
// const class3: CssUtilityClass = 'text-red'; // Type '"text-red"' is not assignable to type 'CssUtilityClass'. 에러 발생!
// const class4: CssUtilityClass = 'font-green-lg'; // Type '"font-green-lg"' is not assignable to type 'CssUtilityClass'. 에러 발생!
```

이제 `text-red-sm`처럼 특정 패턴을 가진 문자열만 허용하고, 오타나 잘못된 조합은 컴파일 시점에 바로 잡아낼 수 있게 되었어요.<br />
모든 가능한 조합을 일일이 작성할 필요 없이, 타입스크립트가 자동으로 만들어준다는 점이 정말 편리하죠?

## 🛠️ 실무에서 템플릿 리터럴 타입 활용하기

이제 실무에서 템플릿 리터럴 타입을 어떻게 활용할 수 있는지 구체적인 예시들을 살펴볼게요.<br />
이 기능이 여러분의 코드에 어떤 마법을 부릴 수 있는지 직접 확인해 보세요.

### 0️⃣ 이벤트 리스너 타입 안전하게 만들기

React나 다른 UI 프레임워크에서 이벤트 핸들러를 정의할 때, `on` 접두사를 가진 이벤트 이름들을 자주 사용해요.<br />
템플릿 리터럴 타입을 사용하면 이를 훨씬 안전하게 만들 수 있어요.

```tsx showLineNumbers
type EventName = 'Click' | 'Focus' | 'Blur' | 'Change';

// 'onClick', 'onFocus' 등의 이벤트를 타입으로 정의해요.
type EventHandlerName = `on${EventName}`;

interface MyComponentProps {
  [key: EventHandlerName]: (event: React.SyntheticEvent) => void;
  // 'onClick', 'onFocus' 등의 속성만 허용해요.
  // 'onPress'와 같은 잘못된 이름은 에러를 발생시켜요.
}

const MyComponent = ({ onClick, onFocus }: MyComponentProps) => {
  return (
    <button onClick={onClick} onFocus={onFocus}>
      클릭하세요.
    </button>
  );
};

// 사용 예시
const ParentComponent = () => {
  const handleClick = (e: React.SyntheticEvent) => console.log('Clicked!', e.type);
  const handleFocus = (e: React.SyntheticEvent) => console.log('Focused!', e.type);

  return (
    <MyComponent
      onClick={handleClick}
      onFocus={handleFocus}
      // onInvalidEvent={console.log} // Type '{ onClick: (e: SyntheticEvent<Element, Event>) => void; onFocus: (e: SyntheticEvent<Element, Event>) => void; onInvalidEvent: (message?: any, ...optionalParams: any[]) => void; }' is not assignable to type 'MyComponentProps'. 에러 발생!
    />
  );
};
```

이제 `MyComponent`의 props로 `onClick`, `onFocus`와 같이 미리 정의된 이벤트 핸들러 이름만 사용할 수 있게 되었어요.<br />
오타로 인한 런타임 버그를 사전에 방지할 수 있겠죠?

### 1️⃣ CSS 유틸리티 클래스 자동 완성 및 검증

Tailwind CSS와 같은 유틸리티 우선(utility-first) CSS 프레임워크를 사용할 때, 다양한 클래스 이름 조합을 다루게 돼요.<br />
템플릿 리터럴 타입을 이용하면 이런 클래스 이름들을 타입으로 정의하고 자동 완성 기능을 활용할 수 있어요.

```ts showLineNumbers
type Spacing = 'px' | 'xs' | 'sm' | 'md' | 'lg' | 'xl';
type Direction = 't' | 'b' | 'l' | 'r' | 'x' | 'y' | ''; // '', 전체 방향

// 'p-md', 'mt-lg', 'py-sm'과 같은 클래스 이름을 정의해요.
type PaddingClass = `p${Direction}-${Spacing}`;
type MarginClass = `m${Direction}-${Spacing}`;

type BorderRadius = 'none' | 'sm' | 'md' | 'lg' | 'full';
type BorderClass = `rounded-${BorderRadius}`;

// 모든 가능한 유틸리티 클래스 조합
type UtilityClass = PaddingClass | MarginClass | BorderClass;

function applyClass(className: UtilityClass) {
  console.log(`Applying class: ${className}`);
}

applyClass('p-md'); // OK
applyClass('mt-lg'); // OK
applyClass('py-sm'); // OK
applyClass('rounded-full'); // OK

// applyClass('padding-md'); // Type '"padding-md"' is not assignable to type 'UtilityClass'. 에러 발생!
// applyClass('p-top-md'); // Type '"p-top-md"' is not assignable to type 'UtilityClass'. 에러 발생!
```

이제 `applyClass` 함수를 호출할 때, `UtilityClass` 타입에 맞는 클래스 이름만 허용돼요.<br />
개발 도구에서 자동 완성 기능을 통해 사용 가능한 클래스 목록을 쉽게 확인할 수 있어서 생산성이 크게 향상될 거예요.<br />
이는 단순히 타입 검사를 넘어 개발 경험을 개선하는 효과까지 가져다준답니다.

### 2️⃣ API 엔드포인트 타입 추론 및 생성

RESTful API를 설계할 때, `/users/:id`, `/products/:category/items`와 같이 경로 변수(path variable)를 포함하는 엔드포인트를 자주 사용해요.<br />
템플릿 리터럴 타입을 이용하면 이런 동적인 엔드포인트 경로를 타입으로 정의하고, 올바른 경로만 사용하도록 강제할 수 있어요.

```ts showLineNumbers
type Resource = 'users' | 'products' | 'orders';
type Action = 'detail' | 'list';

// '/users/detail', '/products/list' 등의 엔드포인트를 정의해요.
type ApiEndpoint = `/${Resource}/${Action}`;

// '/users/:id'와 같이 ID를 포함하는 엔드포인트
type UserDetailEndpoint<ID extends string | number> = `/users/${ID}`;

function fetchApi(endpoint: ApiEndpoint | UserDetailEndpoint<string | number>) {
  console.log(`Fetching data from: ${endpoint}`);
}

fetchApi('/users/list'); // OK
fetchApi('/products/detail'); // OK
fetchApi('/users/123'); // OK (UserDetailEndpoint<number>에 해당)
fetchApi('/users/abc-def'); // OK (UserDetailEndpoint<string>에 해당)

// fetchApi('/posts/list'); // Type '"/posts/list"' is not assignable to type 'ApiEndpoint | UserDetailEndpoint<string | number>'. 에러 발생!
// fetchApi('/users/delete'); // Type '"/users/delete"' is not assignable to type 'ApiEndpoint | UserDetailEndpoint<string | number>'. 에러 발생!
```

이처럼 API 엔드포인트를 타입으로 정의하면, 잘못된 경로로 요청을 보내는 실수를 컴파일 시점에 방지할 수 있어요.<br />
특히 복잡한 API 구조를 가질수록 그 효과는 더욱 커진답니다.

### 3️⃣ `infer` 키워드와 함께 패턴에서 타입 추출하기

템플릿 리터럴 타입은 `infer` 키워드와 함께 사용될 때 더욱 강력해져요.<br />
특정 문자열 패턴에서 원하는 부분의 타입을 '추론'하여 추출할 수 있게 해주거든요.<br />
이는 마치 정규 표현식으로 문자열을 파싱하는 것을 타입 수준에서 하는 것과 비슷하다고 생각하시면 돼요.

```ts showLineNumbers
// FromEvent 타입: 'on' 접두사와 이벤트 이름으로부터 이벤트 이름을 추출해요.
type FromEvent<EventType extends string> = 
  EventType extends `on${infer EventName}` 
    ? EventName 
    : never;

type ClickEvent = FromEvent<'onClick'>; // 'Click'
type MouseEnterEvent = FromEvent<'onMouseEnter'>; // 'MouseEnter'
type CustomEvent = FromEvent<'onMyCustomEvent'>; // 'MyCustomEvent'
type InvalidEvent = FromEvent<'click'>; // never (on 접두사가 없어요)

// ExtractRouteParam 타입: 경로에서 파라미터 이름을 추출해요.
type ExtractRouteParam<Route extends string> = 
  Route extends `${infer Prefix}/:${infer Param}/${infer Suffix}`
    ? Param | ExtractRouteParam<`/${Suffix}`>
    : Route extends `${infer Prefix}/:${infer Param}`
      ? Param
      : never;

type UserRouteParams = ExtractRouteParam<'/users/:id/posts/:postId'>; // 'id' | 'postId'
type ProductRouteParams = ExtractRouteParam<'/products/:productId'>; // 'productId'
type SimpleRouteParams = ExtractRouteParam<'/dashboard'>; // never
```

`FromEvent` 예시에서는 `on` 뒤에 오는 문자열을 `EventName`으로 추출하여 새로운 타입을 만들었어요.<br />
`ExtractRouteParam` 예시에서는 재귀적으로 `infer`를 사용하여 여러 개의 경로 파라미터 이름을 추출하는 복잡한 작업을 해냈죠.<br />
이렇게 `infer`와 템플릿 리터럴 타입을 조합하면, 문자열 패턴을 기반으로 유연하고 강력한 유틸리티 타입을 만들어낼 수 있답니다.

## 💡 템플릿 리터럴 타입 활용 팁과 주의사항

템플릿 리터럴 타입은 강력하지만, 몇 가지 주의할 점이 있어요.

### 0️⃣ 무한 재귀 타입 방지

`infer` 키워드를 재귀적으로 사용할 때, 타입스크립트 컴파일러가 무한 루프에 빠지는 것을 방지하기 위해 깊이 제한을 두어요.<br />
매우 복잡한 재귀 타입을 설계할 때는 이 점을 염두에 두어야 해요.

### 1️⃣ 가독성 유지하기

템플릿 리터럴 타입이 너무 복잡해지면 코드의 가독성이 떨어질 수 있어요.<br />
여러 개의 작은 유틸리티 타입으로 분리하거나, 주석을 충분히 달아서 다른 개발자들이 이해하기 쉽도록 노력해야 해요.<br />
항상 '명확성'과 '재사용성'을 최우선으로 고려하는 것이 중요하답니다.

## 📝 정리: 문자열 타입의 새로운 지평을 열다

### 0️⃣ 핵심 요약

오늘 우리는 TypeScript의 템플릿 리터럴 타입에 대해 자세히 알아보았어요.<br />
이 기능을 통해 다음과 같은 이점들을 얻을 수 있다는 것을 확인했죠.

*   **타입 안전성 향상**: 동적인 문자열 패턴에 대한 오타나 잘못된 조합을 컴파일 시점에 미리 잡아낼 수 있어요.<br />
*   **개발 생산성 증대**: IDE의 자동 완성 기능을 활용하여 코드 작성 시간을 줄이고 실수를 예방할 수 있어요.<br />
*   **유연한 타입 정의**: 유니온 타입, `infer` 키워드와 결합하여 복잡한 문자열 패턴을 유연하게 정의하고 원하는 부분을 추출할 수 있어요.
