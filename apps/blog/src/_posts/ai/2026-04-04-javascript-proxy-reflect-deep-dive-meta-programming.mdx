## 🤔 JavaScript 메타 프로그래밍이란?

### 0️⃣ 객체 동작을 제어하는 능력

메타 프로그래밍은 "프로그램을 조작하는 프로그램"을 의미해요. JavaScript에서는 주로 객체의 기본적인 동작(속성 접근, 할당, 함수 호출 등)을 가로채서(intercept) 커스텀 로직을 추가하거나 변경하는 능력을 말해요. 마치 객체의 행동에 대한 "감시자" 또는 "경비원"을 두는 것과 같죠.<br />
이러한 능력은 객체의 내부 구현을 바꾸지 않고도 외부에서 다양한 기능을 확장하거나 제어할 수 있게 해줘요.

### 1️⃣ Proxy와 Reflect의 등장 배경

과거에는 객체의 동작을 제어하기 위해 `Object.defineProperty()` 같은 메서드를 사용하거나, 클로저를 활용해 객체를 래핑하는 등의 복잡한 패턴을 사용해야 했어요. 하지만 이런 방법들은 특정 속성에만 적용되거나, 모든 객체 동작을 일관되게 제어하기 어려웠어요.<br />
ES6에서 도입된 `Proxy`와 `Reflect`는 이러한 한계를 극복하고, 객체의 모든 메타 동작을 보다 선언적이고 강력하게 제어할 수 있는 표준화된 방법을 제공하기 위해 등장했어요.

## ⚙️ Proxy: 객체의 문지기

`Proxy`는 특정 객체(target)의 작업을 가로채서(intercept) 사용자 정의 동작을 수행할 수 있게 해주는 객체예요. 마치 객체로 들어오고 나가는 모든 요청을 검사하고 처리하는 '문지기'와 같다고 생각하시면 돼요.

### 0️⃣ Proxy의 기본 구조와 동작 원리

`Proxy`는 두 개의 인자를 받아서 새로운 객체를 생성해요.

<br />
<br />
`new Proxy(target, handler)`
<br />
<br />* **`target`**: 프록시될 원본 객체예요. 프록시를 통해 이루어지는 모든
작업의 대상이 되는 객체죠. * **`handler`**: 트랩(trap)이라고 불리는 메서드들을
포함하는 객체예요. 이 트랩들이 특정 작업을 가로채서 사용자 정의 로직을 실행하는
역할을 해요. 예를 들어, `get` 트랩은 속성 읽기 작업을, `set` 트랩은 속성 쓰기
작업을 가로채요.

프록시 객체에 대한 모든 작업은 먼저 `handler` 객체의 해당 트랩을 거치게 돼요. 만약 해당 트랩이 정의되어 있지 않다면, 작업은 자동으로 원본 `target` 객체로 전달되어 기본 동작을 수행해요.

### 1️⃣ `new Proxy(target, handler)`

간단한 예시를 통해 `Proxy`의 동작 방식을 살펴볼게요.

```tsx showLineNumbers
const targetUser = {
  name: "블루",
  age: 10,
};

const userProxy = new Proxy(targetUser, {
  // 'get' 트랩: 속성을 읽을 때 호출돼요
  get: function (target, prop, receiver) {
    console.log(`[Proxy] '${String(prop)}' 속성을 읽으려고 해요.`);
    return target[prop]; // 원본 객체의 속성 값을 반환해요
  },
  // 'set' 트랩: 속성을 설정할 때 호출돼요
  set: function (target, prop, value, receiver) {
    console.log(
      `[Proxy] '${String(prop)}' 속성을 '${value}'로 설정하려고 해요.`,
    );
    target[prop] = value; // 원본 객체의 속성 값을 설정해요
    return true; // 성공적으로 설정되었음을 나타내요
  },
});

console.log(userProxy.name); // [Proxy] 'name' 속성을 읽으려고 해요. / 블루
userProxy.age = 11; // [Proxy] 'age' 속성을 '11'로 설정하려고 해요.
console.log(userProxy.age); // [Proxy] 'age' 속성을 읽으려고 해요. / 11
```

위 코드에서 `userProxy`를 통해 `name` 속성을 읽거나 `age` 속성을 변경할 때마다 `console.log` 메시지가 출력되는 것을 볼 수 있어요. 이는 `get`과 `set` 트랩이 작업을 가로채서 실행되었기 때문이에요.

### 2️⃣ 다양한 `handler` 트랩

`Proxy`는 `get`, `set` 외에도 객체의 다양한 메타 동작을 가로챌 수 있는 13가지 트랩을 제공해요. 몇 가지 주요 트랩을 소개해 드릴게요.

- `get(target, prop, receiver)`: 속성 값을 읽을 때 호출돼요.
- `set(target, prop, value, receiver)`: 속성 값을 설정할 때 호출돼요.
- `has(target, prop)`: `in` 연산자를 사용할 때 호출돼요.
- `deleteProperty(target, prop)`: `delete` 연산자를 사용할 때 호출돼요.
- `apply(target, thisArg, argumentsList)`: 함수를 호출할 때 호출돼요. (프록시 대상이 함수일 경우)
- `construct(target, argumentsList, newTarget)`: `new` 연산자로 생성자를 호출할 때 호출돼요. (프록시 대상이 생성자일 경우)
- `ownKeys(target)`: `Object.keys()`, `Object.getOwnPropertyNames()`, `Object.getOwnPropertySymbols()` 등을 사용할 때 호출돼요.

<Blockquote type="info">
  각 트랩은 특정 작업을 가로챌 때 호출되는 함수예요. 이 함수 안에서 원본
  `target` 객체에 대한 작업을 직접 수행하거나, `Reflect` 객체의 해당 메서드를
  호출하여 기본 동작을 위임할 수 있어요.
</Blockquote>

## ✨ Reflect: 내장된 객체 메서드의 새로운 얼굴

`Reflect`는 객체에 대한 내장된(built-in) 조작들을 위한 정적 메서드를 제공하는 객체예요. `Reflect`는 `Proxy`와 함께 사용될 때 강력한 시너지를 발휘하며, 메타 프로그래밍을 위한 일관되고 안전한 방법을 제공해요.

### 0️⃣ Reflect의 역할과 필요성

기존에는 객체에 대한 작업을 수행할 때 `Object` 객체의 메서드(예: `Object.defineProperty()`, `Object.keys()`)나 연산자(예: `in`, `delete`)를 사용했어요. 하지만 이런 방법들은 몇 가지 문제를 가지고 있었어요.

- **일관성 부족**: `Object.keys()`는 함수지만 `in`이나 `delete`는 연산자예요. 객체 조작 방식이 통일되어 있지 않아요.
- **에러 처리**: 일부 `Object` 메서드는 실패 시 `false`를 반환하는 반면, 다른 메서드는 에러를 발생시켜요.
- **`this` 바인딩 문제**: `Object.prototype`의 메서드를 사용할 때 `this` 바인딩에 신경 써야 할 때가 있어요.

`Reflect`는 이러한 문제들을 해결하기 위해 `Object` 객체에 존재하는 대부분의 내장 메서드와 동일한 기능을 제공하면서도, 함수 형태로 통일하고 일관된 반환 값(성공/실패 여부를 불리언으로 반환하는 등)을 제공해요.

### 1️⃣ `Reflect.get()`, `Reflect.set()`, `Reflect.apply()` 등

`Reflect` 객체는 `Proxy` `handler`의 트랩과 1:1로 대응하는 메서드를 가지고 있어요.

- `Reflect.get(target, prop, receiver)`
- `Reflect.set(target, prop, value, receiver)`
- `Reflect.has(target, prop)`
- `Reflect.deleteProperty(target, prop)`
- `Reflect.apply(target, thisArg, argumentsList)`
- `Reflect.construct(target, argumentsList, newTarget)`
- `Reflect.ownKeys(target)`
- ... 그 외에도 `Reflect.defineProperty`, `Reflect.getOwnPropertyDescriptor`, `Reflect.getPrototypeOf` 등 다양한 메서드가 있어요.

<Blockquote type="tip">
  `Reflect` 메서드는 `Proxy` 트랩 내부에서 원본 객체의 기본 동작을 수행할 때
  유용하게 사용돼요.
  <br />
  `Reflect` 메서드를 사용하면 `target[prop]`처럼 직접 접근하는 대신, 보다
  안전하고 표준화된 방식으로 작업을 위임할 수 있어요.
</Blockquote>

### 2️⃣ 왜 `Reflect`를 사용해야 할까요?

`Proxy` 트랩 내에서 `Reflect`를 사용하는 것은 몇 가지 중요한 이점이 있어요.

- **기본 동작 위임**: `Proxy` 트랩에서 커스텀 로직을 수행한 후, 나머지 기본 동작은 `Reflect`를 통해 원본 `target` 객체에 위임할 수 있어요. 이는 코드를 더 간결하고 안전하게 만들어요.
- **정확한 `receiver` 전달**: `Reflect` 메서드는 세 번째 인자로 `receiver`를 받는데, 이는 `this` 바인딩을 올바르게 유지하는 데 중요해요. 특히 상속 체인이나 다른 프록시가 관련될 때 유용하죠.
- **일관된 API**: `Reflect`는 객체에 대한 모든 메타 작업을 함수 형태로 제공하여 일관된 프로그래밍 인터페이스를 제공해요.

```tsx showLineNumbers
const targetObj = {
  a: 1,
  get b() {
    return this.a + 10;
  },
};

const proxyObj = new Proxy(targetObj, {
  get(target, prop, receiver) {
    console.log(`[Proxy with Reflect] '${String(prop)}' 속성을 읽어요.`);
    // Reflect.get을 사용하여 원본 객체의 속성 값을 가져와요.
    // receiver를 전달하여 getter의 'this'가 프록시 객체를 가리키도록 해요.
    return Reflect.get(target, prop, receiver);
  },
});

console.log(proxyObj.a); // [Proxy with Reflect] 'a' 속성을 읽어요. / 1
console.log(proxyObj.b); // [Proxy with Reflect] 'b' 속성을 읽어요. / 11
```

위 예시에서 `proxyObj.b`를 호출했을 때, `targetObj`의 `b` getter 내부의 `this`는 `proxyObj`를 가리키게 돼요. 이는 `Reflect.get`의 세 번째 인자인 `receiver` 덕분이에요. 만약 `target[prop]`을 직접 사용했다면 `this`는 `targetObj`를 가리켰을 것이고, 이는 예상치 못한 동작을 유발할 수 있어요.

## 🛠️ 실전 활용 사례

이제 `Proxy`와 `Reflect`를 활용하여 실무에서 유용하게 사용할 수 있는 몇 가지 패턴을 살펴볼게요.

### 0️⃣ 데이터 유효성 검사 (Validation)

객체 속성에 값이 할당될 때마다 유효성을 검사하는 로직을 추가할 수 있어요.

```tsx showLineNumbers
const userProfile = {
  name: "Jane Doe",
  email: "jane.doe@example.com",
  age: 25,
};

const validatedProfile = new Proxy(userProfile, {
  set(target, prop, value, receiver) {
    if (prop === "age") {
      if (typeof value !== "number" || value < 0 || value > 150) {
        console.error(
          `[Validation Error] 'age'는 0에서 150 사이의 숫자여야 해요. 현재 값: ${value}`,
        );
        return false; // 설정 실패를 알림
      }
    }
    if (prop === "email") {
      if (!value.includes("@") || !value.includes(".")) {
        console.error(
          `[Validation Error] 'email' 형식이 올바르지 않아요. 현재 값: ${value}`,
        );
        return false;
      }
    }
    console.log(
      `[Validation Success] '${String(prop)}'을 '${value}'로 설정해요.`,
    );
    return Reflect.set(target, prop, value, receiver); // 유효성 검사 통과 시 값 설정
  },
});

validatedProfile.age = 30; // [Validation Success] 'age'을 '30'로 설정해요.
validatedProfile.age = -5; // [Validation Error] 'age'는 0에서 150 사이의 숫자여야 해요. 현재 값: -5
validatedProfile.email = "invalid-email"; // [Validation Error] 'email' 형식이 올바르지 않아요. 현재 값: invalid-email
validatedProfile.email = "valid@email.com"; // [Validation Success] 'email'을 'valid@email.com'로 설정해요.

console.log(validatedProfile.age); // 30
console.log(validatedProfile.email); // valid@email.com
```

`set` 트랩을 사용하여 `age`와 `email` 속성에 대한 유효성 검사를 추가했어요. 유효성 검사에 실패하면 `false`를 반환하여 값 설정을 막고, 성공하면 `Reflect.set`을 통해 값을 안전하게 설정해요.

### 1️⃣ 객체 속성 접근 로깅 (Logging)

디버깅이나 모니터링을 위해 객체 속성에 대한 모든 읽기/쓰기 작업을 로깅할 수 있어요.

```tsx showLineNumbers
const appConfig = {
  apiEndpoint: "https://api.example.com",
  timeout: 5000,
  debugMode: false,
};

const loggedConfig = new Proxy(appConfig, {
  get(target, prop, receiver) {
    console.log(
      `[Log] '${String(prop)}' 속성을 읽었어요. 값: ${Reflect.get(target, prop, receiver)}`,
    );
    return Reflect.get(target, prop, receiver);
  },
  set(target, prop, value, receiver) {
    console.log(
      `[Log] '${String(prop)}' 속성을 '${Reflect.get(target, prop, receiver)}'에서 '${value}'로 변경했어요.`,
    );
    return Reflect.set(target, prop, value, receiver);
  },
});

console.log(loggedConfig.apiEndpoint); // [Log] 'apiEndpoint' 속성을 읽었어요. 값: https://api.example.com / https://api.example.com
loggedConfig.debugMode = true; // [Log] 'debugMode' 속성을 'false'에서 'true'로 변경했어요.
console.log(loggedConfig.debugMode); // [Log] 'debugMode' 속성을 읽었어요. 값: true / true
```

`get`과 `set` 트랩에서 `console.log`를 사용하여 어떤 속성이 읽히고 변경되는지 추적할 수 있어요. 이는 복잡한 애플리케이션에서 데이터 흐름을 이해하는 데 큰 도움이 돼요.

### 2️⃣ 반응형 객체 구현 (Reactivity)

Vue.js나 MobX 같은 반응형 프레임워크의 핵심 개념 중 하나가 바로 `Proxy`를 활용한 반응형 객체예요. 객체의 변경을 감지하여 UI를 업데이트하는 등의 작업을 할 수 있죠.

```tsx showLineNumbers
function createReactive(obj) {
  const handlers = {
    set(target, prop, value, receiver) {
      const oldValue = Reflect.get(target, prop, receiver);
      if (oldValue !== value) {
        console.log(
          `[Reactive] '${String(prop)}' 속성이 '${oldValue}'에서 '${value}'로 변경되었어요!`,
        );
        // 여기에서 UI 업데이트 또는 다른 부수 효과를 트리거할 수 있어요.
        // 예를 들어, observer 패턴을 사용하여 구독자들에게 변경을 알릴 수 있죠.
      }
      return Reflect.set(target, prop, value, receiver);
    },
  };
  return new Proxy(obj, handlers);
}

const state = createReactive({
  count: 0,
  message: "Hello",
});

state.count++; // [Reactive] 'count' 속성이 '0'에서 '1'로 변경되었어요!
state.message = "World"; // [Reactive] 'message' 속성이 'Hello'에서 'World'로 변경되었어요!
state.message = "World"; // 변경 없음, 로그 출력 안 됨
```

`createReactive` 함수는 객체를 받아서 `set` 트랩을 통해 속성 변경을 감지하고 알림을 출력해요. 실제 프레임워크에서는 이 알림을 받아 DOM을 업데이트하는 등의 작업을 수행하겠죠.

### 3️⃣ 읽기 전용(Readonly) 객체 만들기

객체의 모든 속성을 읽기 전용으로 만들어 실수로 변경하는 것을 방지할 수 있어요.

```tsx showLineNumbers
function createReadonly(obj) {
  return new Proxy(obj, {
    set(target, prop, value, receiver) {
      console.warn(
        `[Readonly Error] 읽기 전용 객체의 '${String(prop)}' 속성을 변경할 수 없어요.`,
      );
      return false; // 값 설정 실패를 반환
    },
    deleteProperty(target, prop) {
      console.warn(
        `[Readonly Error] 읽기 전용 객체의 '${String(prop)}' 속성을 삭제할 수 없어요.`,
      );
      return false; // 속성 삭제 실패를 반환
    },
  });
}

const config = {
  version: "1.0.0",
  owner: "블루팀",
};

const readonlyConfig = createReadonly(config);

console.log(readonlyConfig.version); // 1.0.0
readonlyConfig.version = "2.0.0"; // [Readonly Error] 읽기 전용 객체의 'version' 속성을 변경할 수 없어요.
delete readonlyConfig.owner; // [Readonly Error] 읽기 전용 객체의 'owner' 속성을 삭제할 수 없어요.

console.log(readonlyConfig.version); // 여전히 1.0.0
```

`set`과 `deleteProperty` 트랩에서 항상 `false`를 반환하여 객체 변경 및 삭제를 막고 경고 메시지를 출력해요.

## 📝 Proxy와 Reflect를 함께 사용해야 하는 이유

이 글의 예시들을 통해 보셨듯이, `Proxy` 트랩 내부에서 `Reflect` 메서드를 사용하는 것은 단순한 코딩 스타일을 넘어선 중요한 이유가 있어요.

### 0️⃣ 일관된 객체 동작 처리

`Reflect`는 객체의 모든 내장된 메타 동작을 함수 형태로 제공해요. `Proxy` 트랩 내에서 `Reflect`를 사용하면, 어떤 작업을 가로채든 일관된 방식으로 원본 `target` 객체의 기본 동작을 호출할 수 있어요. 이는 코드의 가독성과 유지보수성을 높여줘요.

### 1️⃣ 안전하고 예측 가능한 메타 프로그래밍

특히 객체에 getter/setter가 있거나 상속 체인에 다른 프록시가 있을 때, `target[prop] = value`와 같은 직접적인 접근은 `this` 바인딩 문제를 일으키거나 예상치 못한 부수 효과를 발생시킬 수 있어요.<br />
`Reflect` 메서드는 `receiver` 인자를 통해 올바른 `this` 컨텍스트를 유지하고, 객체 내부의 숨겨진 로직(예: setter)이 정확하게 실행되도록 보장해요. 이는 더욱 안전하고 예측 가능한 메타 프로그래밍을 가능하게 해줘요.

<Blockquote type="tip">
  `Proxy` 트랩 내부에서 원본 `target` 객체의 기본 동작을 수행해야 할 때는 항상
  `Reflect` 메서드를 사용하는 것을 권장해요.
  <br />
  이는 견고하고 미래 지향적인 코드를 작성하는 모범 사례로 여겨져요.
</Blockquote>

## 🚀 정리하며

오늘은 JavaScript의 강력한 메타 프로그래밍 도구인 `Proxy`와 `Reflect`에 대해 심층적으로 살펴보았어요.<br />
`Proxy`는 객체의 모든 메타 동작을 가로채서 커스텀 로직을 추가할 수 있게 해주는 '문지기' 역할을 하고, `Reflect`는 이러한 메타 동작들을 안전하고 일관된 방식으로 수행할 수 있도록 돕는 '내장된 객체 메서드 집합'이라는 것을 이해하셨을 거예요.<br />
데이터 유효성 검사, 로깅, 반응형 객체 구현, 읽기 전용 객체 만들기 등 다양한 실무 활용 사례를 통해 두 API가 어떻게 코드의 유연성과 안정성을 높이는 데 기여하는지 확인했어요.

`Proxy`와 `Reflect`는 처음에는 다소 복잡하게 느껴질 수 있지만, 이들을 잘 활용하면 프레임워크 개발, 라이브러리 작성, 또는 특정 비즈니스 로직에 대한 강력한 제어 메커니즘을 구현할 수 있어요. 여러분의 다음 프로젝트에서 이 강력한 도구들을 활용하여 더욱 견고하고 유연한 코드를 작성해 보시길 바라요!

### 0️⃣ 핵심 요약

- `Proxy`: 객체의 메타 동작(읽기, 쓰기, 함수 호출 등)을 가로채서 커스텀 로직을 추가하는 객체예요.
- `Reflect`: 객체에 대한 내장된 메타 동작을 함수 형태로 제공하며, `Proxy` 트랩 내부에서 기본 동작을 안전하고 일관되게 위임하는 데 사용돼요.
- 함께 사용하면 유효성 검사, 로깅, 반응형 시스템, 읽기 전용 객체 등 다양한 고급 패턴을 구현할 수 있어요.
