> Next.js App Router의 핵심인 Server Components와 Client Components의 차이점, 사용 시나리오, 그리고 실무 적용 노하우를 상세히 안내해 드려요.

## 🤔 문제/배경

Next.js 13부터 App Router가 도입되면서, 서버 컴포넌트(Server Components)와 클라이언트 컴포넌트(Client Components)라는 새로운 개념이 등장했어요. 처음 접하는 분들은 이 두 가지의 차이점과 언제 무엇을 사용해야 할지 혼란스러울 수 있어요. 저 역시 그랬답니다.

### 0️⃣ 왜 이 주제를 다루는가

App Router는 Next.js의 미래이자 React 생태계의 중요한 변화를 이끌고 있어요. 서버 컴포넌트는 애플리케이션의 성능과 개발 경험을 획기적으로 개선할 잠재력을 가지고 있지만, 그만큼 정확한 이해와 활용법이 필요해요. 이 글에서는 서버 컴포넌트와 클라이언트 컴포넌트를 명확히 구분하고, 실제 프로젝트에서 어떻게 적용해야 할지 가이드라인을 제시해 드리려고 해요.

### 1️⃣ 기존 방식의 한계

Next.js의 Pages Router나 전통적인 React 애플리케이션에서는 모든 컴포넌트가 기본적으로 클라이언트에서 렌더링되거나, SSR(Server-Side Rendering)을 하더라도 최종적으로 클라이언트 번들에 포함되어 hydration 과정을 거쳐야 했어요. 이 방식은 몇 가지 한계를 가지고 있었어요.

*   **번들 사이즈 증가:** 모든 컴포넌트와 그 의존성이 클라이언트 번들에 포함되어 다운로드 시간이 길어졌어요.
*   **성능 저하:** Hydration 과정에서 많은 JavaScript가 실행되어 TTI(Time To Interactive)가 느려지는 경우가 많았어요.
*   **데이터 페칭의 복잡성:** 서버에서 데이터를 페칭하려면 `getServerSideProps`나 `getStaticProps` 같은 특별한 함수를 사용해야 했고, 컴포넌트 레벨에서의 직접적인 데이터 페칭은 어려웠어요.

## ⚙️ 해결 방법: Server Components와 Client Components 이해하기

Next.js App Router는 이러한 한계를 극복하기 위해 "서버에서 렌더링되는 컴포넌트"와 "클라이언트에서 렌더링되는 컴포넌트"를 명시적으로 구분하는 모델을 도입했어요.

### 0️⃣ 핵심 아이디어: 렌더링 환경 분리

가장 중요한 아이디어는 컴포넌트의 "실행 위치"를 분리하는 거예요.

*   **Server Components (기본값):** 서버에서만 실행되고 렌더링되는 컴포넌트예요. JavaScript 번들에 포함되지 않아 번들 사이즈를 줄이고, 초기 페이지 로드 성능을 크게 향상시킬 수 있어요. 데이터를 직접 페칭하거나, 서버 전용 API에 접근할 수 있어요.
*   **Client Components:** 클라이언트에서 실행되고 렌더링되는 컴포넌트예요. 기존 React 컴포넌트와 유사하게 `useState`, `useEffect` 등 클라이언트 전용 React Hook을 사용하거나, 브라우저 API에 접근할 수 있어요. 상호작용이 필요한 부분에 사용해요.

<Blockquote type="info">
Next.js 13 App Router에서는 모든 컴포넌트가 기본적으로 Server Components로 간주돼요. 클라이언트 컴포넌트로 만들고 싶다면 파일 맨 위에 `'use client'` 지시자를 명시해야 해요.
</Blockquote>

### 1️⃣ Server Components의 특징과 활용

서버 컴포넌트는 다음과 같은 특징을 가져요.

*   **JavaScript 번들에 포함되지 않아요:** 클라이언트로 전송되지 않으므로 번들 사이즈가 작아져요.
*   **서버에서 데이터 페칭:** `async/await`를 사용하여 데이터베이스나 외부 API에서 직접 데이터를 가져올 수 있어요. 별도의 API 라우터를 만들 필요가 줄어들어요.
*   **보안:** 서버 환경에서만 실행되므로 민감한 정보(API 키 등)를 안전하게 다룰 수 있어요.
*   **성능:** 초기 로딩 속도와 First Contentful Paint(FCP)가 빨라져요.
*   **제한사항:** `useState`, `useEffect`와 같은 클라이언트 전용 훅이나 브라우저 API(DOM, `window` 객체)를 직접 사용할 수 없어요. 이벤트 핸들러도 직접 추가할 수 없어요.

**주요 활용 시나리오:**

*   데이터베이스에서 데이터를 가져와 표시하는 부분
*   API 키나 민감한 정보를 사용하는 로직
*   정적 콘텐츠나 레이아웃 (헤더, 푸터, 사이드바 등)
*   서버에서만 동작하는 라이브러리 사용

### 2️⃣ Client Components의 특징과 활용

클라이언트 컴포넌트는 우리가 기존에 알고 있던 React 컴포넌트와 유사해요.

*   **클라이언트 번들에 포함돼요:** JavaScript 번들의 일부가 되어 클라이언트에 다운로드되고 Hydration 과정을 거쳐요.
*   **상호작용 가능:** `useState`, `useEffect`, `useRef` 등 모든 React Hook과 브라우저 API를 자유롭게 사용할 수 있어요.
*   **이벤트 핸들러:** `onClick`, `onChange` 등 사용자 인터랙션을 처리할 수 있어요.
*   **성능:** 초기 로딩 속도 측면에서는 서버 컴포넌트보다 느릴 수 있지만, 사용자 상호작용 후에는 클라이언트에서 빠르게 반응해요.

**주요 활용 시나리오:**

*   사용자 입력을 받는 폼(`input`, `textarea`, `select`)
*   클라이언트 상태 관리(`useState`, Context API, Zustand, Jotai, Recoil 등)
*   이벤트 핸들러가 필요한 버튼이나 네비게이션
*   브라우저 API(`localStorage`, `geolocation`)를 사용하는 로직
*   클라이언트 전용 라이브러리(차트 라이브러리, 모달, 캐러셀 등)

## 🧪 예시: 실제 코드와 사용 패턴

이제 몇 가지 코드 예시를 통해 서버 컴포넌트와 클라이언트 컴포넌트가 어떻게 사용되는지 알아볼까요?

### 0️⃣ 기본 사용 예시 (`'use client'` 지시자)

모든 `.js`, `.jsx`, `.ts`, `.tsx` 파일은 기본적으로 서버 컴포넌트예요. 클라이언트 컴포넌트로 명시하려면 파일 맨 위에 `'use client'` 지시자를 추가해야 해요.

```tsx showLineNumbers
// app/page.tsx (Server Component)
// 이 파일은 'use client' 지시자가 없으므로 서버 컴포넌트예요.
import Link from 'next/link';
import { fetchPosts } from '../lib/api'; // 서버에서만 실행되는 API 호출

interface Post {
  id: number;
  title: string;
  content: string;
}

export default async function HomePage() {
  const posts: Post[] = await fetchPosts(); // 서버에서 직접 데이터 페칭
  // console.log("서버에서만 보이는 로그입니다."); // 서버 터미널에만 출력돼요.

  return (
    <main>
      <h1>블로그 홈</h1>
      <p>최신 포스트 목록이에요.</p>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>
            <Link href={`/posts/${post.id}`}>
              {post.title}
            </Link>
          </li>
        ))}
      </ul>
      <AddPostButton /> {/* Server Component 안에서 Client Component 사용 예시 */}
    </main>
  );
}

// components/AddPostButton.tsx (Client Component)
// 이 파일은 'use client' 지시자가 있으므로 클라이언트 컴포넌트예요.
'use client';

import { useState } from 'react';

export default function AddPostButton() {
  const [count, setCount] = useState(0); // 클라이언트 상태 관리

  const handleClick = () => {
    alert('새 포스트 추가하기 버튼을 클릭하셨어요!');
    setCount(prev => prev + 1);
  };

  return (
    <div>
      <p>클라이언트에서만 보이는 카운트: {count}</p>
      <button onClick={handleClick}>
        새 포스트 추가하기
      </button>
    </div>
  );
}
```

<Blockquote type="tip">
`'use client'` 지시자는 파일의 가장 상단에 위치해야 하며, 주석이나 다른 코드보다 먼저 와야 해요.
</Blockquote>

### 1️⃣ Server Component 안에서 Client Component 사용하기

서버 컴포넌트 안에서 클라이언트 컴포넌트를 사용하는 것은 매우 자연스럽고 흔한 패턴이에요. 서버에서 기본적인 레이아웃과 데이터를 구성하고, 상호작용이 필요한 부분만 클라이언트 컴포넌트로 감싸는 방식이죠.

```tsx showLineNumbers {16}
// app/dashboard/page.tsx (Server Component)
import { getUserProfile } from '../../lib/auth';
import LogoutButton from './LogoutButton'; // Client Component

export default async function DashboardPage() {
  const user = await getUserProfile();

  return (
    <div className="dashboard-layout">
      <h2>환영합니다, {user.name}님!</h2>
      <p>여기는 서버에서 렌더링된 대시보드 페이지예요.</p>
      {/* 서버 컴포넌트 안에 클라이언트 컴포넌트를 직접 렌더링할 수 있어요. */}
      <LogoutButton />
    </div>
  );
}

// app/dashboard/LogoutButton.tsx (Client Component)
'use client';

import { useRouter } from 'next/navigation';

export default function LogoutButton() {
  const router = useRouter();

  const handleLogout = () => {
    // 로그아웃 API 호출 등 클라이언트에서 처리할 로직
    alert('로그아웃 처리 중입니다...');
    router.push('/login');
  };

  return (
    <button onClick={handleLogout} className="logout-btn">
      로그아웃
    </button>
  );
}
```

### 2️⃣ Client Component 안에서 Server Component (Props로 전달) 사용하기

클라이언트 컴포넌트가 서버 컴포넌트를 직접 import 할 수는 없어요. 하지만 서버 컴포넌트가 렌더링한 결과를 `children` prop이나 다른 prop으로 클라이언트 컴포넌트에 전달하는 방식은 가능해요. 이것을 "컴포넌트 합성(Composition)"이라고 불러요.

```tsx showLineNumbers {11-13, 20-22}
// app/layout.tsx (Server Component) - Root Layout은 항상 서버 컴포넌트예요.
import MyThemeProvider from '../components/MyThemeProvider'; // Client Component

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="ko">
      <body>
        {/* 서버 컴포넌트가 클라이언트 컴포넌트를 감싸고,
            그 안에 서버 컴포넌트인 children을 props로 전달하는 방식이에요. */}
        <MyThemeProvider>
          {children}
        </MyThemeProvider>
      </body>
    </html>
  );
}

// components/MyThemeProvider.tsx (Client Component)
'use client';

import { createContext, useState, useContext } from 'react';

const ThemeContext = createContext<{ theme: string; toggleTheme: () => void } | undefined>(undefined);

export default function MyThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState('light');

  const toggleTheme = () => {
    setTheme((prevTheme) => (prevTheme === 'light' ? 'dark' : 'light'));
  };

  // children (Server Component)은 클라이언트 컴포넌트 내부에서 렌더링되지만,
  // 그 자체는 여전히 서버에서 처리된 결과물이에요.
  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      <div className={theme === 'dark' ? 'dark-mode' : ''}>
        테마: {theme}
        {children}
      </div>
    </ThemeContext.Provider>
  );
}

// app/page.tsx (Server Component) - MyThemeProvider의 children으로 전달될 컴포넌트
export default function HomePage() {
  return (
    <main>
      <h1>홈 페이지 제목</h1>
      <p>이 내용은 서버에서 렌더링됩니다.</p>
    </main>
  );
}
```

<Blockquote type="warning">
클라이언트 컴포넌트 내에서 `'use client'` 지시자가 없는 파일을 `import` 하는 것은 **불가능**해요. 이는 빌드 오류를 발생시키니 주의해야 해요.
</Blockquote>

### 3️⃣ 일반적인 App Router 구조 예시

대부분의 App Router 애플리케이션은 서버 컴포넌트를 기본으로 사용하고, 특정 상호작용이 필요한 부분에만 클라이언트 컴포넌트를 렌더링하는 형태로 구성될 거예요.

```tsx showLineNumbers {5, 12, 18-20, 24-25}
// app/layout.tsx (Server Component)
import Header from '@/components/Header'; // Server Component
import Footer from '@/components/Footer'; // Server Component

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ko">
      <body>
        <Header />
        <div className="container">
          {children} {/* 여기에 개별 페이지 컴포넌트(서버 또는 클라이언트)가 렌더링됩니다. */}
        </div>
        <Footer />
      </body>
    </html>
  );
}

    </article>
  );
}
```

## 📝 정리

서버 컴포넌트와 클라이언트 컴포넌트는 Next.js App Router의 핵심을 이루는 패러다임 변화예요. 처음에는 다소 복잡하게 느껴질 수 있지만, 이 둘의 역할과 장단점을 명확히 이해하면 애플리케이션의 성능과 개발 효율성을 크게 향상시킬 수 있답니다.

### 0️⃣ 핵심 요약

*   **Server Components (기본값):** 서버에서 렌더링되고, 클라이언트 번들에 포함되지 않아요. 데이터 페칭, 서버 로직, 정적 콘텐츠에 적합해요. `useState`, `useEffect` 같은 훅은 사용할 수 없어요.
*   **Client Components (`'use client'`):** 클라이언트에서 렌더링되고, 모든 React 훅과 브라우저 API를 사용할 수 있어요. 사용자 인터랙션, 클라이언트 상태 관리, 클라이언트 전용 라이브러리에 적합해요.
*   **Optimal Strategy:** 기본적으로 서버 컴포넌트를 사용하고, 상호작용이 필요한 최소한의 부분만 클라이언트 컴포넌트로 만들어서 감싸는 것이 가장 좋은 전략이에요.

### 1️⃣ 다음 액션

직접 Next.js App Router 프로젝트를 시작하거나 기존 프로젝트를 마이그레이션하면서 서버 컴포넌트와 클라이언트 컴포넌트의 동작 방식을 실험해 보세요. 이 새로운 패러다임에 익숙해지는 것이 Next.js 개발의 핵심이 될 거예요. 궁금한 점은 언제든지 공식 문서를 참고하거나 커뮤니티에 질문해 보시는 것을 추천해요.
