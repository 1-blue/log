## 🚀 CI/CD, 왜 중요할까요?

### 0️⃣ 개발 생산성 향상과 안정적인 배포의 핵심

개발 과정에서 코드를 변경하고, 테스트하고, 배포하는 일은 반복적이고 시간이 많이 소요되는 작업이에요.<br />
특히 Next.js와 같은 복잡한 프레임워크를 사용하는 프로젝트에서는 빌드, 린트, 타입 체크, 테스트 등 여러 단계를 거쳐야 하죠.<br />
이 모든 과정을 수동으로 진행한다면 개발자의 피로도는 물론, 휴먼 에러 발생 가능성도 높아지게 돼요.<br />

CI/CD는 이러한 반복적인 과정을 자동화하여 개발 생산성을 크게 향상시키고, 더욱 안정적으로 제품을 배포할 수 있도록 도와주는 방법론이에요.<br />
코드를 GitHub에 푸시하는 순간, 정의된 워크플로우에 따라 자동으로 빌드, 테스트, 배포가 진행된다고 생각해보세요.<br />
팀원들은 코드 변경 사항이 문제가 없는지 빠르게 피드백받고, 사용자는 항상 최신 버전의 안정적인 서비스를 경험하게 될 거예요.

### 1️⃣ GitHub Actions, 어떤 장점이 있나요?

GitHub Actions는 GitHub 저장소에서 바로 CI/CD 워크플로우를 정의하고 실행할 수 있는 강력한 도구예요.<br />
별도의 CI/CD 서버를 구축하거나 관리할 필요 없이, `yml` 파일 하나로 모든 과정을 설정할 수 있다는 것이 큰 장점입니다.<br />

주요 장점들은 다음과 같아요.<br />

-   **GitHub 연동성**: 저장소와 완벽하게 통합되어 있어서 설정이 매우 간편해요.<br />
-   **다양한 플랫폼 지원**: Linux, Windows, macOS 등 다양한 운영체제에서 워크플로우를 실행할 수 있어요.<br />
-   **풍부한 마켓플레이스**: 수많은 커뮤니티 액션들을 활용하여 복잡한 작업도 쉽게 구현할 수 있어요.<br />
-   **무료 사용량**: 공개 저장소에 대해서는 무제한, 비공개 저장소도 일정 시간까지는 무료로 사용할 수 있어서 개인 프로젝트나 소규모 팀에 매우 적합해요.

## ⚙️ Next.js App Router CI/CD 워크플로우 구축하기

이제 Next.js App Router 프로젝트를 위한 GitHub Actions CI/CD 워크플로우를 단계별로 구축해볼게요.<br />
이 가이드에서는 코드를 푸시하면 자동으로 빌드, 린트, 타입 체크, 테스트를 수행하고, 최종적으로 Vercel로 배포하는 시나리오를 다룰 거예요.

### 0️⃣ 기본 프로젝트 준비

먼저 Next.js App Router 기반의 프로젝트가 있다고 가정할게요.<br />
새로운 프로젝트라면 다음 명령어로 간단히 생성할 수 있어요.

```bash showLineNumbers
npx create-next-app@latest my-nextjs-app --typescript --eslint --app
cd my-nextjs-app
```

### 1️⃣ GitHub Actions 워크플로우 파일 생성

GitHub Actions 워크플로우는 프로젝트 루트 디렉터리에 `.github/workflows` 경로 안에 `yml` 또는 `yaml` 확장자를 가진 파일로 정의해요.<br />
`main.yml`이라는 파일을 생성해볼게요.

```bash showLineNumbers
mkdir -p .github/workflows
touch .github/workflows/main.yml
```

이제 `main.yml` 파일에 다음 내용을 추가해 주세요.

```yaml showLineNumbers {1-3, 5-7, 9-10}
name: Next.js CI/CD

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

env:
  NODE_VERSION: '20'

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm' # 또는 'yarn', 'pnpm'

      - name: Install dependencies
        run: npm ci # 또는 yarn install --frozen-lockfile, pnpm install --frozen-lockfile

      - name: Run ESLint
        run: npm run lint

      - name: Run TypeScript check
        run: npm run build

      - name: Run tests
        run: npm test # Jest, Vitest 등 설정에 따라

  deploy:
    needs: build-and-test # 빌드 및 테스트 작업이 성공해야 배포 작업 실행
    runs-on: ubuntu-latest
    environment: production # GitHub 환경 설정 (선택 사항)
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
```

### 2️⃣ 워크플로우 내용 상세 설명

위 `main.yml` 파일의 각 부분을 자세히 살펴볼게요.

#### `name` & `on`

-   `name`: 워크플로우의 이름을 정의해요. GitHub Actions UI에서 이 이름으로 워크플로우를 식별할 수 있어요.<br />
-   `on`: 워크플로우가 언제 실행될지 트리거 이벤트를 정의해요.<br />
    -   `push`: `main` 브랜치에 코드가 푸시될 때 실행돼요.<br />
    -   `pull_request`: `main` 브랜치로 풀 리퀘스트가 열리거나 업데이트될 때 실행돼요.

#### `env`

-   `env`: 워크플로우 전체에서 사용할 환경 변수를 정의해요.<br />
    -   `NODE_VERSION`: Node.js 버전을 `20`으로 설정했어요. 이는 `setup-node` 액션에서 사용됩니다.

#### `jobs`

`jobs`는 워크플로우 내에서 실행될 여러 작업(Job)들을 정의하는 섹션이에요.<br />
여기서는 `build-and-test`와 `deploy` 두 가지 작업을 정의했어요.

##### `build-and-test` Job

이 작업은 코드의 빌드, 린트, 타입 체크, 테스트를 담당해요.

-   `runs-on: ubuntu-latest`: 이 작업이 `ubuntu-latest` 환경에서 실행되도록 지정해요.<br />
-   `steps`: 작업을 구성하는 일련의 단계(Step)들이에요.
    -   `actions/checkout@v4`: 저장소 코드를 워크플로우 실행 환경으로 가져와요.<br />
    -   `actions/setup-node@v4`: 지정된 Node.js 버전을 설치하고, `npm` 캐싱을 설정하여 다음 실행 시 의존성 설치 시간을 단축시켜요.<br />
    -   `npm ci`: `package-lock.json` 파일을 기반으로 의존성을 설치해요. `npm install`보다 CI 환경에서 더 안정적이고 빠르게 동작해요.<br />
    -   `npm run lint`, `npm run build`, `npm test`: 각각 ESLint, TypeScript 빌드(타입 체크 포함), 프로젝트 테스트 스크립트를 실행해요. Next.js 프로젝트에서는 `npm run build`가 타입 체크도 함께 수행해요.

##### `deploy` Job

이 작업은 `build-and-test` 작업이 성공적으로 완료된 후에만 실행되어 Vercel로 프로젝트를 배포해요.

-   `needs: build-and-test`: `build-and-test` 작업이 성공적으로 완료되어야 `deploy` 작업이 시작되도록 의존성을 설정해요.<br />
-   `environment: production`: GitHub 환경을 설정할 수 있어요. 이는 배포 보안 강화나 특정 환경 변수 관리에 유용해요.<br />
-   `amondnet/vercel-action@v25`: Vercel로 배포를 자동화해주는 마켓플레이스 액션이에요.<br />
    -   `vercel-token`, `vercel-org-id`, `vercel-project-id`: Vercel 배포에 필요한 인증 정보들이에요. 이 값들은 **GitHub Secrets**에 저장해야 해요.<br />
    -   `vercel-args: '--prod'`: Vercel CLI에 전달할 추가 인자예요. 여기서는 프로덕션 배포임을 명시해요.

<LinkPreviewCard href="https://github.com/amondnet/vercel-action" />

### 3️⃣ GitHub Secrets 설정

배포를 위해서는 민감한 정보인 Vercel API 토큰과 조직/프로젝트 ID를 GitHub 저장소에 안전하게 저장해야 해요.<br />
이때 **GitHub Secrets** 기능을 사용합니다.

1.  **Vercel Token 생성**: Vercel 대시보드에서 Settings > Tokens로 이동하여 새로운 토큰을 생성하세요.<br />
2.  **Vercel Org ID 및 Project ID 확인**: Vercel 프로젝트 대시보드에서 Settings > General로 이동하면 확인할 수 있어요.<br />
3.  **GitHub 저장소 Secrets 설정**: GitHub 저장소로 이동하여 `Settings` > `Secrets and variables` > `Actions` 탭을 클릭하세요.<br />
    `New repository secret` 버튼을 클릭하여 다음 세 가지 Secret을 추가해 주세요.
    -   `VERCEL_TOKEN` (생성한 Vercel 토큰 값)<br />
    -   `VERCEL_ORG_ID` (Vercel 조직 ID 값)<br />
    -   `VERCEL_PROJECT_ID` (Vercel 프로젝트 ID 값)

<Blockquote type="warning">
민감한 정보는 절대 워크플로우 파일(`main.yml`)에 직접 노출하지 마세요. 반드시 GitHub Secrets를 사용해야 해요.
</Blockquote>

## ✨ 워크플로우 최적화 팁

### 0️⃣ 캐싱 활용으로 빌드 시간 단축

의존성 설치는 CI/CD 워크플로우에서 가장 많은 시간을 차지하는 단계 중 하나예요.<br />
GitHub Actions의 캐싱 기능을 활용하면 이 시간을 크게 단축할 수 있어요.

`actions/setup-node@v4` 액션에서 `cache: 'npm'` 옵션을 이미 사용했지만, 좀 더 명시적으로 캐싱 경로를 지정할 수도 있어요.<br />

```diff-yaml showLineNumbers {13-16}
 name: Next.js CI/CD
 
 on:
   push:
     branches:
       - main
   pull_request:
     branches:
       - main
 
 env:
   NODE_VERSION: '20'
 
 jobs:
   build-and-test:
     runs-on: ubuntu-latest
     steps:
       - name: Checkout repository
         uses: actions/checkout@v4
 
       - name: Setup Node.js
         uses: actions/setup-node@v4
         with:
           node-version: ${{ env.NODE_VERSION }}
-          cache: 'npm'
+          cache: 'npm'
+          cache-dependency-path: '**/package-lock.json' # 또는 yarn.lock, pnpm-lock.yaml
 
       - name: Install dependencies
         run: npm ci
 
       - name: Run ESLint
         run: npm run lint
 
       - name: Run TypeScript check
         run: npm run build
 
       - name: Run tests
         run: npm test
 ```

`cache-dependency-path`를 지정하면 `package-lock.json` 파일의 변경 여부에 따라 캐시를 무효화하고 새로 생성할 수 있어서 더 효율적인 캐싱이 가능해요.

### 1️⃣ PR(Pull Request) 시 미리보기 배포 (Preview Deployment)

실제 프로덕션 배포 전에 풀 리퀘스트가 열릴 때마다 미리보기 환경에 배포하여 변경 사항을 검토하는 것은 매우 유용해요.<br />
Vercel은 Git 통합을 통해 자동으로 Preview Deployment를 제공하지만, GitHub Actions를 통해서도 직접 제어할 수 있어요.

`deploy` Job을 수정하여 `pull_request` 이벤트 시 `vercel-args`에 다른 인자를 전달할 수 있어요.<br />

```diff-yaml showLineNumbers {3-4, 18-20}
 jobs:
   build-and-test:
     # ... (생략)
 
   deploy:
     needs: build-and-test
     runs-on: ubuntu-latest
-    environment: production
+    environment: ${{ github.event_name == 'push' && 'production' || 'preview' }}
     steps:
       - name: Checkout repository
         uses: actions/checkout@v4
 
       - name: Deploy to Vercel
         uses: amondnet/vercel-action@v25
         with:
           vercel-token: ${{ secrets.VERCEL_TOKEN }}
           vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
           vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
-          vercel-args: '--prod'
+          vercel-args: ${{ github.event_name == 'push' && '--prod' || '' }}
```

위 설정은 `push` 이벤트일 때만 `production` 환경으로 배포하고 `--prod` 인자를 전달하며, 그 외(`pull_request`)에는 `preview` 환경으로 배포하도록 해요.<br />
`amondnet/vercel-action`은 기본적으로 브랜치 이름과 PR 번호를 기반으로 미리보기 URL을 생성해 줍니다.
