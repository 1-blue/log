## 🤔 클로저(Closure), 왜 중요할까요?

클로저는 JavaScript의 함수형 프로그래밍 패러다임을 이해하고 활용하는 데 있어 필수적인 개념이에요. 단순히 변수에 접근하는 것을 넘어, 함수의 생명 주기와 스코프 관리에 깊이 관여하죠. 클로저를 제대로 이해하면 더 효율적이고 유지보수하기 쉬운 코드를 작성할 수 있게 돼요.

### 0️⃣ 보이지 않는 곳에서 강력하게 작동하는 JS의 핵심

JavaScript는 함수가 일급 객체(First-class Citizen)인 언어예요. 함수를 변수에 할당하거나, 다른 함수의 인자로 전달하거나, 함수가 함수를 반환할 수도 있죠. 클로저는 바로 이 "함수가 함수를 반환할 때" 그 진가를 발휘하는 경우가 많아요.<br />
외부 함수의 실행이 종료된 후에도 내부 함수가 외부 함수의 스코프에 있는 변수에 접근할 수 있게 해주는 마법 같은 존재거든요. 이는 JavaScript의 렉시컬 스코프(Lexical Scope) 규칙과 밀접하게 관련되어 있답니다.

### 1️⃣ 스코프 체인과 변수 생명 주기

모든 함수는 자신이 선언될 때의 렉시컬 환경(Lexical Environment)을 기억해요. 이 렉시컬 환경은 함수가 접근할 수 있는 변수, 함수 선언 등의 정보를 담고 있는 일종의 "사전" 같은 개념이에요. 함수가 실행될 때, 이 렉시컬 환경을 기반으로 스코프 체인을 형성하고, 필요한 변수를 찾아 올라가게 돼요.<br />
일반적으로 함수가 실행을 마치면 그 함수의 로컬 변수들은 메모리에서 해제되지만, 클로저는 외부 함수의 변수들이 "살아남아" 내부 함수에 의해 계속 참조될 수 있도록 해요. 이 점이 클로저를 특별하게 만드는 핵심 원리예요.

## ⚙️ 클로저의 핵심 원리 파헤치기

이제 클로저가 정확히 무엇이고 어떻게 작동하는지 좀 더 자세히 알아볼까요?

### 0️⃣ 클로저는 어떻게 만들어지나요?

클로저는 어떤 함수가 자신의 **렉시컬 환경(Lexical Environment)**을 기억한 채 외부로 반환될 때 형성돼요. 여기서 렉시컬 환경이란, 함수가 선언될 당시의 스코프에 있는 모든 변수와 함수 선언들을 포함하는 개념이에요.<br />
간단히 말해, 함수 A 안에서 함수 B를 정의하고, 함수 B를 함수 A 밖으로 반환하면, 함수 B는 함수 A의 스코프에 있는 변수들을 "기억"하게 되고, 이것이 바로 클로저가 되는 거예요.

<Blockquote type="info">
**렉시컬 스코프(Lexical Scope)란?**<br />
JavaScript에서 변수의 스코프는 함수가 "어디서 선언되었는지"에 따라 결정돼요. 함수가 "어디서 호출되었는지"와는 관계가 없죠. 클로저는 이 렉시컬 스코프의 자연스러운 결과라고 볼 수 있어요.
</Blockquote>

### 1️⃣ 스코프 체인 심화 이해

함수가 어떤 변수에 접근하려고 할 때, 먼저 자신의 로컬 스코프를 탐색하고, 없으면 상위 스코프로, 또 없으면 그 상위 스코프로 계속 찾아 올라가요. 이 연결된 스코프의 고리를 '스코프 체인'이라고 불러요.<br />
클로저는 이 스코프 체인을 통해 외부 함수의 변수에 접근하는 거예요. 외부 함수가 이미 실행을 마치고 스택에서 제거되었더라도, 클로저 내부 함수가 그 변수를 참조하고 있다면 가비지 컬렉션의 대상이 되지 않고 메모리에 유지된답니다.

**📝 간단한 클로저 예시**

아래 코드를 보시면 `createCounter` 함수는 `count`라는 지역 변수를 가지고 있어요. 이 변수는 `increment` 함수에 의해 참조되고 있죠. `createCounter`가 실행을 마친 후에도 `myCounter`는 `count` 변수에 계속 접근할 수 있어요.

```javascript showLineNumbers
function createCounter() {
  let count = 0; // 외부 함수의 지역 변수

  return function increment() { // 내부 함수 (클로저)
    count++;
    console.log(count);
  };
}

const myCounter = createCounter();
myCounter(); // 1
myCounter(); // 2
myCounter(); // 3

const anotherCounter = createCounter();
anotherCounter(); // 1 (myCounter와 독립적인 count 변수를 가짐)
```

`myCounter`와 `anotherCounter`는 각각 `createCounter`가 호출될 때 생성된 독립적인 `count` 변수를 기억하고 있어요. 이것이 바로 클로저의 강력함 중 하나죠.

## 💡 실용적인 클로저 활용 패턴

클로저는 단순히 개념적인 것을 넘어, 실무에서 매우 유용하게 활용될 수 있는 다양한 패턴들을 만들어낼 수 있어요.

### 0️⃣ 정보 은닉과 모듈 패턴

클로저를 사용하면 특정 변수나 함수를 외부에서 직접 접근하지 못하도록 숨기고, 공개하고 싶은 부분만 노출하는 '정보 은닉'이 가능해져요. 이는 마치 객체 지향 프로그래밍의 private 멤버와 유사한 효과를 내죠.

**📝 카운터 팩토리 예시**

아래 `createModule` 함수는 `privateValue`를 외부에 노출하지 않으면서, `publicMethod`를 통해서만 접근하도록 만들어요. 이는 모듈 패턴의 기본적인 형태예요.

```javascript showLineNumbers
function createModule() {
  let privateValue = 0;

  function privateMethod() {
    console.log('Private method called.');
  }

  return {
    publicMethod: function() {
      privateValue++;
      privateMethod();
      console.log('Public method called. Current value:', privateValue);
    },
    getPrivateValue: function() {
      return privateValue;
    }
  };
}

const myModule = createModule();
myModule.publicMethod(); // Private method called. Public method called. Current value: 1
myModule.publicMethod(); // Private method called. Public method called. Current value: 2
// console.log(myModule.privateValue); // undefined (접근 불가)
// myModule.privateMethod(); // TypeError: myModule.privateMethod is not a function (접근 불가)
console.log(myModule.getPrivateValue()); // 2
```

`privateValue`와 `privateMethod`는 `createModule` 함수 스코프 내에 존재하며, 반환된 객체의 `publicMethod`와 `getPrivateValue` 클로저를 통해서만 간접적으로 접근할 수 있어요. 이는 캡슐화를 구현하는 강력한 방법이에요.

### 1️⃣ 부분 적용(Partial Application)과 커링(Currying)

클로저는 여러 인자를 받는 함수를 특정 인자들을 미리 채워 넣어 새로운 함수를 만드는 '부분 적용'이나, 함수가 한 번에 하나의 인자만 받도록 체인 형태로 만드는 '커링' 패턴에 매우 유용해요.

**📝 커링 예시**

`add` 함수는 세 개의 인자를 받지만, 커링을 통해 `add(1)(2)(3)`와 같이 사용할 수 있게 만들 수 있어요. 각 단계에서 클로저가 이전 인자를 기억하고 다음 함수를 반환하죠.

```javascript showLineNumbers
function curry(func) {
  return function curried(...args) {
    if (args.length >= func.length) {
      return func.apply(this, args);
    } else {
      return function(...args2) {
        return curried.apply(this, args.concat(args2));
      };
    }
  };
}

function add(a, b, c) {
  return a + b + c;
}

const curriedAdd = curry(add);

const addOne = curriedAdd(1); // a = 1을 기억하는 클로저
const addOneAndTwo = addOne(2); // b = 2를 기억하는 클로저
console.log(addOneAndTwo(3)); // 6 (1 + 2 + 3)

console.log(curriedAdd(1)(2)(3)); // 6
```

이러한 패턴은 함수 재사용성을 높이고, 특정 조건에 맞는 함수를 동적으로 생성하는 데 도움을 줘요.

### 2️⃣ 이벤트 핸들러와 콜백 함수

루프 안에서 이벤트 핸들러를 등록할 때, 변수 스코프 문제로 예상치 못한 동작을 하는 경우가 있어요. 클로저는 이 문제를 해결하는 데 효과적이에요.

**📝 `setTimeout` 루프 문제 해결 예시**

아래 `badExample`은 모든 `setTimeout`이 루프가 끝난 후의 `i` 값(5)을 참조하게 되지만, `goodExample`은 클로저를 사용하여 각 `i` 값을 독립적으로 캡처해요.

```javascript showLineNumbers
// ❌ 잘못된 예시: 모든 타이머가 5를 출력해요.
function badExample() {
  for (var i = 0; i < 5; i++) {
    setTimeout(function() {
      console.log('Bad Example:', i);
    }, i * 100);
  }
}

// ✅ 클로저를 이용한 올바른 예시: 0, 1, 2, 3, 4를 순서대로 출력해요.
function goodExample() {
  for (var i = 0; i < 5; i++) {
    (function(j) { // 즉시 실행 함수(IIFE)를 사용하여 클로저 생성
      setTimeout(function() {
        console.log('Good Example:', j);
      }, j * 100);
    })(i);
  }
}

// ES6의 `let` 키워드를 사용하면 더 간단하게 해결할 수 있어요. `let`은 블록 스코프를 가지기 때문이죠.
function modernGoodExample() {
  for (let i = 0; i < 5; i++) {
    setTimeout(function() {
      console.log('Modern Good Example:', i);
    }, i * 100);
  }
}

badExample();
// (약간의 딜레이 후) Bad Example: 5
// (약간의 딜레이 후) Bad Example: 5
// ...

goodExample();
// (0.1초 후) Good Example: 0
// (0.2초 후) Good Example: 1
// ...

modernGoodExample();
// (0.1초 후) Modern Good Example: 0
// (0.2초 후) Modern Good Example: 1
// ...
```

`goodExample`에서 즉시 실행 함수(IIFE)는 루프의 각 `i` 값을 `j`라는 새로운 스코프에 캡처하여 클로저를 생성해요. `modernGoodExample`은 `let` 키워드의 블록 스코프 덕분에 더 직관적으로 동일한 결과를 얻을 수 있죠.

### 3️⃣ 메모이제이션(Memoization)

메모이제이션은 함수의 비싼 연산 결과를 캐시하여 동일한 입력에 대해 빠르게 결과를 반환하는 최적화 기법이에요. 클로저는 이전 호출의 결과를 저장하는 데 사용될 수 있어요.

**📝 피보나치 수열 메모이제이션 예시**

재귀적으로 호출되는 피보나치 함수는 중복 계산이 많아 성능이 좋지 않아요. 클로저를 사용하여 계산된 값을 캐싱하면 성능을 크게 향상시킬 수 있어요.

```javascript showLineNumbers
function memoize(func) {
  const cache = {}; // 클로저가 기억할 캐시 객체

  return function(...args) {
    const key = JSON.stringify(args);
    if (cache[key]) {
      console.log('Fetching from cache for key:', key);
      return cache[key];
    }

    console.log('Calculating for key:', key);
    const result = func.apply(this, args);
    cache[key] = result;
    return result;
  };
}

function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}

const memoizedFibonacci = memoize(fibonacci);

console.log(memoizedFibonacci(10)); // 계산 후 캐싱
console.log(memoizedFibonacci(10)); // 캐시에서 가져옴
console.log(memoizedFibonacci(5));  // 계산 후 캐싱
console.log(memoizedFibonacci(5));  // 캐시에서 가져옴
```

`memoize` 함수는 `cache` 객체를 클로저로 유지하며, `memoizedFibonacci` 함수는 이 `cache` 객체를 통해 이전에 계산된 값을 재활용해요. 이렇게 하면 불필요한 재계산을 피할 수 있어 성능 최적화에 큰 도움이 된답니다.

## ⚠️ 클로저 사용 시 흔한 오해와 주의점

클로저는 강력하지만, 몇 가지 오해와 주의할 점이 있어요.

### 0️⃣ 성능 문제? 메모리 누수?

클로저가 외부 스코프의 변수를 참조하면, 그 변수들은 가비지 컬렉션의 대상이 되지 않고 메모리에 유지돼요. 이 때문에 "클로저가 메모리 누수를 일으킨다"는 오해가 생기기도 해요.<br />
하지만 이는 클로저의 **정상적인 동작**이에요. 실제로 필요한 변수만 참조하고 있다면 문제가 되지 않아요. 다만, 불필요하게 거대한 데이터를 클로저가 계속 참조하고 있거나, DOM 요소와 같은 자원을 참조하는 클로저가 제대로 해제되지 않으면 메모리 문제가 발생할 수 있죠.<br />
따라서 클로저를 사용할 때는 어떤 변수를 참조하고 있는지 명확히 인지하고, 필요 없는 참조는 `null` 등으로 명시적으로 해제해 주는 습관을 들이는 것이 좋아요.

### 1️⃣ 클로저가 아닌 경우: 렉시컬 스코프

모든 내부 함수가 클로저인 것은 아니에요. 엄밀히 말하면, **외부 함수의 스코프에 있는 변수를 참조하는 내부 함수**만이 클로저라고 할 수 있어요.<br />
단순히 중첩된 함수는 렉시컬 스코프 규칙에 따라 외부 스코프에 접근할 수 있지만, 외부 스코프의 변수를 "기억하고" 외부로 반환되지 않으면 클로저의 특성을 온전히 활용한다고 보기 어려워요.

### 2️⃣ 'this' 바인딩 문제

클로저는 `this` 키워드의 바인딩 방식에 영향을 주지 않아요. `this`는 함수가 어떻게 호출되었는지에 따라 동적으로 결정되거든요.<br />
클로저 내부에서 `this`를 사용할 때는 주의가 필요하며, 예상치 못한 결과를 피하기 위해 화살표 함수(`=>`)를 사용하거나 `bind`, `call`, `apply` 메서드를 활용하여 `this`를 명시적으로 바인딩하는 것이 일반적이에요.<br />
화살표 함수는 자신이 선언될 당시의 렉시컬 환경의 `this`를 "캡처"하기 때문에, 클로저와 함께 사용하면 `this` 바인딩 문제를 자연스럽게 해결할 수 있는 좋은 방법이 된답니다.

```javascript showLineNumbers
const myObject = {
  value: 10,
  getValue: function() {
    const self = this; // 'this'를 캡처
    setTimeout(function() { // 일반 함수, 'this'는 window 객체를 가리킴
      console.log('Normal function this:', this.value); // undefined (window.value)
      console.log('Captured this:', self.value); // 10
    }, 100);
  },
  getArrowValue: function() {
    setTimeout(() => { // 화살표 함수, 렉시컬 'this'를 캡처
      console.log('Arrow function this:', this.value); // 10
    }, 100);
  }
};

myObject.getValue();
myObject.getArrowValue();
```

`getValue` 내부의 일반 함수는 `setTimeout`에 의해 호출될 때 `this`가 전역 객체(브라우저에서는 `window`)를 가리키게 돼요. 반면 `getArrowValue` 내부의 화살표 함수는 선언될 당시의 `myObject`의 `this`를 렉시컬하게 캡처하여 올바른 `value`를 출력하는 것을 볼 수 있어요.

## 📝 핵심 요약 및 다음 단계

오늘 우리는 JavaScript 클로저의 깊은 세계를 탐험했어요. 클로저는 단순히 변수에 접근하는 것을 넘어, 정보 은닉, 함수형 프로그래밍 패턴 구현, 그리고 복잡한 비동기 로직 처리 등 다양한 상황에서 빛을 발하는 강력한 도구라는 것을 알게 되셨을 거예요.

### 0️⃣ 클로저, 이제 두렵지 않아요!

클로저는 JavaScript의 렉시컬 스코프와 함수의 일급 객체 특성이 결합하여 자연스럽게 발생하는 현상이에요. 외부 함수의 변수를 기억하는 내부 함수라는 핵심 개념만 잘 이해하고 계신다면, 이제 클로저를 두려워할 필요가 없어요.<br />
실무에서 마주하는 다양한 문제들을 클로저를 활용하여 우아하게 해결해 나갈 수 있을 거예요.

### 1️⃣ 더 깊이 탐구해 볼 주제

클로저를 이해하셨다면, 이제 JavaScript의 다른 고급 개념들에도 도전해 볼 때예요. 예를 들어 `Generator` 함수, `Proxy` 객체, 그리고 `Module` 패턴의 진화 과정 등을 공부하시면 JavaScript에 대한 이해도를 더욱 높일 수 있을 거예요.<br />
또한, React의 `useState`, `useRef`, `useEffect`와 같은 훅(Hook)들이 클로저 원리를 어떻게 활용하는지 탐구해 보는 것도 아주 유익한 학습 경험이 될 거예요. 함수형 컴포넌트에서 상태를 기억하고 업데이트하는 방식이 클로저와 밀접하게 관련되어 있거든요.
