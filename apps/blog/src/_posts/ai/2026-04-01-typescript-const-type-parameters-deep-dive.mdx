## 🤔 문제 인식: 제네릭 함수의 넓은 타입 추론

### 0️⃣ 왜 리터럴 타입 추론이 중요한가요?

안녕하세요! 10년 이상 풀스택 개발자로 일하며 수많은 프로젝트를 경험하고, 기술 블로그 SEO 전문가로 활동하는 AI 개발자 블루예요.<br />
오늘은 TypeScript의 강력한 기능 중 하나인 '타입 추론(Type Inference)'을 더욱 정교하게 만들 수 있는 `const` Type Parameters에 대해 이야기해 보려고 해요.

초중급 개발자분들이 TypeScript를 사용하다 보면, 제네릭(Generics) 함수에서 특정 값을 인자로 넘겼을 때 기대했던 것보다 타입이 너무 넓게 추론되어 당황하는 경우가 종종 있어요.<br />
예를 들어, `greet("블루")`라고 호출하면 "블루"라는 정확한 문자열 리터럴 타입으로 추론되기를 바라지만, 실제로는 `string` 타입으로 추론되는 경우가 발생할 수 있거든요.<br />
이런 넓은 타입 추론은 때로는 유연성을 제공하지만, 엄격한 타입 안정성이 필요한 상황에서는 오히려 버그를 유발하거나 코드의 의도를 모호하게 만들 수 있어요.

### 1️⃣ 기존 제네릭 타입 추론의 한계점

TypeScript의 제네릭은 함수나 클래스, 인터페이스를 유연하게 만들면서도 타입 안전성을 유지할 수 있도록 돕는 아주 유용한 도구예요.<br />
하지만 특정 상황, 특히 함수 인자로 리터럴 값(예: `"success"`, `123`, `[1, 2, 3]`)을 전달할 때 기본 타입 추론 방식은 의도와 다르게 동작할 수 있어요.

간단한 예시를 살펴볼까요?

```tsx showLineNumbers
function identity<T>(arg: T): T {
  return arg;
}

const str = identity("hello"); // str의 타입은 "hello"가 아닌 string으로 추론돼요.
const num = identity(123);    // num의 타입은 123이 아닌 number로 추론돼요.
const arr = identity([1, 2, 3]); // arr의 타입은 [number, number, number]가 아닌 number[]로 추론돼요.

console.log(typeof str); // "string"
console.log(typeof num); // "number"
console.log(typeof arr); // "object"
```

위 코드에서 `identity` 함수는 제네릭 `T`를 받아요.<br />
우리는 `identity("hello")`를 호출했을 때 `str` 변수의 타입이 `"hello"` 리터럴 타입이 되기를 기대할 수 있지만, 실제로는 더 넓은 `string` 타입으로 추론돼요.<br />
마찬가지로 숫자 리터럴은 `number`로, 배열 리터럴은 `number[]`와 같이 요소의 타입이 넓게 추론되죠.<br />
이는 TypeScript가 기본적으로 제네릭 타입 변수를 다룰 때 "최소 공통 타입(Least Common Type)"을 찾으려는 경향이 있기 때문이에요.<br />
이러한 동작은 대부분의 경우에 합리적이지만, 때로는 리터럴 값을 그대로 타입으로 사용하고 싶을 때 제약이 될 수 있어요.

## ⚙️ 해결 방법: `const` Type Parameters 도입

### 0️⃣ `const` Type Parameters의 핵심 아이디어

TypeScript 5.0부터 도입된 `const` Type Parameters는 이러한 문제를 해결하기 위해 고안되었어요.<br />
제네릭 타입 매개변수에 `const` 한정자를 붙이면, TypeScript는 해당 타입 매개변수에 전달되는 리터럴 인자들을 가능한 한 가장 좁은(const-like) 리터럴 타입으로 추론하려고 시도해요.<br />
이는 마치 변수에 `const`를 붙여 재할당을 막고 값을 고정하는 것처럼, 타입 추론에서도 리터럴 값을 최대한 고정된 타입으로 유지하겠다는 의미로 이해할 수 있어요.

<Blockquote type="tip">
`const` Type Parameters는 제네릭 타입 매개변수(`T`)에 `const` 키워드를 추가하여, 해당 `T`에 할당되는 리터럴 값을 `as const` 어설션이 적용된 것처럼 가장 좁은 리터럴 타입으로 추론하도록 강제하는 기능이에요.
</Blockquote>

### 1️⃣ `const` Type Parameters 적용 방법

`const` Type Parameters를 사용하는 방법은 아주 간단해요.<br />
제네릭 타입 매개변수를 선언할 때 `const` 키워드를 앞에 붙여주기만 하면 돼요.

```tsx showLineNumbers {1}
function identityWithConst<const T>(arg: T): T {
  return arg;
}

const strWithConst = identityWithConst("hello"); // strWithConst의 타입은 "hello"로 추론돼요!
const numWithConst = identityWithConst(123);    // numWithConst의 타입은 123으로 추론돼요!
const arrWithConst = identityWithConst([1, 2, 3]); // arrWithConst의 타입은 readonly [1, 2, 3]으로 추론돼요!

console.log(strWithConst); // "hello"
console.log(numWithConst); // 123
console.log(arrWithConst); // [1, 2, 3]

// 타입 확인 (컴파일 시점에만 유효)
// strWithConst = "world"; // Error: Type '"world"' is not assignable to type '"hello"'.
// arrWithConst.push(4);   // Error: Property 'push' does not exist on type 'readonly [1, 2, 3]'.
```

위 코드를 보시면, `identityWithConst<const T>` 함수를 사용했을 때 `strWithConst`의 타입은 정확히 `"hello"` 리터럴 타입으로 추론되는 것을 확인할 수 있어요.<br />
숫자 `123`도 마찬가지로 `123` 리터럴 타입으로 추론되고요.<br />
특히 배열의 경우, `readonly [1, 2, 3]`와 같이 `readonly` 속성이 붙은 튜플 타입으로 추론되어 더욱 강력한 타입 안정성을 제공해요.<br />
이는 `as const` 어설션을 사용한 것과 유사한 효과를 제네릭 타입 추론 단계에서 얻을 수 있게 해주는 것이죠.

## 🧪 실용적인 활용 예시

`const` Type Parameters는 단순히 리터럴 타입을 좁히는 것을 넘어, 실무에서 마주하는 다양한 타입 안정성 문제를 해결하는 데 큰 도움이 돼요.

### 0️⃣ 타입 안전성을 높이는 설정 객체 정의

특정 컴포넌트나 유틸리티 함수가 고정된 문자열 리터럴을 키로 가지는 설정 객체를 받을 때 `const` Type Parameters를 활용하면 실수를 줄일 수 있어요.

```tsx showLineNumbers
type FeatureConfig<const T extends Record<string, string>> = {
  [K in keyof T]: T[K];
};

function createFeature<const T extends Record<string, string>>(config: FeatureConfig<T>) {
  console.log("Feature created with config:", config);
  return config;
}

// ✅ 올바른 사용 예시
const myFeatureConfig = createFeature({
  name: "UserProfile",
  version: "1.0.0",
  status: "enabled",
});

// myFeatureConfig의 타입은 { name: "UserProfile"; version: "1.0.0"; status: "enabled"; } 로 추론돼요.
// 즉, 각 속성의 값이 리터럴 타입으로 고정돼요.

// myFeatureConfig.name = "AdminPanel"; // Error: Type '"AdminPanel"' is not assignable to type '"UserProfile"'.

// ❌ 잘못된 키를 사용하면 에러 발생
// createFeature({
//   name: "Blog",
//   ver: "1.1.0" // Error: Property 'ver' does not exist on type '{ name: string; version: string; status: string; }'.
//                // (여기서는 T의 제약조건 때문에 string으로 추론되지만, 실제 사용 시 더 정교한 에러를 유도할 수 있어요.)
// });
```

위 예시에서 `createFeature` 함수는 `const T`를 사용하여 `config` 객체의 속성 값들을 리터럴 타입으로 추론하도록 해요.<br />
이렇게 하면 `myFeatureConfig.name`에 다른 문자열을 할당하려 할 때 타입 에러가 발생하여, 불변성을 유지하고 의도치 않은 변경을 방지할 수 있어요.<br />
이는 특히 설정 객체나 상태 객체의 특정 필드가 고정된 값을 가져야 할 때 유용해요.

### 1️⃣ 타입 안전한 이벤트 리스너 정의

이벤트 시스템을 구현할 때, 특정 이벤트 이름에 대한 리스너를 등록할 때 `const` Type Parameters를 활용하여 이벤트 이름의 오타를 방지하고 타입 안정성을 높일 수 있어요.

```tsx showLineNumbers {1, 10}
type EventMap = {
  "userLoggedIn": { userId: string };
  "itemAdded": { itemId: string; quantity: number };
  "appError": { message: string; code: number };
};

class EventEmitter<const T extends EventMap> {
  private listeners: { [K in keyof T]?: ((payload: T[K]) => void)[] } = {};

  on<K extends keyof T>(eventName: K, listener: (payload: T[K]) => void) {
    if (!this.listeners[eventName]) {
      this.listeners[eventName] = [];
    }
    this.listeners[eventName]?.push(listener);
  }

  emit<K extends keyof T>(eventName: K, payload: T[K]) {
    this.listeners[eventName]?.forEach(listener => listener(payload));
  }
}

const myEmitter = new EventEmitter<EventMap>();

myEmitter.on("userLoggedIn", (data) => {
  console.log("User logged in:", data.userId); // data는 { userId: string } 타입
});

myEmitter.on("itemAdded", (data) => {
  console.log("Item added:", data.itemId, data.quantity); // data는 { itemId: string; quantity: number } 타입
});

// myEmitter.on("unknownEvent", () => {}); // Error: Argument of type '"unknownEvent"' is not assignable to parameter of type 'keyof EventMap'.

myEmitter.emit("userLoggedIn", { userId: "user123" });
myEmitter.emit("itemAdded", { itemId: "productA", quantity: 5 });
// myEmitter.emit("appError", { msg: "Something went wrong" }); // Error: Argument of type '{ msg: string; }' is not assignable to parameter of type '{ message: string; code: number; }'.
```

`EventEmitter` 클래스에 `const T extends EventMap`를 적용함으로써, `on` 및 `emit` 메서드의 `eventName` 인자가 `EventMap`에 정의된 리터럴 키(`"userLoggedIn"`, `"itemAdded"`, `"appError"`)로만 허용되도록 강제할 수 있어요.<br />
이는 이벤트 이름을 잘못 입력하는 휴먼 에러를 컴파일 시점에 잡아내어 런타임 오류를 방지하는 데 매우 효과적이에요.

### 2️⃣ `as const` 어설션과의 차이점

`const` Type Parameters를 이야기할 때 `as const` 어설션과의 차이점을 이해하는 것이 중요해요.<br />
`as const`는 변수나 리터럴에 직접 적용하여 해당 값의 타입을 가능한 가장 좁은 리터럴 타입으로 만드는 역할을 해요.

```tsx showLineNumbers
const userStatus = "active"; // type: string
const userStatusAsConst = "active" as const; // type: "active"

const numbers = [1, 2, 3]; // type: number[]
const numbersAsConst = [1, 2, 3] as const; // type: readonly [1, 2, 3]
```

`as const`는 특정 *값*에 대한 타입 추론을 변경하는 반면, `const` Type Parameters는 *제네릭 타입 매개변수*에 대한 타입 추론 방식을 변경해요.<br />
즉, 함수를 호출할 때 전달되는 인자의 타입을 `as const`를 사용한 것처럼 추론하도록 지시하는 것이죠.<br />
함수 인자로 들어오는 값에 매번 `as const`를 붙여주는 번거로움 없이, 제네릭 함수 자체에서 더 스마트한 추론을 할 수 있게 해준다는 점에서 큰 차이가 있어요.

<Blockquote type="info">
`as const`는 특정 표현식의 타입을 리터럴로 고정하는 반면, `const` Type Parameters는 제네릭 타입 추론 메커니즘 자체를 변경하여 리터럴 인자를 자동으로 `as const`처럼 처리하도록 만들어요.
</Blockquote>

## 📝 정리: `const` Type Parameters, 언제 사용해야 할까요?

### 0️⃣ 핵심 요약 및 활용 가이드라인

TypeScript의 `const` Type Parameters는 제네릭을 사용하는 함수나 클래스에서 리터럴 타입의 인자를 더욱 정교하게 다루고 싶을 때 매우 유용한 기능이에요.<br />
이를 통해 다음과 같은 이점을 얻을 수 있어요.

*   **정확한 리터럴 타입 추론:** `"success"`, `123`, `[1, 2, 3]`와 같은 리터럴 값이 제네릭 함수에 전달될 때, `string`, `number`, `(string | number)[]`와 같이 넓게 추론되는 대신 정확한 리터럴 타입이나 `readonly` 튜플 타입으로 추론돼요.
*   **타입 안정성 강화:** 설정 객체, 이벤트 이름, 상태 값 등 특정 필드가 고정된 리터럴 값을 가져야 하는 상황에서 잘못된 값을 할당하는 것을 컴파일 시점에 방지할 수 있어요.
*   **코드의 의도 명확화:** 함수가 특정 리터럴 타입의 인자를 기대한다는 것을 제네릭 선언 자체에서 명확히 표현할 수 있어요.
*   **불필요한 `as const` 감소:** 함수 호출 시 인자에 일일이 `as const`를 붙이지 않아도 되므로 코드 가독성이 향상돼요.
