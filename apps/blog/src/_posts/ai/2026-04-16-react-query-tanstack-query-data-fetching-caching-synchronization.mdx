## 🤔 React Query, 왜 필요할까요?

React 애플리케이션에서 서버 데이터를 다루는 것은 생각보다 복잡한 일이에요. 단순히 데이터를 불러오는 것 외에도 다양한 고려사항들이 있거든요.<br />

### 0️⃣ 데이터 관리의 어려움

프런트엔드 개발자라면 누구나 한 번쯤 겪어봤을 만한 상황들이 있어요. 데이터를 불러오는 동안 로딩 스피너를 보여줘야 하고, 에러가 발생하면 사용자에게 알려야 해요. 또한, 한 번 불러온 데이터는 캐싱해서 불필요한 네트워크 요청을 줄여야 하고, 데이터가 변경되면 화면을 최신 상태로 업데이트해야 하죠. 이 모든 과정을 직접 구현하는 것은 생각보다 많은 노력과 코드를 필요로 해요.<br />

### 1️⃣ 기존 방식의 한계

기존에는 `useEffect` 훅과 `useState`를 조합하여 데이터를 페칭하고 관리하는 경우가 많았어요. 하지만 이 방식은 몇 가지 한계를 가지고 있어요.

<Blockquote type="warning">
**기존 `useEffect` + `useState` 방식의 한계점**<br />

- **보일러플레이트 코드 증가:** 로딩, 에러, 데이터 상태를 각각 관리해야 해서 코드가 길어져요.<br />
- **수동적인 캐싱:** 데이터를 캐싱하려면 직접 로직을 구현해야 하고, 캐시 무효화 및 업데이트가 복잡해요.<br />
- **데이터 동기화 문제:** 여러 컴포넌트에서 동일한 데이터를 요청할 때, 각기 다른 시점에 페칭이 일어나 데이터 불일치가 발생할 수 있어요.<br />
- **포커스 시 리페칭:** 브라우저 탭을 다시 포커스 했을 때 데이터를 최신화하는 로직을 직접 구현해야 해요.<br />
- **오프라인 모드:** 네트워크 연결이 끊겼을 때 캐시된 데이터를 보여주는 로직 구현이 번거로워요.

</Blockquote>

이러한 문제들을 해결하기 위해 탄생한 것이 바로 React Query와 같은 서버 상태 관리 라이브러리들이에요.

## 🚀 React Query 핵심 개념 파헤치기

React Query는 서버 상태를 클라이언트 상태와 분리하여 관리하는 데 초점을 맞춰요. 이를 통해 개발자는 서버 데이터의 복잡성을 줄이고, 사용자 경험을 향상시키는 데 집중할 수 있게 돼요.<br />

### 0️⃣ Query Client와 Provider

React Query를 사용하려면 가장 먼저 `QueryClient` 인스턴스를 생성하고, 이를 `QueryClientProvider`를 통해 애플리케이션 전역에 제공해야 해요. `QueryClient`는 React Query의 모든 캐싱, 페칭, 동기화 로직을 담당하는 핵심 객체예요.

```tsx showLineNumbers
// src/main.tsx 또는 src/App.tsx
import React from "react";
import ReactDOM from "react-dom/client";
import App from "./App";
import "./index.css";

import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { ReactQueryDevtools } from "@tanstack/react-query-devtools";

// 쿼리 클라이언트 인스턴스 생성
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      // 기본적으로 쿼리 실패 시 재시도하지 않도록 설정 (실제 앱에서는 필요에 따라 조절해요)
      retry: false,
      // 쿼리가 'stale' 상태로 간주되기 전까지의 시간 (ms)
      // 이 시간 동안은 캐시된 데이터를 사용하고, 백그라운드에서 리페칭하지 않아요.
      staleTime: 1000 * 5, // 5초 동안은 'fresh' 상태 유지
      // 쿼리 데이터가 캐시에서 제거되기 전까지의 시간 (ms)
      // 이 시간이 지나면 가비지 컬렉션 대상이 돼요.
      gcTime: 1000 * 60 * 5, // 5분 동안 캐시에 유지
    },
  },
});

ReactDOM.createRoot(document.getElementById("root")!).render(
  <React.StrictMode>
    <QueryClientProvider client={queryClient}>
      <App />
      {/* 개발 도구는 개발 환경에서만 활성화하는 것이 좋아요 */}
      <ReactQueryDevtools initialIsOpen={false} />
    </QueryClientProvider>
  </React.StrictMode>,
);
```

`defaultOptions`를 통해 전역적인 쿼리 설정을 할 수 있어요. 특히 `staleTime`과 `gcTime`은 React Query의 강력한 캐싱 전략의 핵심이니, 잠시 후 자세히 알아볼게요.

### 1️⃣ `useQuery` 훅: 데이터 페칭의 시작

`useQuery` 훅은 React Query의 가장 기본적인 훅으로, 데이터를 페칭하고 관리하는 데 사용돼요. 이 훅은 세 가지 주요 상태 (`loading`, `error`, `success`)를 자동으로 처리해 주고, 캐싱, 백그라운드 리페칭 등 다양한 기능을 제공해요.

`useQuery`는 최소한 두 개의 인자를 필요로 해요.

1.  **`queryKey`**: 쿼리를 고유하게 식별하는 키 배열이에요. 이 키를 통해 React Query는 캐시된 데이터를 관리하고, 특정 쿼리를 무효화하거나 업데이트할 수 있어요. 같은 데이터를 요청하는 모든 `useQuery` 호출은 동일한 `queryKey`를 사용해야 해요.<br />
2.  **`queryFn`**: 데이터를 실제로 페칭하는 비동기 함수예요. 이 함수는 `Promise`를 반환해야 해요.

```tsx showLineNumbers
// src/components/Posts.tsx
import React from "react";
import { useQuery } from "@tanstack/react-query";

interface Post {
  id: number;
  title: string;
  body: string;
}

const fetchPosts = async (): Promise<Post[]> => {
  const response = await fetch("https://jsonplaceholder.typicode.com/posts");
  if (!response.ok) {
    throw new Error("Failed to fetch posts");
  }
  return response.json();
};

export default function Posts() {
  const { data, isLoading, isError, error } = useQuery<Post[], Error>({
    queryKey: ["posts"], // 쿼리를 식별하는 고유 키
    queryFn: fetchPosts, // 데이터를 페칭하는 함수
  });

  if (isLoading) {
    return <Blockquote type="info">게시글을 불러오는 중이에요...</Blockquote>;
  }

  if (isError) {
    return (
      <Blockquote type="error">에러가 발생했어요: {error?.message}</Blockquote>
    );
  }

  return (
    <div>
      <h2>✨ 게시글 목록이에요</h2>
      <ul>
        {data?.map((post) => (
          <li key={post.id}>
            <h3>{post.title}</h3>
            <p>{post.body.substring(0, 100)}...</p>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

위 예시처럼 `useQuery` 훅을 사용하면 `isLoading`, `isError`, `data` 등의 상태를 쉽게 활용할 수 있어요. React Query는 자동으로 데이터를 캐싱하고, 백그라운드에서 데이터를 최신 상태로 유지하려고 노력해요.

<LinkPreviewCard href="https://tanstack.com/query/latest/docs/react/guides/queries" />

### 2️⃣ 강력한 캐싱 전략: `staleTime`과 `gcTime`

React Query의 핵심은 **캐싱**과 **데이터 신선도 관리**에 있어요. 모든 쿼리 데이터는 `QueryClient` 내부에 캐시되며, 이 데이터는 `fresh` 또는 `stale` 상태를 가질 수 있어요.

- **`fresh` (신선한) 상태**: 데이터가 최신 상태라고 간주되는 기간이에요. 이 기간 동안에는 React Query는 네트워크 요청 없이 캐시된 데이터를 즉시 반환해요. `staleTime` 옵션으로 이 기간을 설정할 수 있어요. 기본값은 `0`으로, 쿼리가 마운트되자마자 `stale` 상태로 간주돼요.<br />
- **`stale` (오래된) 상태**: 데이터가 최신이 아닐 수도 있다고 간주되는 상태예요. `stale` 상태의 쿼리는 특정 조건(예: 컴포넌트 마운트, 윈도우 포커스, 네트워크 재연결 등)에서 백그라운드에서 자동으로 리페칭을 시도해요. 이 과정에서 사용자에게는 캐시된 데이터를 보여주면서 부드러운 사용자 경험을 제공할 수 있어요.

`gcTime` (Garbage Collection Time, 이전에는 `cacheTime`)은 쿼리 데이터가 캐시에서 제거되기 전까지의 시간이에요. 쿼리 인스턴스가 더 이상 사용되지 않을 때 (예: 컴포넌트 언마운트), React Query는 `gcTime` 동안 데이터를 캐시에 보관해요. 이 시간 내에 동일한 쿼리가 다시 사용되면 네트워크 요청 없이 캐시된 데이터를 즉시 사용할 수 있어요. `gcTime`이 지나면 해당 쿼리 데이터는 가비지 컬렉션 대상이 되어 캐시에서 완전히 제거돼요. 기본값은 `1000 * 60 * 5` (5분)이에요.

**`staleTime`과 `gcTime`의 관계**

- `staleTime`: 데이터의 **논리적인 신선도**를 정의해요. 이 시간 동안은 리페칭 없이 캐시 데이터를 사용해요.<br />
- `gcTime`: 데이터의 **물리적인 캐시 유지 시간**을 정의해요. 컴포넌트 언마운트 후 캐시에서 데이터가 언제 사라질지를 결정해요.

<Blockquote type="tip">
  `staleTime`을 길게 설정하면 네트워크 요청을 줄일 수 있지만, 데이터의 최신성이
  떨어질 수 있어요. 반대로 `staleTime`을 짧게 설정하거나 `0`으로 두면 항상 최신
  데이터를 유지하려고 시도하지만, 네트워크 요청이 늘어날 수 있으니
  애플리케이션의 요구사항에 맞춰 적절히 조절해야 해요.
</Blockquote>

### 3️⃣ 백그라운드 리페칭과 데이터 동기화

React Query는 `stale` 상태의 쿼리에 대해 다음과 같은 상황에서 자동으로 백그라운드 리페칭을 시도하여 데이터를 최신 상태로 동기화해요.

- 새로운 쿼리 인스턴스가 마운트될 때 (`staleTime`이 지난 경우)<br />
- 윈도우가 다시 포커스 될 때 (브라우저 탭 활성화 시)<br />
- 네트워크가 다시 연결될 때<br />
- 옵션으로 설정된 `refetchInterval`에 따라 주기적으로<br />
- `queryClient.invalidateQueries()` 또는 `queryClient.refetchQueries()`를 통해 수동으로

이러한 자동 동기화 기능은 개발자가 직접 `setInterval`이나 `addEventListener`를 사용하여 데이터를 최신화하는 복잡한 로직을 구현할 필요 없이, 항상 최신 데이터를 사용자에게 제공할 수 있게 도와줘요.

## 🛠️ 실전 활용: 데이터 페칭 및 최적화 패턴

이제 React Query의 핵심 개념을 바탕으로 실제 애플리케이션에서 자주 사용되는 패턴들을 살펴볼게요.

### 0️⃣ 기본 데이터 페칭 예시

앞서 살펴본 `Posts` 컴포넌트처럼 간단한 데이터 페칭은 `useQuery`를 통해 쉽게 구현할 수 있어요. 여기서 중요한 것은 `queryKey`를 통해 쿼리를 명확하게 식별하는 것이에요.

만약 특정 ID를 가진 게시글을 불러오고 싶다면, `queryKey`에 ID를 추가하여 각 게시글을 고유하게 캐싱할 수 있어요.

```tsx showLineNumbers
// src/components/PostDetail.tsx
import React from "react";
import { useQuery } from "@tanstack/react-query";

interface Post {
  id: number;
  title: string;
  body: string;
}

interface PostDetailProps {
  postId: number;
}

const fetchPostById = async (id: number): Promise<Post> => {
  const response = await fetch(
    `https://jsonplaceholder.typicode.com/posts/${id}`,
  );
  if (!response.ok) {
    throw new Error(`Failed to fetch post with ID ${id}`);
  }
  return response.json();
};

export default function PostDetail({ postId }: PostDetailProps) {
  const { data, isLoading, isError, error } = useQuery<Post, Error>({
    // queryKey에 postId를 포함하여 각 게시글을 고유하게 캐싱해요.
    queryKey: ["post", postId],
    queryFn: () => fetchPostById(postId), // queryFn은 인자를 받지 않는 함수여야 하므로 래핑해요.
    // postId가 유효하지 않으면 쿼리를 비활성화할 수 있어요.
    enabled: !!postId,
  });

  if (isLoading) {
    return (
      <Blockquote type="info">
        게시글 상세 정보를 불러오는 중이에요...
      </Blockquote>
    );
  }

  if (isError) {
    return (
      <Blockquote type="error">에러가 발생했어요: {error?.message}</Blockquote>
    );
  }

  if (!data) {
    return <Blockquote type="warning">게시글을 찾을 수 없어요.</Blockquote>;
  }

  return (
    <div>
      <h2>📝 {data.title}</h2>
      <p>{data.body}</p>
    </div>
  );
}
```

`queryKey`는 배열 안에 문자열, 숫자, 객체 등 어떤 직렬화 가능한 값이라도 포함할 수 있어요. 이를 통해 쿼리의 인자를 명확히 표현하고, React Query가 캐시를 효율적으로 관리하도록 할 수 있어요.

### 1️⃣ 캐시 무효화 (Invalidation)와 수동 업데이트 (Mutation)

데이터를 생성, 업데이트, 삭제하는 작업(Mutation)은 서버 상태를 변경하므로, 관련 쿼리의 캐시를 최신 상태로 유지해야 해요. React Query의 `useMutation` 훅과 `queryClient.invalidateQueries()`를 통해 이를 효과적으로 처리할 수 있어요.

```tsx showLineNumbers
// src/components/CreatePostForm.tsx
import React, { useState } from "react";
import { useMutation, useQueryClient } from "@tanstack/react-query";

interface NewPost {
  title: string;
  body: string;
  userId: number;
}

interface CreatedPost extends NewPost {
  id: number;
}

const createPost = async (newPost: NewPost): Promise<CreatedPost> => {
  const response = await fetch("https://jsonplaceholder.typicode.com/posts", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(newPost),
  });
  if (!response.ok) {
    throw new Error("Failed to create post");
  }
  return response.json();
};

export default function CreatePostForm() {
  const queryClient = useQueryClient();
  const [title, setTitle] = useState("");
  const [body, setBody] = useState("");

  const createPostMutation = useMutation<CreatedPost, Error, NewPost>({
    mutationFn: createPost,
    onSuccess: () => {
      // 게시글 생성 성공 시, 'posts' 쿼리 캐시를 무효화하여 새로운 데이터를 다시 불러오도록 해요.
      queryClient.invalidateQueries({ queryKey: ["posts"] });
      setTitle("");
      setBody("");
      alert("게시글이 성공적으로 생성되었어요!");
    },
    onError: (error) => {
      alert(`게시글 생성에 실패했어요: ${error.message}`);
    },
  });

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    if (!title || !body) return;
    createPostMutation.mutate({ title, body, userId: 1 });
  };

  return (
    <form
      onSubmit={handleSubmit}
      style={{ border: "1px solid #ccc", padding: "20px", borderRadius: "8px" }}
    >
      <h3>새 게시글 작성하기 ✍️</h3>
      <div>
        <label htmlFor="title">제목:</label>
        <br />
        <input
          id="title"
          type="text"
          value={title}
          onChange={(e) => setTitle(e.target.value)}
          disabled={createPostMutation.isPending}
          style={{ width: "100%", padding: "8px", marginBottom: "10px" }}
        />
      </div>
      <div>
        <label htmlFor="body">내용:</label>
        <br />
        <textarea
          id="body"
          value={body}
          onChange={(e) => setBody(e.target.value)}
          disabled={createPostMutation.isPending}
          rows={5}
          style={{ width: "100%", padding: "8px", marginBottom: "10px" }}
        />
      </div>
      <button type="submit" disabled={createPostMutation.isPending}>
        {createPostMutation.isPending ? "생성 중..." : "게시글 생성하기"}
      </button>
      {createPostMutation.isError && (
        <Blockquote type="error">
          오류: {createPostMutation.error?.message}
        </Blockquote>
      )}
      {createPostMutation.isSuccess && (
        <Blockquote type="success">게시글 생성 성공!</Blockquote>
      )}
    </form>
  );
}
```

`onSuccess` 콜백에서 `queryClient.invalidateQueries({ queryKey: ['posts'] })`를 호출하여 `posts` 쿼리의 캐시를 무효화했어요. 이렇게 하면 `posts` 쿼리를 사용하는 모든 컴포넌트에서 데이터가 `stale` 상태로 간주되고, 다음 번 렌더링 또는 윈도우 포커스 시 백그라운드에서 자동으로 리페칭되어 최신 데이터를 가져오게 돼요. 이는 데이터 일관성을 유지하는 매우 효과적인 방법이에요.

### 2️⃣ 무한 스크롤 구현 (`useInfiniteQuery`)

무한 스크롤은 많은 양의 데이터를 효율적으로 보여줄 때 유용한 패턴이에요. React Query는 `useInfiniteQuery` 훅을 통해 무한 스크롤 기능을 쉽게 구현할 수 있도록 도와줘요.

`useInfiniteQuery`는 `getNextPageParam` 옵션을 통해 다음 페이지를 가져올 방법을 정의하고, `fetchNextPage` 함수를 통해 다음 페이지를 요청할 수 있게 해요.

```tsx showLineNumbers
// src/components/InfinitePosts.tsx
import React from "react";
import { useInfiniteQuery } from "@tanstack/react-query";

interface Post {
  id: number;
  title: string;
  body: string;
}

interface PostsPage {
  data: Post[];
  nextCursor: number | undefined;
}

const fetchInfinitePosts = async ({ pageParam = 0 }): Promise<PostsPage> => {
  // pageParam은 0부터 시작한다고 가정하고, 한 페이지에 10개씩 불러와요.
  const limit = 10;
  const start = pageParam * limit;
  const response = await fetch(
    `https://jsonplaceholder.typicode.com/posts?_start=${start}&_limit=${limit}`,
  );
  if (!response.ok) {
    throw new Error("Failed to fetch infinite posts");
  }
  const data = await response.json();
  // 다음 페이지가 있는지 확인하기 위해 현재 페이지의 마지막 ID를 커서로 사용해요.
  const nextCursor = data.length === limit ? pageParam + 1 : undefined;
  return { data, nextCursor };
};

export default function InfinitePosts() {
  const {
    data,
    fetchNextPage,
    hasNextPage,
    isFetchingNextPage,
    isLoading,
    isError,
    error,
  } = useInfiniteQuery<PostsPage, Error>({
    queryKey: ["infinitePosts"],
    queryFn: fetchInfinitePosts,
    initialPageParam: 0, // 첫 페이지 파라미터
    // 다음 페이지를 가져오는 방법을 정의해요.
    getNextPageParam: (lastPage) => lastPage.nextCursor,
  });

  if (isLoading) {
    return (
      <Blockquote type="info">
        무한 스크롤 게시글을 불러오는 중이에요...
      </Blockquote>
    );
  }

  if (isError) {
    return (
      <Blockquote type="error">에러가 발생했어요: {error?.message}</Blockquote>
    );
  }

  return (
    <div>
      <h2>🔄 무한 스크롤 게시글이에요</h2>
      <ul>
        {data?.pages.map((page, i) => (
          <React.Fragment key={i}>
            {page.data.map((post) => (
              <li key={post.id}>
                <h3>{post.title}</h3>
                <p>{post.body.substring(0, 100)}...</p>
              </li>
            ))}
          </React.Fragment>
        ))}
      </ul>
      <button
        onClick={() => fetchNextPage()}
        disabled={!hasNextPage || isFetchingNextPage}
        style={{ marginTop: "20px", padding: "10px 20px", cursor: "pointer" }}
      >
        {isFetchingNextPage
          ? "더 불러오는 중..."
          : hasNextPage
            ? "더 불러오기"
            : "모든 게시글을 불러왔어요!"}
      </button>
      {isFetchingNextPage && (
        <Blockquote type="info">다음 페이지를 불러오는 중이에요...</Blockquote>
      )}
    </div>
  );
}
```

`data.pages` 배열에 각 페이지의 데이터가 저장되어 있어요. `fetchNextPage`를 호출하면 `getNextPageParam`에서 반환된 값을 `queryFn`의 `pageParam`으로 전달하여 다음 페이지를 가져오게 돼요. 이를 통해 복잡한 무한 스크롤 로직을 간결하게 구현할 수 있어요.

### 3️⃣ SSR/SSG 환경에서의 Hydration

Next.js와 같은 SSR/SSG 프레임워크에서 React Query를 사용하면, 서버에서 미리 데이터를 페칭하여 HTML에 포함시키고, 클라이언트에서는 이 데이터를 재사용(Hydration)하여 초기 로딩 성능을 크게 향상시킬 수 있어요.

이를 위해서는 서버에서 `QueryClient`를 생성하고 데이터를 미리 페칭한 후, `dehydrate` 함수를 사용하여 쿼리 캐시를 직렬화하고, 클라이언트에서 `hydrate` 함수를 사용하여 이 캐시를 재구성해야 해요.

```tsx showLineNumbers
// pages/posts/index.tsx (Next.js Pages Router 예시)
import {
  dehydrate,
  QueryClient,
  HydrationBoundary,
} from "@tanstack/react-query";
import Posts from "../../components/Posts";
import { fetchPosts } from "../../api/posts"; // fetchPosts 함수는 위에서 정의한 것과 동일하다고 가정해요.

export async function getServerSideProps() {
  const queryClient = new QueryClient();

  // 서버에서 'posts' 쿼리를 미리 페칭해요.
  await queryClient.prefetchQuery({
    queryKey: ["posts"],
    queryFn: fetchPosts,
  });

  return {
    props: {
      dehydratedState: dehydrate(queryClient),
    },
  };
}

export default function PostsPage({ dehydratedState }) {
  return (
    // HydrationBoundary로 서버에서 페칭된 캐시를 클라이언트에서 재사용해요.
    <HydrationBoundary state={dehydratedState}>
      <h1>모든 게시글</h1>
      <Posts />
    </HydrationBoundary>
  );
}
```

`getServerSideProps`에서 `prefetchQuery`를 사용하여 서버에서 `posts` 데이터를 미리 가져온 후, `dehydrate(queryClient)`를 통해 이 캐시를 직렬화된 형태로 반환해요. 클라이언트에서는 `HydrationBoundary` 컴포넌트에 이 `dehydratedState`를 전달하여 서버에서 가져온 데이터를 즉시 사용할 수 있게 돼요. 이로 인해 초기 로딩 시 깜빡임 없이 데이터를 보여줄 수 있고, SEO에도 유리해요.

<Blockquote type="tip">
  Next.js App Router에서는 `QueryClientProvider`를 클라이언트 컴포넌트로
  분리하고, 서버 컴포넌트에서 데이터를 페칭한 후 클라이언트 컴포넌트로 전달하는
  방식으로 Hydration을 구현할 수 있어요. 이는 Pages Router와는 약간 다른 접근
  방식을 가지니 공식 문서를 참고해 주세요.
</Blockquote>

## 📝 정리하며: React Query로 개발 생산성을 높여요

React Query는 단순히 데이터를 페칭하는 것을 넘어, 서버 상태 관리의 복잡성을 해결하고 개발 생산성과 사용자 경험을 동시에 향상시키는 강력한 도구예요.

### 0️⃣ 핵심 요약

- **서버 상태와 클라이언트 상태 분리:** React Query는 서버 상태 관리에 특화되어 복잡한 비동기 로직을 추상화해요.<br />
- **지능적인 캐싱:** `staleTime`과 `gcTime`을 통해 데이터의 신선도와 캐시 유지 시간을 효율적으로 관리해요.<br />
- **자동 동기화:** 윈도우 포커스, 네트워크 재연결 등 다양한 조건에서 자동으로 데이터를 리페칭하여 최신 상태를 유지해요.<br />
- **간결한 API:** `useQuery`, `useMutation`, `useInfiniteQuery` 등 직관적인 훅을 통해 데이터를 쉽게 다룰 수 있어요.<br />
- **SSR/SSG 지원:** Next.js와 같은 프레임워크에서 Hydration을 통해 초기 로딩 성능과 SEO를 개선할 수 있어요.

### 1️⃣ 다음 스텝

이 글에서 다룬 내용은 React Query의 기본적인 개념과 활용 패턴이에요. 실제 프로젝트에서는 더 많은 기능과 옵션들을 활용하여 애플리케이션의 요구사항에 맞는 최적화를 진행할 수 있어요. 다음 단계로 아래 내용을 학습해 보시길 추천해요.

- **옵티미스틱 업데이트:** Mutation 시 서버 응답 전에 UI를 미리 업데이트하여 사용자 경험을 향상시키는 방법<br />
- **쿼리 캐시 수동 업데이트:** `queryClient.setQueryData()`를 사용하여 캐시 데이터를 직접 업데이트하는 방법<br />
- **커스텀 훅:** `useQuery`를 감싸는 커스텀 훅을 만들어 쿼리 로직을 재사용하고 추상화하는 방법<br />
- **에러 핸들링 전략:** `onError` 콜백, `useErrorBoundary`, `suspense` 옵션을 활용한 고급 에러 처리

React Query는 한 번 익혀두면 React 개발을 훨씬 즐겁고 효율적으로 만들어 줄 거예요. 여러분의 프로젝트에서 React Query의 강력함을 충분히 경험해 보시길 바라요!
