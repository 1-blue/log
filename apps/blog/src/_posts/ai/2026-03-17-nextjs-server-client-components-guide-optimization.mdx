## 🤔 Next.js App Router의 핵심, Server & Client Components?

Next.js 13부터 도입된 App Router는 React Server Components(RSC)를 기반으로 하고 있어요.<br />
이 새로운 패러다임은 웹 애플리케이션의 성능과 개발 경험을 한 단계 끌어올렸지만, 동시에 Server Components와 Client Components 중 어떤 것을 사용해야 할지 많은 개발자분들이 혼란을 겪고 계시는 것 같아요.

### 0️⃣ 왜 이 둘을 알아야 할까요?

App Router 환경에서는 기본적으로 모든 컴포넌트가 Server Components로 간주돼요.<br />
하지만 특정 기능, 예를 들어 사용자 인터랙션이 필요하거나 브라우저 API를 사용해야 할 때는 Client Components를 명시적으로 사용해야 하죠.<br />
이 둘의 차이점을 명확히 이해하고 적절하게 활용하는 것이 Next.js 애플리케이션의 성능을 최적화하고 복잡성을 줄이는 데 매우 중요해요. 잘못된 선택은 불필요한 클라이언트 측 JavaScript 번들 크기를 늘리거나, 서버 리소스를 낭비하게 만들 수 있거든요.

### 1️⃣ 기존 React 방식과의 차이점은 무엇인가요?

기존 React 개발에서는 모든 컴포넌트가 기본적으로 클라이언트에서 렌더링되고 실행되었어요.<br />
Next.js Pages Router에서도 SSR(Server-Side Rendering)을 통해 초기 HTML을 서버에서 생성했지만, 이후에는 클라이언트에서 모든 JavaScript를 다운로드하여 하이드레이션(Hydration) 과정을 거쳤죠.<br />
Server Components는 이러한 방식을 근본적으로 바꿔요.<br />
이제 컴포넌트의 일부는 서버에서만 렌더링되고, 클라이언트로 전송되는 JavaScript 번들에는 포함되지 않아요.<br />
이는 클라이언트 측 JavaScript 양을 획기적으로 줄여 웹 페이지 로딩 속도를 향상시키는 데 큰 기여를 한답니다.

## ⚙️ Server Components 깊이 이해하기

Server Components는 Next.js App Router의 가장 큰 특징 중 하나예요.<br />
그럼 Server Components가 정확히 무엇이고, 언제 사용해야 하며, 어떤 제약사항이 있는지 자세히 알아볼까요?

### 0️⃣ Server Components란 무엇인가요?

Server Components는 이름 그대로 서버에서 렌더링되는 React 컴포넌트예요.<br />
이 컴포넌트들은 빌드 시 또는 요청 시 서버에서 실행되어 HTML과 React Server Component Payload(RSC Payload)라는 특별한 데이터 형식으로 변환된 후 클라이언트로 전송돼요.<br />
클라이언트는 이 HTML을 즉시 렌더링하고, RSC Payload를 사용하여 나머지 UI 트리를 구성하죠.<br />
가장 중요한 점은 Server Components는 클라이언트 측 JavaScript 번들에 포함되지 않는다는 것이에요.<br />
따라서 번들 크기를 줄이고, 초기 로딩 성능을 크게 개선할 수 있어요.

### 1️⃣ 언제 Server Components를 사용해야 할까요? (장점 위주)

Server Components는 다음과 같은 상황에서 빛을 발해요.

- **데이터 페칭 (Data Fetching):** 데이터베이스 접근, 외부 API 호출 등 서버에서만 가능한 데이터 작업을 직접 수행할 수 있어요.<br />
  클라이언트에서 데이터를 가져올 필요 없이 서버에서 데이터를 가져와 바로 렌더링하므로, 추가적인 클라이언트-서버 왕복(roundtrip)이 줄어들어요.
- **백엔드 리소스 접근:** 파일 시스템 접근, 환경 변수 사용 등 서버 환경에만 접근 가능한 리소스들을 안전하게 사용할 수 있어요.
- **번들 크기 최적화:** 클라이언트 측 JavaScript 번들에 포함되지 않으므로, 복잡한 로직이나 대용량 라이브러리(예: 마크다운 파서, 날짜 라이브러리)를 서버에서만 사용하고 클라이언트에는 결과물만 전달하여 번들 크기를 크게 줄일 수 있어요.
- **초기 페이지 로딩 속도 개선:** 서버에서 미리 HTML을 생성하여 전송하기 때문에, 사용자는 더 빠르게 콘텐츠를 볼 수 있어요. 이는 Core Web Vitals 점수 향상에도 도움이 된답니다.
- **보안 강화:** 민감한 정보(API 키, 데이터베이스 자격 증명 등)를 클라이언트에 노출하지 않고 서버에서 안전하게 처리할 수 있어요.

### 2️⃣ Server Components의 제약사항은 무엇인가요?

Server Components는 강력하지만, 몇 가지 제약사항이 있어요.

- **클라이언트 측 상태 관리 (`useState`, `useReducer` 등) 불가:** Server Components는 서버에서 한 번 렌더링된 후에는 더 이상 상호작용하지 않아요.<br />
  따라서 클라이언트에서 변경될 수 있는 상태를 가질 수 없어요.
- **클라이언트 측 이펙트 (`useEffect` 등) 불가:** 브라우저 API(예: `window`, `document`)나 DOM에 직접 접근하는 로직을 사용할 수 없어요.
- **이벤트 핸들러 (`onClick`, `onChange` 등) 불가:** 사용자 인터랙션을 직접 처리할 수 없어요.
- **브라우저 API 접근 불가:** `localStorage`, `sessionStorage` 등 브라우저 전용 API를 사용할 수 없어요.
- **`'use client'` 지시자 사용 불가:** Server Components 파일에는 `'use client'` 지시자를 선언할 수 없어요.

<Blockquote type="info">
  기억하세요! Server Components는 '정적인 HTML'을 만드는 데 최적화되어 있어요.
  <br />
  사용자와의 상호작용이 필요 없는 콘텐츠를 보여주는 데 아주 효과적이죠.
</Blockquote>

## 💡 Client Components 깊이 이해하기

Server Components가 정적 콘텐츠에 강하다면, Client Components는 상호작용에 특화되어 있어요.

### 0️⃣ Client Components란 무엇인가요?

Client Components는 기존 React 컴포넌트와 동일하게 동작해요.<br />
파일 상단에 `'use client'` 지시자를 추가하여 명시적으로 선언해야 해요.<br />
이 지시자를 만나면 Next.js는 해당 컴포넌트와 그 자식 컴포넌트들을 클라이언트 측 번들에 포함시키고, 클라이언트에서 렌더링 및 하이드레이션이 이루어지도록 해요.<br />
즉, Server Components가 서버에서 HTML을 생성하여 보내면, Client Components는 그 HTML 위에서 JavaScript를 통해 상호작용을 추가하는 역할을 하는 셈이죠.

### 1️⃣ 언제 Client Components를 사용해야 할까요? (장점 위주)

Client Components는 다음과 같은 상황에서 필수적이에요.

- **사용자 인터랙션:** `onClick`, `onChange`, `onSubmit` 등 이벤트 핸들러를 사용하여 사용자 입력을 처리해야 할 때 사용해요.
- **상태 관리:** `useState`, `useReducer`, `useContext`와 같은 React 훅을 사용하여 클라이언트 측 상태를 관리해야 할 때 사용해요.
- **브라우저 API 접근:** `window`, `document`, `localStorage`, `geolocation` 등 브라우저 전용 API에 접근해야 할 때 사용해요.
- **커스텀 훅:** `useEffect`를 포함하는 커스텀 훅을 사용해야 할 때, 해당 훅을 사용하는 컴포넌트는 Client Component여야 해요.
- **클라이언트 측 라이브러리 사용:** 상태 관리 라이브러리(Redux, Zustand), 애니메이션 라이브러리(Framer Motion) 등 클라이언트 측에서만 동작하는 라이브러리를 사용할 때 해당 라이브러리를 사용하는 컴포넌트는 Client Component여야 해요.

### 2️⃣ Server Components와 함께 사용하는 방법은 무엇인가요?

Client Components는 Server Components의 자식으로 렌더링될 수 있어요.<br />
이것이 바로 Next.js App Router의 강력한 점인데요.<br />
부모 Server Component는 서버에서 데이터를 가져오고, 자식 Client Component는 그 데이터를 받아 클라이언트에서 상호작용을 추가하는 방식으로 동작할 수 있어요.<br />
하지만 **주의할 점**이 있어요.<br />
Client Component 안에서는 Server Component를 직접 `import`할 수 없어요.<br />
만약 Client Component 내에서 Server Component를 사용하고 싶다면, Server Component를 Client Component의 `children` prop으로 전달하는 패턴을 사용해야 한답니다.

<Blockquote type="warning">
  Client Component 안에서 Server Component를 import하는 것은 불가능해요.
  <br />
  `children` prop으로 전달하는 "Pass Server Components to Client Components as
  Props" 패턴을 기억해 주세요.
</Blockquote>

## 🚀 실전! 현명한 선택과 최적화 전략

이제 Server Components와 Client Components의 기본 개념을 이해하셨으니, 실무에서 어떻게 현명하게 선택하고 최적화할 수 있는지 구체적인 전략을 알아볼까요?

### 0️⃣ 기본은 Server Components, 필요할 때 Client Components로!

Next.js App Router의 핵심 철학은 "기본적으로 Server Components를 사용하고, 클라이언트 상호작용이 필요한 부분만 Client Components로 분리하라"는 것이에요.<br />
불필요하게 `'use client'` 지시자를 남용하면 Server Components의 장점(번들 크기 감소, 초기 로딩 속도 개선)을 잃을 수 있어요.<br />
항상 "이 컴포넌트가 클라이언트에서 꼭 상호작용해야 하는가?"라는 질문을 던져보세요.<br />
만약 아니라면 Server Component로 유지하는 것이 좋습니다.

### 1️⃣ Client Components를 'boundary'로 활용하는 패턴

앞서 Client Component는 Server Component를 import할 수 없다고 말씀드렸죠?<br />
이 문제를 해결하고 Server Components의 이점을 최대한 활용하는 좋은 패턴이 바로 Client Component를 '경계(boundary)'로 사용하는 것이에요.<br />
즉, Server Component에서 데이터를 페칭하고, 그 데이터를 Client Component에 props로 전달하며, Client Component의 `children`으로 Server Component를 다시 전달하는 방식이죠.<br />
이렇게 하면 Client Component는 최소한의 상호작용 로직만 담당하고, 대부분의 콘텐츠는 Server Component에서 효율적으로 처리할 수 있어요.

<LinkPreviewCard href="https://nextjs.org/docs/app/building-your-application/rendering/server-components#passing-server-components-to-client-components-as-props" />

### 2️⃣ 데이터 페칭 전략: Server Components에서만 가능할까요?

Next.js App Router에서는 Server Components에서 데이터를 페칭하는 것이 가장 권장되는 방식이에요.<br />
하지만 Client Components에서도 데이터를 페칭할 수 없는 것은 아니에요.<br />
`useEffect` 훅을 사용하여 클라이언트에서 데이터를 가져올 수 있죠.<br />
하지만 이 경우, 초기 로딩 시 데이터가 없는 상태로 렌더링되고, 데이터 로딩 후 다시 렌더링되는 워터폴(waterfall) 현상이 발생할 수 있어요.<br />
따라서 특별한 경우가 아니라면, 초기 데이터는 Server Components에서 가져오는 것이 좋아요.<br />
실시간 업데이트가 필요하거나 사용자 상호작용에 따라 동적으로 데이터를 가져와야 할 때만 Client Components에서 페칭하는 것을 고려해 보세요.

## 🧪 코드 예시로 보는 활용법

이론만으로는 부족하죠? 실제 코드를 통해 Server Components와 Client Components의 활용법을 살펴볼게요.

### 0️⃣ Server Component에서 데이터 페칭하기

먼저, 서버에서 데이터를 가져와 표시하는 Server Component를 만들어볼게요.<br />
이 컴포넌트는 `'use client'` 지시자가 없으므로 기본적으로 Server Component로 동작해요.

```tsx showLineNumbers {4-6, 11}
// app/products/page.tsx (Server Component)
import { Product } from "@/lib/types"; // 타입 정의는 Client/Server 모두 사용 가능해요.

async function getProducts(): Promise<Product[]> {
  // 실제 DB 접근이나 외부 API 호출 로직이 여기에 들어갈 수 있어요.
  const res = await fetch("https://api.example.com/products", {
    cache: "no-store",
  });
  if (!res.ok) {
    throw new Error("Failed to fetch products");
  }
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts(); // 서버에서 직접 데이터 페칭!

  return (
    <div className="container mx-auto p-4">
      <h1 className="mb-4 text-2xl font-bold">상품 목록</h1>
      <ul className="grid grid-cols-1 gap-4 md:grid-cols-3">
        {products.map((product) => (
          <li key={product.id} className="rounded-lg border p-4 shadow">
            <h2 className="text-xl font-semibold">{product.name}</h2>
            <p className="text-gray-600">{product.description}</p>
            <p className="mt-2 text-lg font-bold">${product.price}</p>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

위 코드에서 `getProducts` 함수는 서버에서만 실행되며, `fetch` API를 사용하여 데이터를 가져와요.<br />
`cache: 'no-store'` 옵션은 데이터가 캐시되지 않도록 설정한 예시인데, 실제 프로덕션에서는 Next.js의 강력한 캐싱 전략을 활용하는 것이 좋아요.

### 1️⃣ Client Component로 상호작용 추가하기

이제 위 상품 목록에 '장바구니 담기' 버튼과 같은 상호작용을 추가하고 싶다고 가정해 볼까요?<br />
이때는 Client Component를 사용해야 해요.

```tsx showLineNumbers {1, 5, 8-10, 15}
// components/AddToCartButton.tsx (Client Component)
"use client"; // 이 지시자가 핵심이에요!

import { useState } from "react";

interface AddToCartButtonProps {
  productId: string;
  productName: string;
}

export default function AddToCartButton({
  productId,
  productName,
}: AddToCartButtonProps) {
  const [quantity, setQuantity] = useState(0);

  const handleAddToCart = () => {
    setQuantity((prev) => prev + 1);
    alert(`${productName} ${quantity + 1}개를 장바구니에 담았어요!`);
    // 실제 장바구니 로직 (API 호출 등)은 여기에 구현해요.
  };

  return (
    <button
      onClick={handleAddToCart}
      className="mt-4 rounded bg-blue-500 px-4 py-2 font-bold text-white hover:bg-blue-700"
    >
      🛒 장바구니 담기 ({quantity})
    </button>
  );
}
```

`'use client'` 지시자를 통해 이 컴포넌트가 클라이언트에서 실행됨을 명시했어요.<br />
`useState` 훅과 `onClick` 이벤트 핸들러를 사용하여 사용자 상호작용을 처리하는 것을 볼 수 있죠.

### 2️⃣ Server Component 내부에 Client Component 사용하기

이제 이 `AddToCartButton`을 Server Component인 `ProductsPage`에서 사용해볼게요.<br />
Server Component는 Client Component를 자식으로 포함할 수 있어요.

```diff-tsx showLineNumbers {1, 23-26}
// app/products/page.tsx (Server Component)
import { Product } from '@/lib/types';
+import AddToCartButton from '@/components/AddToCartButton'; // Client Component import!

async function getProducts(): Promise<Product[]> {
  const res = await fetch('https://api.example.com/products', { cache: 'no-store' });
  if (!res.ok) {
    throw new Error('Failed to fetch products');
  }
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts();

  return (
    <div className="container mx-auto p-4">
      <h1 className="text-2xl font-bold mb-4">상품 목록</h1>
      <ul className="grid grid-cols-1 md:grid-cols-3 gap-4">
        {products.map((product) => (
          <li key={product.id} className="border p-4 rounded-lg shadow">
            <h2 className="text-xl font-semibold">{product.name}</h2>
            <p className="text-gray-600">{product.description}</p>
            <p className="text-lg font-bold mt-2">${product.price}</p>
            {/* Server Component 안에 Client Component를 렌더링해요. */}
            <AddToCartButton productId={product.id} productName={product.name} />
          </li>
        ))}
      </ul>
    </div>
  );
}
```

보시는 것처럼 `ProductsPage` (Server Component) 안에서 `AddToCartButton` (Client Component)을 직접 import하고 렌더링할 수 있어요.<br />
Next.js는 이 구조를 파악하여 `ProductsPage`는 서버에서 렌더링하고, `AddToCartButton`은 클라이언트 번들에 포함시켜 하이드레이션 과정을 거치도록 처리한답니다.<br />
이렇게 하면 상품 목록 자체는 서버에서 빠르게 로딩되고, 개별 상품의 장바구니 버튼만 클라이언트에서 상호작용 기능을 가지게 되어 효율적이에요.

## 📝 정리하며: 핵심 요약 및 다음 스텝

지금까지 Next.js App Router의 Server Components와 Client Components에 대해 자세히 알아보았어요.<br />
어떻게 하면 이 둘을 실전에서 효과적으로 활용할 수 있는지 감이 좀 잡히셨나요?

### 0️⃣ Server/Client Components 선택 체크리스트

다음 질문들을 스스로에게 던져보면서 어떤 컴포넌트를 사용해야 할지 결정해 보세요.

- **데이터 페칭이 필요한가요?**
  - 예: 서버에서 데이터를 가져와야 한다면, **Server Component**를 고려하세요.
- **클라이언트 측 상태(state)가 필요한가요?**
  - 예: `useState`, `useReducer`와 같은 훅이 필요하다면, **Client Component**를 사용해야 해요.
- **사용자 인터랙션(클릭, 입력 등)이 필요한가요?**
  - 예: `onClick`, `onChange`와 같은 이벤트 핸들러가 필요하다면, **Client Component**를 사용해야 해요.
- **브라우저 전용 API(window, localStorage 등)에 접근해야 하나요?**
  - 예: 브라우저 API가 필요하다면, **Client Component**를 사용해야 해요.
- **번들 크기를 최대한 줄이고 싶나요?**
  - 예: 클라이언트 번들에서 제외하고 싶다면, **Server Component**를 사용하는 것이 좋아요.

<Blockquote type="tip">
  기본적으로는 Server Components를 사용하고, 위에 제시된 조건에 해당할 때만
  Client Components로 전환하는 것이 가장 좋은 전략이에요.
</Blockquote>

### 1️⃣ 더 나은 Next.js 개발을 위한 조언

Server Components와 Client Components는 Next.js App Router의 강력한 기능이지만, 올바르게 사용하지 않으면 오히려 복잡성을 증가시키거나 성능을 저해할 수 있어요.<br />
항상 "최대한 Server Component로 만들고, 꼭 필요한 부분만 Client Component로 분리한다"는 원칙을 마음속에 새겨두시면 좋을 것 같아요.<br />
또한, `<Suspense>`와 `<ErrorBoundary>`를 적절히 활용하여 데이터 로딩 중이나 에러 발생 시 사용자 경험을 개선하는 것도 잊지 마세요.<br />
이 두 가지 개념을 마스터하신다면, Next.js를 이용한 고성능 웹 애플리케이션 개발에 한 걸음 더 다가설 수 있을 거예요.
