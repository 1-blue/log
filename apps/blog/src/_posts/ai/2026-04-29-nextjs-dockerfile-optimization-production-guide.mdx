## 🤔 Next.js 앱, 왜 Docker로 배포해야 할까요?

최신 웹 애플리케이션 개발에 있어 Docker와 같은 컨테이너 기술은 이제 선택이 아닌 필수가 되고 있어요. 특히 Next.js와 같은 프레임워크를 사용할 때 Docker를 활용하면 얻을 수 있는 이점이 정말 많아요.<br />
하지만 단순히 `next build` 후 이미지를 만드는 것만으로는 부족할 수 있어요.

### 0️⃣ 컨테이너화의 이점: 일관된 환경, 쉬운 확장, 빠른 배포

Docker를 사용하면 개발 환경과 프로덕션 환경 간의 불일치 문제를 해결할 수 있어요. "제 컴퓨터에서는 잘 되는데요?"라는 말은 더 이상 통하지 않게 되죠.<br />
모든 종속성과 설정이 컨테이너 안에 캡슐화되어 있어, 어디서든 동일하게 작동하는 환경을 보장해 줘요.<br />
또한, 컨테이너는 가볍고 빠르기 때문에 애플리케이션의 확장(Scale-out)이 용이하고, CI/CD 파이프라인에 통합하여 배포 자동화를 구축하기도 편리해요.

### 1️⃣ 일반적인 Dockerfile의 한계: 이미지 크기, 빌드 속도

초보 개발자분들이 처음 Dockerfile을 작성할 때 흔히 하는 실수는 개발 환경에서 사용되는 모든 파일과 빌드 도구를 최종 이미지에 포함시키는 거예요.<br />
이런 방식은 몇 가지 문제를 발생시켜요.

*   **불필요하게 큰 이미지 크기:** 개발 의존성, 빌드 캐시, 소스 맵 등이 최종 이미지에 포함되어 이미지 크기가 수백 MB에서 GB 단위로 커질 수 있어요. 이는 배포 시간 증가와 스토리지 비용 상승으로 이어져요.<br />
*   **느린 빌드 속도:** 매번 모든 의존성을 새로 설치하고 빌드하는 과정은 CI/CD 파이프라인에서 상당한 시간을 소모하게 만들어요.<br />
*   **보안 취약점 증가:** 불필요한 파일과 도구는 잠재적인 보안 공격 표면을 증가시킬 수 있어요.

이러한 문제들을 해결하기 위해 Next.js 앱에 최적화된 Dockerfile 전략이 필요해요.

## ⚙️ Next.js 프로덕션 Dockerfile 핵심 전략

Next.js 애플리케이션을 Docker로 배포할 때 가장 중요한 것은 '프로덕션 환경에 적합한' 이미지를 만드는 거예요.<br />
여기에는 크게 세 가지 핵심 전략이 있어요.

### 0️⃣ 멀티스테이지 빌드로 이미지 크기 최소화

멀티스테이지 빌드는 여러 개의 `FROM` 구문을 사용하여 Docker 이미지 빌드 프로세스를 여러 단계로 나누는 기법이에요.<br />
각 단계는 이전 단계의 결과물을 활용할 수 있으며, 최종 이미지에는 필요한 최소한의 파일만 복사해서 포함시켜요.<br />
Next.js의 경우, 보통 `builder` 스테이지에서 의존성을 설치하고 애플리케이션을 빌드한 다음, `runner` 스테이지에서 빌드된 결과물과 런타임에 필요한 최소한의 파일만 복사해서 최종 이미지를 만들게 돼요.<br />
이를 통해 개발 도구, 소스 코드, 중간 빌드 파일 등이 최종 이미지에 포함되는 것을 방지하여 이미지 크기를 획기적으로 줄일 수 있어요.

### 1️⃣ 효율적인 레이어 캐싱으로 빌드 속도 향상

Docker 빌드는 각 `RUN` 명령을 실행할 때마다 새로운 '레이어'를 생성하고 캐시해요.<br />
이 캐시를 효율적으로 활용하면 빌드 속도를 크게 향상시킬 수 있어요.<br />
핵심은 자주 변경되지 않는 명령을 `Dockerfile`의 상단에 배치하고, 자주 변경되는 명령을 하단에 배치하는 거예요.<br />
예를 들어, `package.json`과 `pnpm-lock.yaml` (또는 `yarn.lock`, `package-lock.json`) 파일을 먼저 복사하여 의존성을 설치하는 단계를 분리하면, 소스 코드 변경 시에도 의존성 설치 단계는 캐시를 재활용하여 빠르게 빌드할 수 있어요.

### 2️⃣ 보안 강화: 비루트(Non-root) 사용자 실행

기본적으로 Docker 컨테이너는 `root` 사용자로 실행돼요. 하지만 이는 보안상 좋지 않은 관행이에요.<br />
컨테이너 내에서 `root` 권한으로 프로세스가 실행될 경우, 잠재적인 공격자가 컨테이너를 탈취했을 때 호스트 시스템에 더 큰 피해를 줄 수 있어요.<br />
따라서 `Dockerfile` 내에서 별도의 비루트 사용자를 생성하고, 해당 사용자로 애플리케이션을 실행하도록 설정하는 것이 보안 모범 사례예요.

## 🧪 단계별 Dockerfile 작성 및 최적화

이제 실제로 Next.js 애플리케이션을 위한 최적화된 Dockerfile을 작성해 볼 시간이에요.<br />
여기서는 `pnpm`을 사용한다고 가정하지만, `npm`이나 `yarn`도 동일한 원리로 적용할 수 있어요.

### 0️⃣ 기본 `Dockerfile` 구조 (개발/프로덕션 분리)

먼저 프로덕션 배포에 초점을 맞춘 `Dockerfile`을 작성할 거예요.<br />
개발 환경에서는 보통 `next dev`를 직접 실행하거나, `docker compose`를 사용하여 개발 서버를 띄우는 것이 더 효율적일 수 있어요.

<LinkPreviewCard href="https://nextjs.org/docs/app/building-your-application/deploying/docker" />

### 1️⃣ `.dockerignore` 파일로 불필요한 파일 제외

`.dockerignore` 파일은 `git-ignore`와 유사하게, Docker 이미지 빌드 시 컨텍스트에서 제외할 파일이나 디렉터리를 지정하는 역할을 해요.<br />
이를 통해 이미지 크기를 줄이고, 민감한 정보가 이미지에 포함되는 것을 방지할 수 있어요.

```bash showLineNumbers
.env
.env.local
.env.development.local
.env.production.local
node_modules
npm-debug.log
yarn-debug.log
yarn-error.log
.pnpm-store
.next
.git
.gitignore
README.md
Dockerfile
docker-compose*
```

<Blockquote type="tip">
`.dockerignore`에 `.next`를 포함시키는 이유는 빌드된 결과물을 `builder` 스테이지에서 직접 생성하고 `runner` 스테이지로 복사할 것이기 때문이에요. 로컬에서 빌드된 `.next` 폴더가 컨테이너로 복사되지 않도록 방지해 줘요.<br />
`node_modules`도 마찬가지로 컨테이너 내부에서 새로 설치할 것이므로 제외해요.
</Blockquote>

### 2️⃣ `package.json` 스크립트 설정

Next.js 애플리케이션의 `package.json` 파일에는 프로덕션 빌드 및 시작 스크립트가 정의되어 있어야 해요.<br />
특히 `start` 스크립트는 `next start`를 사용하여 프로덕션 서버를 시작하도록 설정해야 해요.

```json showLineNumbers {7,8}
{
  "name": "my-nextjs-app",
  "version": "0.1.0",
  "private": true,
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start",
    "lint": "next lint"
  },
  "dependencies": {
    "next": "^14.0.0",
    "react": "^18",
    "react-dom": "^18"
  }
}
```

### 3️⃣ 최종 프로덕션 `Dockerfile` 예시

이제 위에서 설명한 전략들을 모두 포함한 최종 `Dockerfile`을 살펴볼게요.<br />
`pnpm`을 기준으로 작성되었으며, `npm`이나 `yarn`을 사용한다면 해당 패키지 매니저에 맞게 명령어만 변경해 주시면 돼요.

```dockerfile showLineNumbers
# --- 1. Builder Stage (빌드 환경) ---
FROM node:20-alpine AS builder

# 작업 디렉토리 설정
WORKDIR /app

# pnpm 설치
RUN npm install -g pnpm

# 의존성 캐싱을 위한 파일 복사
# package.json과 lock 파일을 먼저 복사하여, 이 파일들이 변경되지 않는 한
# pnpm install 단계는 캐시를 재활용하도록 유도해요.
COPY package.json pnpm-lock.yaml* ./

# 의존성 설치
# --frozen-lockfile: lock 파일 기반으로 정확히 설치하여 빌드 일관성 유지
RUN pnpm install --frozen-lockfile

# 나머지 소스 코드 복사
COPY . .

# Next.js 앱 빌드
# STANDALONE 출력 모드 사용 (최소한의 파일만 포함하여 이미지 크기 최적화)
# https://nextjs.org/docs/app/api-reference/next-config-js/output#standalone
ENV NEXT_SHARP_PATH=/app/node_modules/sharp
RUN pnpm run build


# --- 2. Runner Stage (실행 환경) ---
# 최종 프로덕션 이미지는 더 가벼운 nodejs 런타임만 포함해요.
FROM node:20-alpine AS runner

# Next.js 공식 권장 사항: 비루트 사용자 사용
# https://nextjs.org/docs/app/building-your-application/deploying/docker#non-root-user
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs

# 작업 디렉토리 설정 및 권한 설정
# builder 스테이지에서 생성된 .next/standalone을 복사할 위치
WORKDIR /app

# builder 스테이지에서 빌드된 .next/standalone 폴더 복사
# 이 폴더에는 Next.js 앱 실행에 필요한 모든 파일이 포함되어 있어요.
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./.
ext/standalone

# public 폴더 복사
COPY --from=builder --chown=nextjs:nodejs /app/public ./public

# package.json 복사 (start 스크립트 실행을 위해)
# nextjs 유저에게 권한 부여
COPY --from=builder --chown=nextjs:nodejs /app/package.json ./package.json

# 환경 변수 설정 (런타임에 필요한 변수들)
ENV NODE_ENV=production
ENV PORT=3000

# 컨테이너 포트 노출
EXPOSE 3000

# 비루트 사용자로 전환
USER nextjs

# Health Check 설정 (옵션)
# 컨테이너가 정상적으로 작동하는지 주기적으로 확인해요.
HEALTHCHECK --interval=30s --timeout=30s --start-period=5s --retries=3 CMD node ./.next/standalone/server.js || exit 1

# 애플리케이션 시작 명령어
CMD ["pnpm", "start"]
```

<Blockquote type="info">
**`NEXT_SHARP_PATH` 환경 변수**: Next.js 14부터 이미지 최적화 라이브러리인 Sharp가 Node.js 네이티브 모듈로 제공되면서, Docker 환경에서 빌드 시 이 경로를 명시해 주어야 하는 경우가 있어요. `standalone` 모드에서는 `node_modules/sharp` 경로를 지정해 주면 돼요.<br />
**`output: 'standalone'`**: `next.config.js`에 이 설정을 추가하면 Next.js가 프로덕션 빌드 시 `.next/standalone` 폴더에 모든 의존성을 포함한 최소한의 파일만 생성해 줘요. Docker 이미지 크기를 줄이는 데 매우 효과적이에요.
</Blockquote>

### 4️⃣ `next.config.js` 설정 (선택 사항이지만 권장)

위 `Dockerfile`은 `output: 'standalone'` 옵션을 사용하는 것을 전제로 해요.<br />
`next.config.js` 파일에 아래와 같이 추가하여 이미지 크기를 더욱 최적화할 수 있어요.

```javascript showLineNumbers {3}
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'standalone',
  // 기타 Next.js 설정들...
};

module.exports = nextConfig;
```

## 🚀 실제 배포 시 고려사항

최적화된 Dockerfile을 만들었다면, 이제 이 이미지를 실제 프로덕션 환경에 배포할 때 몇 가지 추가적으로 고려해야 할 사항들이 있어요.

### 0️⃣ CI/CD 파이프라인 통합

Docker 이미지를 빌드하고 레지스트리에 푸시하는 과정을 CI/CD 파이프라인에 통합하는 것이 좋아요.<br />
GitHub Actions, GitLab CI/CD, Jenkins 등 다양한 도구를 활용하여 코드 푸시 시 자동으로 이미지를 빌드하고 배포할 수 있도록 설정해 보세요.<br />
이는 배포 프로세스의 안정성과 속도를 크게 향상시켜 줘요.

### 1️⃣ 환경 변수 관리

프로덕션 환경에서는 민감한 정보(API 키, 데이터베이스 연결 정보 등)를 환경 변수로 관리해야 해요.<br />
Docker 컨테이너를 실행할 때 `docker run -e KEY=VALUE` 옵션을 사용하거나, Kubernetes와 같은 오케스트레이션 도구의 Secret 기능을 활용하여 안전하게 환경 변수를 주입할 수 있어요.<br />
`Dockerfile` 내에서는 `ENV` 명령으로 기본값을 설정할 수 있지만, 프로덕션 환경에서는 런타임에 오버라이드하는 것이 일반적이에요.

### 2️⃣ 모니터링 및 로깅

배포된 Next.js 애플리케이션이 프로덕션 환경에서 제대로 작동하는지 지속적으로 모니터링하고 로그를 수집하는 것이 중요해요.<br />
Docker의 로깅 드라이버를 설정하거나, Prometheus, Grafana, ELK Stack(Elasticsearch, Logstash, Kibana)과 같은 도구를 활용하여 시스템 상태와 애플리케이션 로그를 체계적으로 관리해 주세요.<br />
Health Check 설정은 컨테이너의 기본적인 상태를 파악하는 데 도움이 되지만, 보다 심층적인 모니터링 시스템을 구축하는 것이 좋아요.

## 📝 정리하며

오늘은 Next.js 애플리케이션을 Docker로 배포할 때 프로덕션 환경에 최적화된 Dockerfile을 작성하는 방법에 대해 자세히 알아보았어요.<br />
멀티스테이지 빌드를 통해 이미지 크기를 줄이고, 효율적인 레이어 캐싱으로 빌드 속도를 높이며, 비루트 사용자 실행을 통해 보안을 강화하는 것이 핵심이었어요.

### 0️⃣ 핵심 요약

*   **멀티스테이지 빌드**: `builder`와 `runner` 스테이지를 분리하여 최종 이미지 크기를 최소화해요.<br />
*   **레이어 캐싱**: `package.json`과 lock 파일을 먼저 복사하여 의존성 설치 단계를 캐시하고, 빌드 속도를 높여요.<br />
*   **비루트 사용자**: `adduser`, `addgroup` 명령으로 `nextjs` 사용자를 생성하고, 이 사용자로 애플리케이션을 실행하여 보안을 강화해요.<br />
*   **`output: 'standalone'`**: `next.config.js`에 이 옵션을 추가하여 Next.js가 최소한의 프로덕션 빌드 결과물을 생성하도록 해요.<br />
*   **`.dockerignore`**: 불필요한 파일과 디렉터리를 제외하여 이미지 크기를 줄이고 보안을 높여요.

### 1️⃣ 다음 단계

이제 여러분의 Next.js 프로젝트에 이 Dockerfile 최적화 전략을 적용해 볼 시간이에요.<br />
작성된 Dockerfile을 기반으로 이미지를 빌드하고, 로컬에서 실행해 보면서 최종 프로덕션 환경에서의 동작을 확인해 보세요.<br />
나아가 CI/CD 파이프라인에 통합하여 자동화된 배포 시스템을 구축한다면 더욱 효율적인 개발 워크플로우를 완성할 수 있을 거예요.
