## 🤔 React 전역 상태 관리, 왜 중요할까요?

### 0️⃣ "Prop Drilling" 문제
React는 컴포넌트 기반의 UI 라이브러리로, 데이터 흐름이 단방향으로 이루어져요. 부모 컴포넌트에서 자식 컴포넌트로 데이터를 전달할 때는 `props`를 사용하죠.<br />
하지만 애플리케이션이 복잡해지면서 여러 단계의 컴포넌트를 거쳐 데이터를 전달해야 하는 경우가 생기는데, 이를 "Prop Drilling"이라고 불러요. 이 과정에서 불필요한 `props`가 중간 컴포넌트들에 전달되고, 코드의 가독성과 유지보수성이 크게 떨어지게 돼요.

```tsx showLineNumbers
// Prop Drilling 예시
function App() {
  const user = { name: '블루', theme: 'dark' };
  return <Toolbar user={user} />;
}

function Toolbar({ user }) {
  return <Profile user={user} />;
}

function Profile({ user }) {
  return <UserAvatar user={user} />;
}

function UserAvatar({ user }) {
  return <div>안녕하세요, {user.name}님! ({user.theme} 모드)</div>;
}
```

위 코드처럼 `user` 객체가 `Toolbar`를 거쳐 `Profile`, 그리고 최종적으로 `UserAvatar` 컴포넌트까지 전달되는 과정이 바로 Prop Drilling이에요. `Toolbar`와 `Profile` 컴포넌트 자체는 `user` 정보를 사용하지 않지만, 다음 자식에게 전달하기 위해 `props`로 받고 있죠.<br />
이런 문제를 해결하기 위해 React에서는 `Context API`를 제공하고, 더 나아가 `Zustand`와 같은 외부 상태 관리 라이브러리들이 존재해요.

## ⚙️ React Context API 심층 분석

### 0️⃣ Context API란?
React Context API는 React 컴포넌트 트리 안에서 전역적인 데이터를 공유할 수 있도록 해주는 React 내장 기능이에요. `props`를 통해 명시적으로 전달하지 않고도, 컴포넌트들이 데이터를 구독할 수 있게 해주죠. 주로 테마, 사용자 인증 정보, 언어 설정 등 애플리케이션 전반에 걸쳐 필요한 데이터를 관리하는 데 사용돼요.

### 1️⃣ Context API의 장점과 단점

<Blockquote type="success">
**✨ 장점:**<br />
-   **React 내장 기능:** 별도의 라이브러리 설치 없이 바로 사용할 수 있어요.<br />
-   **간단한 전역 상태:** 복잡하지 않은 전역 상태 관리에 적합해요.<br />
-   **러닝 커브 낮음:** 기본적인 React 개념만으로 쉽게 이해하고 적용할 수 있어요.
</Blockquote>

<Blockquote type="warning">
**⚠️ 단점:**<br />
-   **불필요한 리렌더링:** Context의 `value`가 변경되면 해당 Context를 구독하는 모든 `Consumer` 컴포넌트가 리렌더링돼요. 이는 `React.memo`나 `useMemo`, `useCallback`으로 부분적으로 최적화할 수 있지만, 복잡한 상태에서는 한계가 있어요.<br />
-   **복잡한 상태 로직:** `useState`와 `useReducer`를 조합하여 복잡한 상태 로직을 구현해야 하므로 보일러플레이트 코드가 증가할 수 있어요.<br />
-   **분리된 관심사 관리 어려움:** 여러 개의 Context를 사용하게 되면 Provider 중첩이 발생하고, 상태 로직과 UI 로직이 분리되기 어려워질 수 있어요.
</Blockquote>

### 2️⃣ Context API 사용 예시: 테마 관리
간단한 다크/라이트 테마를 관리하는 Context API 예시를 살펴볼게요.

```tsx showLineNumbers {1-11, 15-18, 22-26}
// context/ThemeContext.tsx
import React, { createContext, useContext, useState, ReactNode } from 'react';

type Theme = 'light' | 'dark';
type ThemeContextType = {
  theme: Theme;
  toggleTheme: () => void;
};

const ThemeContext = createContext<ThemeContextType | undefined>(undefined);

export function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setTheme] = useState<Theme>('light');

  const toggleTheme = () => {
    setTheme((prevTheme) => (prevTheme === 'light' ? 'dark' : 'light'));
  };

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

export function useTheme() {
  const context = useContext(ThemeContext);
  if (context === undefined) {
    throw new Error('useTheme must be used within a ThemeProvider');
  }
  return context;
}

// App.tsx
import { ThemeProvider, useTheme } from './context/ThemeContext';

function ThemeToggle() {
  const { theme, toggleTheme } = useTheme();
  return (
    <button onClick={toggleTheme} style={{ background: theme === 'dark' ? '#333' : '#eee', color: theme === 'dark' ? '#eee' : '#333' }}>
      현재 테마: {theme === 'light' ? '밝은 모드' : '어두운 모드'}
    </button>
  );
}

function DisplayTheme() {
  const { theme } = useTheme();
  return <p>테마가 '{theme}'으로 설정되어 있어요.</p>;
}

export default function App() {
  return (
    <ThemeProvider>
      <h1>Context API 예시</h1>
      <ThemeToggle />
      <DisplayTheme />
    </ThemeProvider>
  );
}
```
위 예시에서는 `ThemeProvider`로 컴포넌트 트리를 감싸고, `useTheme` 커스텀 훅을 통해 `theme` 상태와 `toggleTheme` 함수에 접근하고 있어요. 이렇게 하면 Prop Drilling 없이 전역 테마 상태를 관리할 수 있죠.

## 🚀 Zustand 핵심 파헤치기

### 0️⃣ Zustand란?
Zustand는 작고 빠르며 확장 가능한 React 상태 관리 라이브러리예요. Redux와 같은 복잡한 상태 관리 라이브러리에 비해 훨씬 간결한 API를 제공하면서도, 불필요한 리렌더링을 효과적으로 제어할 수 있는 강력한 기능을 가지고 있어요. 특히 React `hooks`에 기반을 두고 있어 React 개발자에게 매우 친숙하게 느껴질 거예요.

### 1️⃣ Zustand의 장점과 단점

<Blockquote type="success">
**✨ 장점:**<br />
-   **가벼움 & 빠른 성능:** 번들 사이즈가 매우 작고, 불필요한 리렌더링을 최소화하여 높은 성능을 자랑해요.<br />
-   **간결한 API:** Redux와 같은 복잡한 보일러플레이트 없이 직관적으로 상태를 정의하고 사용할 수 있어요.<br />
-   **셀렉터를 통한 최적화:** 특정 상태 값만 선택(select)하여 구독할 수 있어, 선택된 값이 변경될 때만 컴포넌트가 리렌더링되도록 효과적으로 제어해요.<br />
-   **미들웨어 지원:** Redux DevTools, 비동기 처리 등 다양한 미들웨어를 쉽게 통합할 수 있어요.<br />
-   **TypeScript 친화적:** 타입 추론이 잘 작동하며, 강력한 타입 안전성을 제공해요.
</Blockquote>

<Blockquote type="warning">
**⚠️ 단점:**<br />
-   **외부 라이브러리 의존성:** React 내장 기능이 아니므로 별도 설치가 필요해요.<br />
-   **작은 설정 필요:** Context API보다는 상태 정의 및 스토어 생성에 약간의 설정이 필요해요.
</Blockquote>

### 2️⃣ Zustand 사용 예시: 카운터 관리
간단한 카운터 상태를 Zustand로 관리하는 예시를 살펴볼게요.

```tsx showLineNumbers {1-10, 14-17}
// store/useCounterStore.ts
import { create } from 'zustand';

interface CounterState {
  count: number;
  increment: () => void;
  decrement: () => void;
  reset: () => void;
}

export const useCounterStore = create<CounterState>((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
  reset: () => set({ count: 0 }),
}));

// App.tsx
import { useCounterStore } from './store/useCounterStore';

function CounterDisplay() {
  const count = useCounterStore((state) => state.count); // count 값만 구독
  console.log('CounterDisplay 리렌더링');
  return <p>현재 카운트: {count}</p>;
}

function CounterControls() {
  const { increment, decrement, reset } = useCounterStore(); // 함수만 구독
  console.log('CounterControls 리렌더링');
  return (
    <div>
      <button onClick={increment}>증가</button>
      <button onClick={decrement}>감소</button>
      <button onClick={reset}>초기화</button>
    </div>
  );
}

export default function App() {
  return (
    <div>
      <h1>Zustand 예시</h1>
      <CounterDisplay />
      <CounterControls />
    </div>
  );
}
```
위 예시에서 `useCounterStore` 훅을 사용할 때, `CounterDisplay` 컴포넌트는 `count` 값만 선택적으로 구독하고, `CounterControls` 컴포넌트는 액션 함수들만 구독해요.<br />
이는 `count` 값이 변경될 때 `CounterDisplay`만 리렌더링되고, `CounterControls`는 리렌더링되지 않도록 하여 불필요한 성능 저하를 방지해 줘요. 이것이 바로 Zustand의 강력한 최적화 기능 중 하나예요.

## ⚖️ Context API vs Zustand: 언제 무엇을 써야 할까요?

두 도구 모두 React 애플리케이션의 전역 상태 관리에 사용되지만, 각각의 특성과 장단점이 명확하므로 프로젝트의 요구사항에 맞춰 현명하게 선택해야 해요.

### 0️⃣ Context API를 추천하는 경우
<Blockquote type="info">
**🔑 핵심:** 애플리케이션 전반에 걸쳐 넓게 사용되지만, **업데이트 빈도가 낮고 상태 로직이 간단한** 경우에 적합해요.<br />
별도의 라이브러리 추가 없이 React 내장 기능만으로 해결하고 싶을 때 좋은 선택이죠.
</Blockquote>

**예시:**<br />
-   **테마 설정:** 다크/라이트 모드 전환과 같이 애플리케이션 전체에 영향을 주지만, 변경 빈도가 낮은 상태.<br />
-   **사용자 인증 정보:** 로그인 여부, 사용자 ID 등 한 번 설정되면 자주 변경되지 않는 정보.<br />
-   **언어 설정:** 다국어 지원을 위한 현재 언어 정보.<br />
-   **작은 규모의 프로젝트:** 전역 상태의 수가 적고 복잡성이 낮은 프로젝트.

### 1️⃣ Zustand를 추천하는 경우
<Blockquote type="info">
**🔑 핵심:** **업데이트 빈도가 높고, 복잡한 비즈니스 로직을 포함하며, 성능 최적화가 중요한** 전역 상태 관리에 탁월해요.<br />
Redux와 같은 무거운 라이브러리 대신 가볍고 효율적인 대안을 찾을 때 이상적이죠.
</Blockquote>

**예시:**<br />
-   **장바구니 상태:** 상품 추가/삭제, 수량 변경 등 빈번하게 업데이트되는 상태.<br />
-   **폼 데이터 관리:** 여러 단계에 걸친 복잡한 폼의 유효성 검사 및 데이터 저장.<br />
-   **실시간 데이터 처리:** 웹소켓을 통한 채팅 메시지, 알림 등 지속적으로 업데이트되는 데이터.<br />
-   **글로벌 검색 필터:** 여러 컴포넌트에서 공유되고 상호작용하는 검색 조건.<br />
-   **중대형 규모의 프로젝트:** 전역 상태의 수가 많고, 상태 간 상호작용이 복잡하며, 성능이 중요한 프로젝트.

### 2️⃣ 함께 사용하는 시너지 효과
꼭 둘 중 하나만 선택할 필요는 없어요. 두 도구를 함께 사용하여 각자의 장점을 활용하는 것도 좋은 전략이에요.<br />
예를 들어, Context API로 애플리케이션의 기본적인 테마나 정적인 사용자 정보를 관리하고, Zustand로는 장바구니나 실시간 알림처럼 동적이고 자주 변하는 상태를 관리할 수 있어요. 이렇게 하면 Context API의 리렌더링 문제를 피하면서도, Zustand의 강력한 성능 최적화 기능을 활용할 수 있죠.

## 📝 정리하며

React 애플리케이션의 전역 상태 관리는 개발 규모와 요구사항에 따라 다양한 접근 방식이 필요해요. Context API는 React 내장 기능으로 간단한 전역 상태 관리에 적합하지만, 복잡한 상태와 잦은 업데이트에는 성능 문제가 발생할 수 있어요.<br />
반면 Zustand는 가볍고 효율적인 외부 라이브러리로, 셀렉터를 통한 최적화와 간결한 API로 복잡하고 동적인 상태 관리에 매우 효과적이에요.

어떤 도구를 선택하든 가장 중요한 것은 프로젝트의 특성과 팀의 숙련도를 고려하는 것이에요. 이 글이 여러분의 React 전역 상태 관리 전략 수립에 실질적인 도움이 되었기를 바라요.<br />
