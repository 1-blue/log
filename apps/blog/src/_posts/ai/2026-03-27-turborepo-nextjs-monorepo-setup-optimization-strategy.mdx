안녕하세요, 10년 이상 개발 현장에서 풀스택 개발자로 일해온 블루예요. 현대 웹 개발 환경은 점점 복잡해지고, 여러 개의 프런트엔드와 백엔드 서비스, 그리고 공유 라이브러리들을 효율적으로 관리하는 것이 중요해지고 있어요.<br />
이러한 요구사항에 맞춰 '모노레포(Monorepo)' 아키텍처는 많은 팀에서 선택하는 대안이 되고 있답니다. 특히 Next.js 프로젝트가 많아질수록 모노레포의 장점은 더욱 빛을 발하죠.<br />
오늘은 Next.js 프로젝트를 Turborepo와 함께 모노레포로 구축하고, 이를 통해 개발 효율성과 성능을 최적화하는 실질적인 방법을 자세히 알려드릴게요.

## 🤔 왜 모노레포와 Turborepo인가요?

### 0️⃣ 모노레포의 부상과 장점

모노레포는 여러 개의 프로젝트 코드를 하나의 Git 저장소에서 관리하는 방식을 말해요. 처음에는 다소 생소할 수 있지만, 대규모 애플리케이션이나 여러 관련 서비스를 개발할 때 다음과 같은 강력한 장점들을 제공한답니다.<br />

*   **코드 공유의 용이성**: 공통 UI 컴포넌트, 유틸리티 함수, 타입 정의 등을 쉽게 공유하고 재사용할 수 있어요. 이는 중복 코드를 줄이고 일관성을 유지하는 데 큰 도움이 되죠.<br />
*   **단일 버전 관리**: 모든 프로젝트가 동일한 버전의 라이브러리를 사용하도록 강제하거나, 적어도 어떤 버전이 어디에 사용되는지 한눈에 파악하기 쉬워요. 버전 불일치로 인한 문제를 줄일 수 있답니다.<br />
*   **통합된 개발 경험**: 모든 코드가 한곳에 있어 개발자가 다른 프로젝트의 코드를 쉽게 탐색하고 이해할 수 있어요. 또한, 통일된 빌드, 테스트, 린트 스크립트를 적용하여 개발 환경을 일관되게 유지할 수 있죠.<br />
*   **원자적(Atomic) 변경**: 여러 프로젝트에 걸쳐 변경이 필요한 경우, 하나의 커밋으로 모든 관련 변경사항을 함께 관리할 수 있어요. 이는 코드 리뷰를 단순화하고 배포 오류 가능성을 줄여준답니다.

### 1️⃣ 기존 멀티레포의 한계

모노레포의 장점은 기존 멀티레포(Multi-repo) 방식의 한계와 비교하면 더욱 명확해져요. 멀티레포는 각 프로젝트를 별도의 Git 저장소로 관리하는 방식인데, 다음과 같은 문제점들이 흔히 발생하곤 했어요.<br />

*   **코드 중복**: 여러 프로젝트에서 동일한 기능이나 UI 컴포넌트가 필요할 때, 이를 복사하거나 별도의 패키지로 만들어 관리해야 했어요. 패키지 의존성 관리도 복잡해지고요.<br />
*   **버전 불일치**: 공유 라이브러리의 새 버전이 나왔을 때, 모든 관련 프로젝트에서 수동으로 업데이트해야 했어요. 이 과정에서 버전 불일치가 발생하고, 예상치 못한 버그로 이어질 수 있었죠.<br />
*   **복잡한 CI/CD**: 각 레포마다 CI/CD 파이프라인을 구축해야 해서 관리 비용이 증가하고, 여러 레포에 걸친 통합 테스트가 어려웠어요.

### 2️⃣ Turborepo가 제공하는 가치

모노레포를 구축할 때, 수많은 프로젝트와 스크립트를 효율적으로 관리하려면 강력한 빌드 시스템이 필요해요. 이때 `Turborepo`가 등장한답니다.<br />
Turborepo는 Vercel에서 만든 고성능 빌드 시스템으로, 모노레포 환경에서 다음과 같은 핵심 가치를 제공하며 개발 생산성을 극대화해줘요.<br />

*   **증분 빌드(Incremental Builds)**: 변경된 부분만 빌드하고, 이전에 빌드했던 결과물을 캐싱하여 재사용해요. 이는 빌드 시간을 획기적으로 단축시켜 준답니다.<br />
*   **병렬 실행(Parallel Execution)**: 의존성이 없는 작업들을 동시에 실행하여 전체 빌드 시간을 줄여줘요. CPU 코어를 최대한 활용하는 거죠.<br />
*   **원격 캐싱(Remote Caching)**: 로컬 캐시뿐만 아니라 원격 캐시를 지원하여 팀원 간 또는 CI/CD 파이프라인 간 빌드 결과물을 공유할 수 있어요. 한 번 빌드된 결과는 어디서든 재사용된답니다.<br />
*   **간결한 설정**: `turbo.json` 파일 하나로 모든 빌드 파이프라인을 선언적으로 관리할 수 있어서 설정이 매우 간결해요.

## 🚀 Turborepo & Next.js 모노레포 구축 가이드

이제 Turborepo를 활용하여 Next.js 모노레포를 실제로 구축하는 과정을 단계별로 살펴볼게요. 여기서는 `pnpm`을 패키지 매니저로 사용할 거예요. `pnpm`은 워크스페이스(workspace) 기능을 지원하며, 모노레포 환경에서 특히 효율적인 의존성 관리를 제공한답니다.

### 0️⃣ 초기 프로젝트 설정

먼저, 모노레포의 루트 디렉토리를 생성하고 `pnpm` 워크스페이스를 설정해 줄게요.

```bash showLineNumbers
mkdir my-nextjs-monorepo
cd my-nextjs-monorepo
pnpm init
```

`pnpm init` 명령을 실행하면 `package.json` 파일이 생성돼요. 이제 이 파일에 `workspaces` 설정을 추가하여 모노레포 내의 앱(apps)과 패키지(packages)들을 관리할 수 있도록 해줄게요.

```json showLineNumbers {5-8}
// package.json (루트)
{
  "name": "my-nextjs-monorepo",
  "version": "1.0.0",
  "private": true, // 이 레포는 publish하지 않을 것이기 때문에 private: true로 설정해요.
  "scripts": {
    "build": "turbo run build",
    "dev": "turbo run dev",
    "lint": "turbo run lint",
    "test": "turbo run test"
  },
  "workspaces": [
    "apps/*",
    "packages/*"
  ],
  "devDependencies": {
    "turbo": "latest"
  }
}
```

`private: true`는 이 루트 프로젝트가 npm에 배포되지 않도록 하는 설정이에요.<br />
`workspaces` 필드는 `apps/` 디렉토리와 `packages/` 디렉토리 아래의 모든 서브디렉토리를 워크스페이스로 인식하도록 알려준답니다. 그리고 `turbo`를 개발 의존성으로 설치해 주세요.

```bash showLineNumbers
pnpm install
```

이제 `apps`와 `packages` 디렉토리를 생성해 줄게요.

```bash showLineNumbers
mkdir apps packages
```

### 1️⃣ 워크스페이스 구조 이해하기

일반적인 Turborepo 모노레포 구조는 다음과 같아요.

```
my-nextjs-monorepo/
├── apps/             # 개별 애플리케이션(Next.js 앱, Express API 등)이 위치해요.
│   ├── web/          # Next.js 웹 애플리케이션
│   └── docs/         # 또 다른 Next.js 문서 사이트
├── packages/         # 공유 라이브러리(UI 컴포넌트, 유틸리티, 설정 등)가 위치해요.
│   ├── ui/           # 공유 UI 컴포넌트 라이브러리
│   ├── config/       # 공유 ESLint/TypeScript 설정
│   └── utils/        # 공유 유틸리티 함수
├── package.json      # 루트 package.json (워크스페이스 설정)
├── pnpm-workspace.yaml # pnpm 워크스페이스 설정 파일 (선택 사항, package.json에 설정했다면 필요 없어요)
└── turbo.json        # Turborepo 설정 파일
```

### 2️⃣ 공유 컴포넌트/유틸리티 패키지 생성

먼저, `packages/ui`라는 공유 UI 컴포넌트 패키지를 만들어볼게요. 간단한 `Button` 컴포넌트를 포함시킬 거예요.

```bash showLineNumbers
mkdir packages/ui
cd packages/ui
pnpm init
```

`packages/ui/package.json` 파일을 다음과 같이 수정해 주세요.

```json showLineNumbers {2,5}
// packages/ui/package.json
{
  "name": "@repo/ui", // @repo/ 접두사는 모노레포 내 패키지임을 나타내는 일반적인 컨벤션이에요.
  "version": "0.0.0",
  "main": "./src/index.tsx", // 메인 진입점
  "types": "./src/index.tsx", // 타입 진입점
  "license": "MIT",
  "scripts": {
    "lint": "eslint .",
    "generate:component": "turbo gen react-component"
  },
  "devDependencies": {
    "@repo/eslint-config": "workspace:*", // 워크스페이스 내의 다른 패키지를 참조해요.
    "@types/react": "^18.2.37",
    "eslint": "^8.53.0",
    "react": "^18.2.0",
    "typescript": "^5.2.2"
  },
  "peerDependencies": {
    "react": "^18.2.0"
  }
}
```

`@repo/ui`는 이 패키지의 이름이에요. `workspace:*`는 `pnpm`에게 워크스페이스 내의 `@repo/eslint-config` 패키지를 사용하라고 알려준답니다.<br />
이제 간단한 `Button` 컴포넌트를 만들어볼게요. `packages/ui/src/Button.tsx` 파일을 생성해 주세요.

```tsx showLineNumbers
// packages/ui/src/Button.tsx
import * as React from "react";

interface ButtonProps {
  children: React.ReactNode;
  onClick?: () => void;
  className?: string;
}

export function Button({ children, onClick, className }: ButtonProps) {
  return (
    <button
      onClick={onClick}
      className={`px-4 py-2 rounded-md font-semibold ${className || ''}`}
      style={{
        backgroundColor: '#0070f3',
        color: 'white',
        border: 'none',
        cursor: 'pointer'
      }}
    >
      {children}
    </button>
  );
}
```

그리고 `packages/ui/src/index.tsx`에서 이 컴포넌트를 내보내도록 설정해 주세요.

```tsx showLineNumbers
// packages/ui/src/index.tsx
export * from "./Button";
```

마지막으로, 공유 `tsconfig.json`과 `eslint-config`를 `packages/config`에 만들어볼게요.

```bash showLineNumbers
mkdir packages/config
cd packages/config
pnpm init
```

`packages/config/package.json`은 다음과 같이 설정해요.

```json showLineNumbers
// packages/config/package.json
{
  "name": "@repo/config",
  "version": "0.0.0",
  "private": true,
  "main": "index.js",
  "license": "MIT",
  "publishConfig": {
    "access": "public"
  }
}
```

그리고 `packages/config/eslint-config/library.js` 파일을 생성하고 다음 내용을 추가해 주세요. (실제 ESLint 설정은 프로젝트 요구사항에 따라 달라질 수 있어요.)

```javascript showLineNumbers
// packages/config/eslint-config/library.js
module.exports = {
  extends: [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended",
    "prettier"
  ],
  plugins: ["@typescript-eslint"],
  parser: "@typescript-eslint/parser",
  parserOptions: {
    ecmaVersion: "latest",
    sourceType: "module"
  },
  env: {
    node: true,
    browser: true
  },
  rules: {
    // 여기에 프로젝트에 맞는 규칙을 추가해주세요.
  }
};
```

### 3️⃣ Next.js 앱에서 공유 패키지 사용하기

이제 `apps/web`이라는 Next.js 애플리케이션을 생성하고, 위에서 만든 공유 `ui` 패키지를 사용해볼게요. Turborepo는 Next.js 앱 템플릿을 제공하니, 이를 활용하는 것이 편리해요.

```bash showLineNumbers
pnpm dlx create-next-app@latest apps/web --ts --eslint --tailwind --app
cd apps/web
pnpm install @repo/ui @repo/config
```

`apps/web/package.json`을 열어보면 `@repo/ui`와 `@repo/config`가 의존성으로 추가되어 있을 거예요. `pnpm`이 자동으로 워크스페이스 내의 패키지를 링크해 준답니다.<br />

Next.js 앱에서 워크스페이스 내의 패키지를 사용하려면 `next.config.js`에 `transpilePackages` 설정을 추가해야 할 때가 많아요. 특히 TypeScript나 Tailwind CSS를 사용하는 공유 컴포넌트의 경우 필수적이에요.

```javascript showLineNumbers {3-5}
// apps/web/next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  transpilePackages: ["@repo/ui", "@repo/config"],
};

module.exports = nextConfig;
```

이제 `apps/web/app/page.tsx`에서 공유 `Button` 컴포넌트를 사용해볼게요.

```tsx showLineNumbers {2,7-10}
// apps/web/app/page.tsx
import { Button } from "@repo/ui";

export default function HomePage() {
  return (
    <main className="flex min-h-screen flex-col items-center justify-between p-24">
      <h1>Welcome to Next.js Monorepo!</h1>
      <Button onClick={() => alert("Hello from shared UI!")}>
        Click Me (Shared UI)
      </Button>
    </main>
  );
}
```

이제 루트 디렉토리로 돌아와서 개발 서버를 실행해 보세요.

```bash showLineNumbers
cd ../..
pnpm run dev
```

`http://localhost:3000`으로 접속하면 Next.js 앱과 공유 `Button` 컴포넌트가 잘 작동하는 것을 확인할 수 있을 거예요.

## ⚙️ Turborepo 핵심 기능 활용 및 최적화

Turborepo의 진정한 힘은 `turbo.json` 설정을 통해 발휘된답니다. 이 파일을 사용하여 빌드 파이프라인을 정의하고, 캐싱 전략을 최적화할 수 있어요.

### 0️⃣ 캐싱과 병렬 실행으로 빌드 속도 향상

루트 디렉토리에 `turbo.json` 파일을 생성하고 다음과 같이 설정해 주세요.

```json showLineNumbers
// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "pipeline": {
    "build": {
      "dependsOn": ["^build"], // 의존성 관계에 있는 패키지의 빌드가 먼저 완료되어야 함을 의미해요.
      "outputs": [".next/**", "dist/**"]
    },
    "lint": {
      "dependsOn": ["^lint"]
    },
    "dev": {
      "cache": false, // 개발 서버는 캐시하지 않아요.
      "persistent": true // 프로세스가 계속 실행되도록 해요.
    },
    "test": {
      "dependsOn": ["^build"],
      "outputs": ["coverage/**"]
    }
  }
}
```

*   `pipeline`: Turborepo가 실행할 작업(task)들을 정의해요.<br />
*   `build`: 각 워크스페이스의 `build` 스크립트에 대한 설정이에요.<br />
    *   `dependsOn`: `^build`는 현재 워크스페이스의 의존성(dependency)에 해당하는 워크스페이스의 `build` 스크립트가 먼저 실행되어야 함을 의미해요. 예를 들어, `apps/web`을 빌드할 때 `packages/ui`가 먼저 빌드되도록 하는 거죠.<br />
    *   `outputs`: 이 작업의 결과물(캐시할 파일)을 지정해요. `.next/**`는 Next.js 빌드 결과물이고, `dist/**`는 다른 라이브러리의 빌드 결과물일 수 있어요.<br />
*   `dev`: 개발 서버는 지속적으로 실행되어야 하므로 `cache: false`와 `persistent: true`로 설정해요.

이 설정을 통해 Turborepo는 변경된 파일만 감지하고, 이전에 빌드된 결과는 캐시에서 가져와 재사용해요. 여러 프로젝트를 동시에 빌드할 때도 의존성 트리를 분석하여 병렬로 실행하므로, 빌드 시간이 놀라울 정도로 단축된답니다.

<Blockquote type="tip">
**원격 캐싱(Remote Caching) 활성화하기**<br />
Turborepo는 Vercel 계정과 연동하여 원격 캐싱을 지원해요. `turbo login` 명령으로 로그인하고 `turbo link`로 레포지토리를 연결하면, 팀원 간 또는 CI/CD 환경에서 빌드 캐시를 공유할 수 있어요. 이는 특히 대규모 팀이나 복잡한 CI/CD 파이프라인에서 빌드 시간을 획기적으로 줄여주는 강력한 기능이랍니다.
</Blockquote>

### 1️⃣ 파이프라인 설정으로 작업 자동화

`turbo.json`의 `pipeline`은 `build` 외에도 `lint`, `test`, `dev` 등 다양한 작업을 정의할 수 있어요. 예를 들어, `lint` 스크립트를 정의하면 모든 워크스페이스에서 일관된 린팅을 실행할 수 있답니다.

루트 `package.json`의 스크립트를 다시 한번 확인해볼게요.

```json showLineNumbers {3-6}
// package.json (루트)
{
  "scripts": {
    "build": "turbo run build",
    "dev": "turbo run dev",
    "lint": "turbo run lint",
    "test": "turbo run test"
  },
  // ...
}
```

이제 루트에서 `pnpm run lint`를 실행하면, Turborepo는 `turbo.json`에 정의된 `lint` 파이프라인 설정을 기반으로 모든 워크스페이스의 `lint` 스크립트를 효율적으로 실행해요. 만약 특정 워크스페이스에 `lint` 스크립트가 없다면, 해당 워크스페이스는 건너뛰게 된답니다.

### 2️⃣ CI/CD 환경에서의 Turborepo

CI/CD 파이프라인에서 Turborepo는 빛을 발해요. 원격 캐싱과 증분 빌드 덕분에 CI/CD 시간을 크게 단축할 수 있답니다.<br />
예를 들어, GitHub Actions에서 다음과 같이 Turborepo를 활용할 수 있어요.

```yaml showLineNumbers
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v3
        with:
          version: 8

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Cache Turborepo build outputs
        uses: actions/cache@v4
        with:
          path: .turbo
          key: ${{ runner.os }}-turbo-${{ github.sha }}
          restore-keys: |
            ${{ runner.os }}-turbo-

      - name: Build all projects
        run: pnpm run build

      - name: Lint all projects
        run: pnpm run lint

      - name: Test all projects
        run: pnpm run test
```

이 워크플로우는 Turborepo의 캐싱 디렉토리(`.turbo`)를 GitHub Actions 캐시에 저장하고 복원하여, 이전 빌드의 결과물을 재사용할 수 있도록 한답니다.<br />
이를 통해 변경되지 않은 프로젝트는 다시 빌드되거나 테스트되지 않으므로, CI/CD 실행 시간을 크게 절약할 수 있어요.

Turborepo와 함께 Next.js 모노레포를 구축하는 것은 초기 설정에 약간의 노력이 필요하지만, 장기적으로 개발 생산성과 코드 관리 효율성을 크게 향상시켜주는 강력한 전략이에요.<br />
공유 코드베이스의 이점, Turborepo의 고성능 빌드 시스템, 그리고 CI/CD 최적화 가능성까지, 모노레포는 현대 웹 개발 팀에 필수적인 아키텍처라고 생각해요.

### 0️⃣ 핵심 요약

*   **모노레포**: 코드 공유, 일관성 유지, 단일 버전 관리, 원자적 변경의 이점을 제공해요.<br />
*   **Turborepo**: 증분 빌드, 병렬 실행, 원격 캐싱을 통해 모노레포의 빌드 속도를 획기적으로 개선해요.<br />
*   **구축**: `pnpm` 워크스페이스와 `turbo.json` 설정을 통해 Next.js 앱과 공유 패키지를 효율적으로 관리할 수 있어요.<br />
*   **최적화**: `turbo.json`의 `pipeline` 설정과 CI/CD에서의 캐싱 활용으로 개발 및 배포 시간을 단축할 수 있답니다.
