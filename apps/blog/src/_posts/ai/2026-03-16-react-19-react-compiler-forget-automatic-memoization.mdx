## 🤔 문제/배경

### 0️⃣ 왜 이 주제를 다루는가

프론트엔드 개발에서 React는 가장 널리 사용되는 라이브러리 중 하나예요.<br />
하지만 React 애플리케이션의 성능을 최적화하는 것은 생각보다 까다로운 일이죠.<br />
특히 불필요한 리렌더링은 사용자 경험을 저하시키고 애플리케이션의 반응성을 떨어뜨리는 주범이에요.<br />

그동안 우리는 `useMemo`와 `useCallback` 같은 훅을 사용하여 수동으로 메모이제이션을 적용해왔어요.<br />
하지만 이 훅들은 올바르게 사용하기 어렵고, 오히려 코드의 복잡성을 높이거나 잘못 사용하면 성능 저하를 일으키기도 했어요.<br />

React 19에 도입될 React Compiler (코드명 Forget)는 이러한 고민을 해결해 줄 혁신적인 도구로 기대를 모으고 있어요.<br />
이 컴파일러는 개발자가 수동으로 메모이제이션을 신경 쓰지 않아도 자동으로 불필요한 리렌더링을 방지해준다고 해요.<br />
이제 `useMemo`와 `useCallback`의 복잡함에서 벗어나 더 직관적인 코드를 작성하면서도 뛰어난 성능을 얻을 수 있는 길이 열린 것이죠.<br />

이 포스팅에서는 React Compiler가 무엇인지, 어떻게 동작하는지, 그리고 우리 개발자들이 어떻게 이 기술을 활용하여 더 나은 React 애플리케이션을 만들 수 있는지 자세히 알아볼 거예요.<br />

### 1️⃣ 기존 방식의 한계

React 컴포넌트는 부모 컴포넌트가 리렌더링되거나, 자신의 `state` 또는 `props`가 변경될 때 리렌더링돼요.<br />
이때 `props`로 전달되는 객체나 함수가 비록 내용상으로는 같더라도 참조 값이 다르면 자식 컴포넌트는 불필요하게 리렌더링될 수 있어요.<br />

이러한 문제를 해결하기 위해 React는 `useMemo`, `useCallback`, `React.memo`와 같은 API를 제공해왔어요.<br />

*   **`useMemo`**: 값의 메모이제이션을 위해 사용해요. 복잡한 계산 결과나 객체 생성 비용을 줄일 때 유용하죠.<br />
*   **`useCallback`**: 함수의 메모이제이션을 위해 사용해요. 자식 컴포넌트에 콜백 함수를 `props`로 전달할 때 불필요한 리렌더링을 방지하는 데 쓰여요.<br />
*   **`React.memo`**: 컴포넌트 자체를 메모이제이션하여 `props`가 변경되지 않으면 리렌더링을 건너뛰게 해요.<br />

하지만 이 방식들은 몇 가지 한계를 가지고 있어요.<br />

*   **개발자의 부담**: 개발자가 직접 어떤 값을 메모이제이션할지, 어떤 의존성 배열을 사용할지 결정해야 해요. 이 과정에서 실수가 발생하기 쉽죠.<br />
*   **높은 학습 곡선**: `useMemo`와 `useCallback`의 정확한 동작 원리와 의존성 배열의 중요성을 이해하는 것이 초중급 개발자에게는 어려울 수 있어요.<br />
*   **코드의 복잡성 증가**: 메모이제이션을 위해 코드를 감싸는 형태로 작성해야 하므로 가독성이 떨어지고 코드가 길어지는 단점이 있어요.<br />
*   **잘못된 사용**: 의존성 배열을 잘못 지정하거나, 불필요한 곳에 메모이제이션을 적용하면 오히려 오버헤드가 발생하여 성능이 저하될 수도 있어요.<br />

이러한 한계점들 때문에 많은 개발자들이 React 애플리케이션의 성능 최적화에 어려움을 겪어왔어요. React Compiler는 이러한 수동적인 접근 방식의 대안으로 등장했어요.<br />

## ⚙️ 해결 방법

### 0️⃣ 핵심 아이디어

React Compiler (코드명 Forget)의 핵심 아이디어는 **컴파일러가 자동으로 React 컴포넌트의 메모이제이션을 처리**하는 것이에요.<br />
개발자가 `useMemo`나 `useCallback`을 명시적으로 작성하지 않아도, 컴파일러가 코드를 분석하여 변경되지 않은 `props`나 `state`를 사용하는 부분은 리렌더링을 건너뛰도록 최적화된 코드를 생성해주는 것이죠.<br />

이 컴파일러는 JavaScript 코드를 분석하여 **"순수 함수(Pure Function)"**의 개념을 React 컴포넌트에 적용하려고 해요.<br />
순수 함수는 동일한 입력에 대해 항상 동일한 출력을 반환하고, 외부 상태를 변경하지 않는 함수를 말해요.<br />
React 컴포넌트도 이러한 순수 함수처럼 동작한다면, 입력(`props`, `state`)이 변하지 않는 한 리렌더링할 필요가 없다는 원리이죠.<br />

React Compiler는 다음과 같은 방식으로 동작해요.<br />

1.  **코드 분석**: 컴파일러는 React 컴포넌트의 JavaScript/TypeScript 코드를 분석하여 각 표현식(expression)과 선언(declaration)이 언제 재평가되어야 하는지 파악해요.<br />
2.  **의존성 추론**: `useMemo`나 `useCallback`의 의존성 배열처럼, 컴파일러가 각 값이나 함수의 변경 여부를 판단하는 데 필요한 의존성을 자동으로 추론해요.<br />
3.  **자동 메모이제이션 적용**: 추론된 의존성을 바탕으로 컴파일러는 불필요한 리렌더링을 방지하기 위해 필요한 메모이제이션 로직을 컴파일 타임에 자동으로 삽입해요.<br />
    이는 마치 `useMemo`나 `useCallback`을 개발자가 직접 작성한 것과 유사한 효과를 내지만, 개발자는 원본 코드를 더 간결하게 유지할 수 있게 돼요.<br />

결과적으로, 개발자는 `useMemo`와 `useCallback`을 제거하고 마치 바닐라 JavaScript 코드를 작성하듯이 React 컴포넌트를 작성할 수 있게 돼요.<br />
컴파일러가 복잡한 성능 최적화 로직을 대신 처리해주기 때문에, 개발자는 비즈니스 로직에 더 집중할 수 있게 되는 것이죠.<br />

### 1️⃣ 적용 방법

React Compiler는 아직 개발 중이거나 초기 단계에 있는 기술로, 실제 프로덕션 환경에 바로 적용하기 위해서는 몇 가지 고려사항이 있어요.<br />

**1. React Compiler 설치 및 설정 (예상)**<br />

React Compiler는 Babel 플러그인이나 Next.js 같은 프레임워크에 통합되는 형태로 제공될 가능성이 높아요.<br />
현재 공식적으로 발표된 설치 방법은 없지만, 일반적으로 다음과 같은 형태로 사용될 것으로 예상돼요.<br />

```bash showLineNumbers
npm install @react/compiler # 또는 yarn add @react/compiler
```

`babel.config.js` 또는 `next.config.js`에 컴파일러를 추가하는 방식으로 설정할 수 있을 거예요.<br />

```javascript showLineNumbers
// babel.config.js (예상)
module.exports = {
  plugins: [
    '@babel/plugin-transform-react-jsx',
    ['@react/compiler', { /* options */ }]
  ]
};
```

Next.js의 경우, `next.config.js`에서 플래그를 통해 활성화하는 방식이 될 수 있어요.<br />

```javascript showLineNumbers
// next.config.js (예상)
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    reactCompiler: true, // 이런 옵션으로 활성화될 수 있어요.
  },
};

module.exports = nextConfig;
```

**2. 컴파일러의 제약 사항 및 "Forget" 규칙 이해**<br />

React Compiler는 모든 JavaScript 코드를 마법처럼 최적화하는 것은 아니에요.<br />
컴파일러가 올바르게 작동하려면 몇 가지 규칙을 지켜야 해요.<br />

*   **순수성 (Purity)**: 컴포넌트 내부에서 사이드 이펙트(Side Effect)를 일으키는 코드를 최소화해야 해요.<br />
    예를 들어, 렌더링 로직 내에서 전역 변수를 수정하거나, DOM을 직접 조작하는 등의 행위는 컴파일러가 최적화를 방해할 수 있어요.<br />
    사이드 이펙트는 `useEffect` 훅 내에서 처리하는 것이 좋아요.<br />
*   **불변성 (Immutability)**: 객체나 배열을 직접 수정하는 대신, 새로운 객체나 배열을 생성하여 변경된 상태를 나타내는 불변성 원칙을 지켜야 해요.<br />
    `state`를 업데이트할 때 `setState(prevState => ({ ...prevState, newProp: value }))`와 같이 사용하는 것이 좋은 예시예요.<br />
*   **안정적인 참조 (Stable References)**: 컴포넌트 외부에서 가져온 함수나 객체는 렌더링 간에 동일한 참조를 유지하는 것이 좋아요.<br />
    `useRef`나 `useCallback` (컴파일러 도입 이전)을 사용하여 안정적인 참조를 유지해왔던 이유이기도 하죠.<br />
    컴파일러는 이러한 안정적인 참조를 자동으로 추론하려고 노력하지만, 개발자가 명시적으로 불변성을 지키는 것이 중요해요.<br />

이러한 규칙들은 사실 React의 기존 모범 사례와 크게 다르지 않아요.<br />
React Compiler는 이러한 모범 사례를 따르는 코드를 더 효율적으로 최적화할 수 있도록 돕는 도구라고 이해할 수 있어요.<br />

## 🧪 예시

### 0️⃣ 코드/설정 예시

React Compiler가 적용되기 전과 후의 코드를 비교해볼게요.<br />
아래 `Counter` 컴포넌트는 `count` 상태와 `increment` 함수를 가지고 있고, 자식 컴포넌트인 `Display`에 `count`와 `increment` 함수를 `props`로 전달해요.<br />

**React Compiler 적용 전 (수동 메모이제이션)**<br />

`useCallback`과 `React.memo`를 사용하여 불필요한 리렌더링을 방지하고 있어요.<br />

```tsx showLineNumbers
import React, { useState, useCallback, memo } from 'react';

// 자식 컴포넌트: props가 변경될 때만 리렌더링되도록 memo로 감싸줍니다.
const Display = memo(({ count, onIncrement }: { count: number; onIncrement: () => void }) => {
  console.log('Display Component Rendered');
  return (
    <div>
      <p>현재 카운트: {count}</p>
      <button onClick={onIncrement}>증가</button>
    </div>
  );
});

// 부모 컴포넌트
function Counter() {
  const [count, setCount] = useState(0);
  const [text, setText] = useState('');

  // increment 함수는 count가 변경될 때만 새로 생성되도록 useCallback으로 감싸줍니다.
  const increment = useCallback(() => {
    setCount(prevCount => prevCount + 1);
  }, []); // 의존성 배열을 비워두어 처음 한 번만 생성되도록 합니다.

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    setText(e.target.value);
  };

  console.log('Counter Component Rendered');

  return (
    <div>
      <h1>카운터 앱</h1>
      <input type="text" value={text} onChange={handleChange} placeholder="텍스트 입력" />
      <p>입력된 텍스트: {text}</p>
      <Display count={count} onIncrement={increment} />
    </div>
  );
}

export default Counter;
```

위 코드에서 `text` 상태만 변경해도 `Counter` 컴포넌트는 리렌더링되지만, `increment` 함수는 `useCallback`으로 인해 재생성되지 않으므로 `Display` 컴포넌트는 리렌더링되지 않아요.<br />
하지만 `useCallback`과 `memo`를 직접 작성해야 하는 번거로움이 있었죠.<br />

**React Compiler 적용 후 (자동 메모이제이션)**<br />

React Compiler가 활성화되면 `useCallback`과 `memo`를 제거하고 더 간결하게 코드를 작성할 수 있어요.<br />
컴파일러가 자동으로 필요한 메모이제이션 로직을 삽입해줄 거예요.<br />

### 1️⃣ 적용 결과

React Compiler가 적용된 애플리케이션에서는 다음과 같은 이점을 기대할 수 있어요.<br />

*   **개발 생산성 향상**: `useMemo`, `useCallback` 같은 성능 최적화 관련 훅을 수동으로 관리할 필요가 없어지므로, 개발자는 비즈니스 로직 구현에 더 집중할 수 있어요.<br />
    코드의 양이 줄어들고 가독성이 향상되는 효과도 있죠.<br />
*   **성능 최적화의 자동화**: 컴파일러가 휴먼 에러 없이 일관된 방식으로 메모이제이션을 적용해주기 때문에, 수동으로 최적화할 때 발생할 수 있는 실수를 방지할 수 있어요.<br />
    이는 특히 복잡한 컴포넌트 트리에서 큰 이점으로 작용해요.<br />
*   **번들 사이즈 감소**: `useMemo`, `useCallback`과 같은 훅들을 제거함으로써, 최종 JavaScript 번들 사이즈가 미미하게나마 줄어들 수 있어요.<br />
    이는 초기 로딩 성능에 긍정적인 영향을 줄 수 있죠.<br />
*   **일관된 성능**: 개발자의 숙련도에 관계없이 컴파일러가 최적화를 담당하므로, 애플리케이션 전체적으로 일관된 렌더링 성능을 유지할 수 있게 돼요.<br />

물론, 컴파일러의 도입 초기에는 예상치 못한 동작이나 디버깅의 어려움이 있을 수 있어요.<br />
하지만 React 팀은 이러한 문제를 최소화하기 위해 노력하고 있으며, 장기적으로 React 개발의 패러다임을 바꿀 중요한 변화가 될 것으로 예상돼요.<br />

## 📝 정리

### 0️⃣ 핵심 요약

React 19에 도입될 React Compiler (코드명 Forget)는 React 애플리케이션의 렌더링 성능 최적화를 위한 혁신적인 접근 방식이에요.<br />

*   **자동 메모이제이션**: 개발자가 `useMemo`나 `useCallback`을 수동으로 작성하지 않아도, 컴파일러가 코드를 분석하여 자동으로 불필요한 리렌더링을 방지하는 로직을 삽입해요.<br />
*   **개발자 경험 개선**: 코드의 복잡성을 줄이고 가독성을 높여 개발자가 비즈니스 로직에 더 집중할 수 있도록 도와줘요.<br />
*   **성능 향상**: 휴먼 에러 없이 일관된 방식으로 최적화를 적용하여 애플리케이션의 전반적인 성능과 반응성을 향상시킬 수 있어요.<br />
*   **순수성과 불변성**: 컴파일러가 제대로 작동하려면 React의 모범 사례인 순수 함수와 불변성 원칙을 지키는 것이 중요해요.<br />

React Compiler는 아직 초기 단계에 있지만, 미래의 React 개발에서 성능 최적화의 부담을 크게 줄여줄 것으로 기대돼요.<br />

### 1️⃣ 다음 액션

React Compiler는 아직 활발히 개발 중인 기술이므로, 현재로서는 다음 액션으로 다음 사항들을 권장해요.<br />

1.  **공식 문서 주시**: React Compiler의 공식 발표 및 문서를 지속적으로 확인하여 최신 정보와 적용 방법을 숙지하세요.<br />
    특히 설치 및 설정 방법, 그리고 컴파일러가 요구하는 특정 코드 패턴이나 제약 사항을 파악하는 것이 중요해요.<br />
2.  **모범 사례 유지**: React Compiler가 요구하는 순수 함수와 불변성 원칙은 이미 좋은 React 코드 작성의 기본이에요.<br />
    현재 개발하는 프로젝트에서도 이러한 모범 사례를 꾸준히 적용하는 연습을 해보세요.<br />
    이는 컴파일러 도입 시 더 매끄러운 전환을 가능하게 할 거예요.<br />
3.  **실험적인 적용**: 개인 프로젝트나 내부 테스트 환경에서 React Compiler가 공개되면 미리 적용해보세요.<br />
    실제로 코드가 어떻게 변환되고, 어떤 성능 개선 효과를 가져오는지 경험해보는 것이 중요해요.<br />
    예상치 못한 문제나 개선점을 발견할 수도 있어요.<br />

React Compiler는 React 개발의 새로운 시대를 열 잠재력을 가지고 있어요.<br />
이 기술을 미리 이해하고 준비한다면, 더욱 효율적이고 성능 좋은 React 애플리케이션을 만들 수 있을 거예요.<br />
