## 🤔 Next.js App Router와 데이터 캐싱의 중요성

Next.js의 App Router는 Server Components를 도입하면서 데이터 패칭 방식에 큰 변화를 가져왔어요.<br />
이제 데이터는 서버에서 직접 가져와 렌더링되므로, 효율적인 캐싱 전략이 애플리케이션의 성능을 좌우하는 핵심 요소가 되었답니다.

### 0️⃣ Server Components 시대의 데이터 관리

Server Components는 클라이언트 측 JavaScript 번들을 줄이고 초기 로딩 속도를 향상하는 데 큰 이점을 제공해요.<br />
이들은 서버에서 데이터를 가져오고 HTML을 생성한 후 클라이언트로 보내주는데요, 이때 데이터가 캐시되지 않으면 매 요청마다 동일한 데이터를 다시 가져와야 하므로 비효율적일 수 있어요.<br />
따라서 Server Components 환경에서는 데이터 캐싱이 더욱 중요해져요.

<Blockquote type="info">
Next.js는 `fetch` API를 확장하여 요청을 자동으로 캐싱하고 재검증하는 강력한 메커니즘을 제공해요. 이를 통해 개발자는 복잡한 캐싱 로직을 직접 구현할 필요 없이 선언적으로 캐싱 전략을 관리할 수 있답니다.
</Blockquote>

### 1️⃣ `fetch` API의 강력함

여러분은 이미 브라우저에서 `fetch` API를 사용하여 데이터를 가져오는 것에 익숙하실 거예요.<br />
Next.js는 이 `fetch` API를 한 단계 더 발전시켜, 자체적인 캐싱 레이어와 재검증(revalidation) 기능을 추가했어요.<br />
기존 `fetch`는 단순한 HTTP 요청을 보내지만, Next.js의 `fetch`는 요청 결과를 서버에 캐시하고, 특정 조건에 따라 이 캐시를 재검증하여 항상 최신 데이터를 유지할 수 있게 해줘요.

## ⚙️ Next.js 15의 `fetch` 캐싱 심층 분석

Next.js 15에서는 `fetch` 캐싱과 재검증 옵션이 더욱 안정화되고 강력해졌어요.<br />
이제 이 핵심 기능들을 어떻게 활용하는지 자세히 알아볼까요?

### 0️⃣ `fetch` 캐싱의 기본 동작과 설정

Next.js에서 `fetch`를 사용할 때, 기본적으로 데이터는 서버에 캐시돼요.<br />
이 캐싱 동작은 `fetch` 옵션을 통해 세밀하게 제어할 수 있답니다.

*   **`cache: 'force-cache'` (기본값):**<br />
    이 옵션은 Next.js `fetch`의 기본 동작으로, 데이터를 캐시하고 가능한 경우 캐시된 데이터를 사용해요.<br />
    만약 캐시된 데이터가 없다면 서버에서 데이터를 가져와 캐시한답니다.<br />
    대부분의 정적 데이터나 자주 변경되지 않는 데이터를 가져올 때 유용해요.

*   **`cache: 'no-store'`:**<br />
    이 옵션을 사용하면 `fetch` 요청의 결과가 캐시되지 않아요.<br />
    매 요청마다 항상 최신 데이터를 서버에서 가져오게 되죠.<br />
    사용자별로 실시간으로 변하는 데이터나 민감한 정보를 가져올 때 사용해요.

*   **`revalidate: number`:**<br />
    이 옵션은 캐시된 데이터의 유효 기간을 초 단위로 설정해요.<br />
    설정된 시간(예: 60초)이 지나면 다음 요청 시 백그라운드에서 데이터를 재검증(revalidate)하고, 새로운 데이터를 가져와 캐시를 업데이트한답니다.<br />
    이것은 ISR(Incremental Static Regeneration)과 유사한 동작을 개별 `fetch` 요청에 적용하는 것이라고 생각하시면 돼요.

```tsx showLineNumbers
async function getPost(id: string) {
  const res = await fetch(`https://api.example.com/posts/${id}`, {
    // 캐시를 60초 동안 유효하게 유지하고, 그 이후에는 백그라운드에서 재검증해요.
    next: { revalidate: 60 }, 
  });

  if (!res.ok) {
    throw new Error('Failed to fetch data');
  }

  return res.json();
}

async function getRealtimeData(userId: string) {
  const res = await fetch(`https://api.example.com/users/${userId}/dashboard`, {
    // 이 데이터는 캐시하지 않고, 매 요청마다 최신 데이터를 가져와요.
    cache: 'no-store',
  });

  if (!res.ok) {
    throw new Error('Failed to fetch data');
  }

  return res.json();
}
```

### 1️⃣ `revalidate` 옵션의 활용

`revalidate` 옵션은 `fetch` 요청별로 설정할 수도 있고, 라우트 세그먼트 전체에 적용할 수도 있어요.<br />
이 두 가지 방식의 우선순위와 상호작용을 이해하는 것이 중요해요.

*   **`fetch` 단위의 재검증 (`next: { revalidate: number }`):**<br />
    위 코드 예시처럼 특정 `fetch` 호출에만 `revalidate` 옵션을 직접 적용하는 방식이에요.<br />
    이는 해당 `fetch` 요청의 캐시 유효 기간을 개별적으로 제어할 때 유용하죠.

*   **라우트 세그먼트 단위의 재검증 (`export const revalidate = number | 'force-cache' | 'no-store'`):**<br />
    App Router에서는 특정 라우트 세그먼트(예: `app/blog/[slug]/page.tsx`) 전체에 대한 캐싱 정책을 `export const revalidate` 변수를 통해 설정할 수 있어요.<br />
    이 설정은 해당 라우트 세그먼트 내의 모든 `fetch` 요청에 기본적으로 적용된답니다.

```tsx showLineNumbers
// app/products/[id]/page.tsx

// 이 라우트 세그먼트 내의 모든 fetch 요청은 3600초(1시간)마다 재검증돼요.
export const revalidate = 3600;

async function getProduct(id: string) {
  // fetch 요청에 개별 revalidate 옵션이 없으므로, 위에서 설정한 3600초가 적용돼요.
  const res = await fetch(`https://api.example.com/products/${id}`);
  return res.json();
}

async function getProductReviews(id: string) {
  const res = await fetch(`https://api.example.com/products/${id}/reviews`, {
    // 이 fetch 요청은 개별적으로 revalidate: 60초를 설정했어요.
    // 따라서 라우트 세그먼트의 revalidate 값(3600초)보다 이 값이 우선 적용된답니다.
    next: { revalidate: 60 }, 
  });
  return res.json();
}

export default async function ProductPage({ params }: { params: { id: string } }) {
  const product = await getProduct(params.id);
  const reviews = await getProductReviews(params.id);

  return (
    <div>
      <h1>{product.name}</h1>
      {/* ... */}
      <h2>Reviews</h2>
      {/* ... */}
    </div>
  );
}
```

<Blockquote type="tip">
**우선순위:** `fetch` 요청에 직접 설정한 `cache` 또는 `revalidate` 옵션이 라우트 세그먼트에 설정된 `export const revalidate` 값보다 항상 우선해요.<br />
따라서 세밀한 제어가 필요할 때는 개별 `fetch` 옵션을, 전체 라우트의 기본 정책을 설정할 때는 `export const revalidate`를 사용하시면 된답니다.
</Blockquote>

### 2️⃣ `noStore()` 함수

Next.js 15에서는 `next/cache` 모듈에서 제공하는 `noStore()` 함수를 사용하여 현재 실행 중인 요청을 캐시하지 않도록 강제할 수 있어요.<br />
이는 특히 Server Components에서 동적인 데이터를 처리할 때 유용하답니다.<br />
`noStore()`를 호출하면 해당 Server Component의 렌더링 과정에서 발생하는 모든 `fetch` 요청이 `cache: 'no-store'`로 동작하도록 강제돼요.

```tsx showLineNumbers
import { noStore } from 'next/cache';

async function getUserProfile(userId: string) {
  noStore(); // 이 함수가 호출된 시점부터 해당 요청은 캐시되지 않아요.

  const res = await fetch(`https://api.example.com/users/${userId}/profile`);
  if (!res.ok) {
    throw new Error('Failed to fetch profile');
  }
  return res.json();
}

export default async function ProfilePage({ params }: { params: { userId: string } }) {
  const userProfile = await getUserProfile(params.userId);

  return (
    <div>
      <h1>{userProfile.name}'s Profile</h1>
      {/* ... */}
    </div>
  );
}
```

<Blockquote type="info">
`noStore()`는 Server Component나 Server Action 내부에서만 호출할 수 있어요.<br />
클라이언트 컴포넌트나 API Route Handler에서는 사용할 수 없으니 주의해 주세요.
</Blockquote>

## 🧪 실전 Next.js 15 캐싱 전략 예시

이제 실제 시나리오에서 Next.js 15의 캐싱 전략을 어떻게 적용할 수 있는지 구체적인 예시를 통해 알아볼까요?

### 0️⃣ 정적 데이터와 동적 데이터 혼합 처리

블로그 웹사이트를 개발한다고 가정해 볼게요.<br />
블로그 포스트 목록은 자주 변경되지 않지만, 사용자별 알림이나 실시간 인기 글 목록은 자주 변경될 수 있어요.<br />
이때 적절한 캐싱 전략을 적용하여 성능과 데이터 신선도를 동시에 확보할 수 있답니다.

```tsx showLineNumbers
// app/blog/page.tsx
import { noStore } from 'next/cache';

interface Post {
  id: string;
  title: string;
  content: string;
}

interface Notification {
  id: string;
  message: string;
}

// 블로그 포스트 목록은 1시간(3600초)마다 재검증되도록 설정해요.
async function getPosts() {
  const res = await fetch('https://api.example.com/blog/posts', {
    next: { revalidate: 3600 }, 
  });
  if (!res.ok) throw new Error('Failed to fetch posts');
  return res.json();
}

// 사용자 알림은 실시간으로 보여줘야 하므로 캐싱하지 않아요.
async function getUserNotifications(userId: string): Promise<Notification[]> {
  noStore(); // 이 함수 내의 fetch는 캐시되지 않아요.
  const res = await fetch(`https://api.example.com/users/${userId}/notifications`);
  if (!res.ok) throw new Error('Failed to fetch notifications');
  return res.json();
}

export default async function BlogPage() {
  const posts: Post[] = await getPosts();
  const userId = 'user123'; // 실제로는 인증된 사용자 ID를 사용하겠죠?
  const notifications: Notification[] = await getUserNotifications(userId);

  return (
    <div>
      <h1>블로그 포스트</h1>
      <ul>
        {posts.map(post => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>

      <h2>내 알림</h2>
      {notifications.length > 0 ? (
        <ul>
          {notifications.map(notif => (
            <li key={notif.id}>{notif.message}</li>
          ))}
        </ul>
      ) : (
        <p>새로운 알림이 없어요.</p>
      )}
    </div>
  );
}
```

위 예시에서는 블로그 포스트 목록은 `revalidate: 3600`으로 1시간마다 재검증되도록 설정하여 성능을 확보하고, 사용자 알림은 `noStore()`를 사용하여 항상 최신 데이터를 가져오도록 했어요.<br />
이렇게 하면 각 데이터의 특성에 맞는 캐싱 전략을 유연하게 적용할 수 있답니다.

### 1️⃣ `fetch` 옵션을 활용한 세밀한 제어: `tags`와 `revalidateTag`

Next.js 15에서는 `fetch` 요청에 `tags` 옵션을 추가하여 더욱 세밀한 캐시 재검증이 가능해졌어요.<br />
`tags`를 사용하면 특정 태그가 지정된 `fetch` 캐시만 선택적으로 무효화할 수 있답니다.<br />
이는 특히 Server Actions와 함께 사용할 때 강력한 시너지를 발휘해요.

```tsx showLineNumbers
// app/products/actions.ts (Server Action)
'use server';

import { revalidateTag } from 'next/cache';

interface Product {
  id: string;
  name: string;
  price: number;
}

export async function addProduct(productData: Omit<Product, 'id'>) {
  // 데이터베이스에 새로운 제품을 추가하는 로직
  const newProduct = await fetch('https://api.example.com/products', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(productData),
  }).then(res => res.json());

  // 'products' 태그가 지정된 모든 fetch 캐시를 재검증해요.
  // 이로 인해 products 목록을 가져오는 페이지가 업데이트된 데이터를 보여주게 된답니다.
  revalidateTag('products');

  return newProduct;
}

// app/products/page.tsx

async function getProducts(): Promise<Product[]> {
  const res = await fetch('https://api.example.com/products', {
    // 이 fetch 요청에 'products' 태그를 지정했어요.
    next: { tags: ['products'] }, 
  });
  if (!res.ok) throw new Error('Failed to fetch products');
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts();

  return (
    <div>
      <h1>제품 목록</h1>
      <ul>
        {products.map(product => (
          <li key={product.id}>{product.name} - {product.price}원</li>
        ))}
      </ul>
      {/* 제품 추가 폼 (Server Action 호출) */}
    </div>
  );
}
```

<Blockquote type="tip">
`revalidateTag()`는 특정 `tags`를 가진 `fetch` 캐시만 무효화하므로, 전체 라우트를 재검증하는 `revalidatePath()`보다 훨씬 효율적이고 정교한 캐시 관리가 가능해요.<br />
특히 데이터 변경이 빈번한 부분만 업데이트해야 할 때 유용하게 사용할 수 있답니다.
</Blockquote>

## 📝 정리하며: 최적의 성능을 위한 캐싱 전략

Next.js 15는 `fetch` API를 통한 캐싱과 재검증 기능을 더욱 강력하게 제공하며, 개발자가 웹 애플리케이션의 성능을 최적화하고 데이터를 효율적으로 관리할 수 있도록 돕고 있어요.<br />

### 0️⃣ 핵심 요약

*   **`fetch` 기본 캐싱:** Next.js `fetch`는 기본적으로 응답을 캐시하며, `cache: 'force-cache'`와 동일하게 동작해요.<br />
*   **`cache: 'no-store'`:** 실시간 데이터나 캐시가 필요 없는 데이터는 이 옵션을 사용하여 항상 최신 데이터를 가져오세요.<br />
*   **`revalidate: number`:** ISR과 유사하게 특정 시간 간격으로 데이터를 재검증하여 신선도를 유지할 수 있어요.<br />
*   **`export const revalidate`:** 라우트 세그먼트 전체에 대한 기본 캐싱 정책을 설정할 수 있으며, 개별 `fetch` 옵션이 우선해요.<br />
*   **`noStore()`:** `next/cache`에서 임포트하여 현재 요청을 캐시하지 않도록 강제할 때 사용해요.<br />
*   **`tags`와 `revalidateTag()`:** `fetch` 요청에 `tags`를 부여하고, `revalidateTag()`를 통해 특정 태그의 캐시만 선택적으로 무효화하여 정교한 캐시 관리를 할 수 있어요.

### 1️⃣ 다음 액션

이러한 캐싱 전략들을 여러분의 Next.js 15 프로젝트에 적극적으로 적용해 보세요.<br />
데이터의 특성과 변경 주기를 고려하여 적절한 `cache` 및 `revalidate` 옵션을 선택하고, `noStore()`와 `revalidateTag()`를 활용하여 더욱 유연하고 효율적인 데이터 관리 시스템을 구축하는 것이 중요해요.<br />
성능 모니터링 도구를 사용하여 캐싱 전략 변경 전후의 성능 변화를 측정해 보는 것도 좋은 방법이 될 거예요.
