## 🤔 왜 컴포넌트 테스트가 중요할까요?

### 0️⃣ 테스트, 선택이 아닌 필수예요
복잡한 웹 애플리케이션을 개발하다 보면 예상치 못한 버그가 발생하거나, 새로운 기능을 추가했을 때 기존 기능이 오작동하는 경우가 빈번하게 발생해요. 이러한 문제를 사전에 방지하고, 코드의 신뢰성을 높이며, 장기적으로 유지보수 비용을 절감하기 위해선 테스트가 필수적이에요.<br />
특히 UI 컴포넌트는 사용자와 직접 상호작용하는 부분이므로, 예상대로 동작하는지 확인하는 것이 매우 중요하답니다.

### 1️⃣ Vitest와 React Testing Library의 조합
기존에는 Jest와 Enzyme 또는 Jest와 RTL 조합이 많이 사용되었지만, 최근에는 Vite 생태계의 성장에 힘입어 Vitest가 큰 주목을 받고 있어요. Vitest는 Jest와 유사한 API를 제공하면서도 더 빠른 테스트 실행 속도와 Vite 개발 서버와의 통합이라는 장점을 가지고 있어요.<br />
여기에 React Testing Library(RTL)는 사용자 관점에서 컴포넌트가 어떻게 동작하는지를 테스트하는 데 초점을 맞춰, 구현 디테일보다는 실제 사용자 경험을 검증하는 데 매우 효과적이에요. 이 두 도구의 조합은 Next.js 프로젝트에서 효율적이고 견고한 컴포넌트 테스트 환경을 구축하는 데 아주 좋은 선택이 될 거예요.

## ⚙️ Vitest와 React Testing Library 설정하기

Next.js 프로젝트에 Vitest와 RTL을 설정하는 방법을 단계별로 알아볼게요.

### 0️⃣ 필요한 패키지 설치
먼저, 프로젝트에 필요한 패키지들을 설치해 주세요. `jsdom`은 브라우저 환경을 시뮬레이션하여 DOM을 테스트할 수 있게 해주고, `@testing-library/jest-dom`은 Jest 스타일의 매처(matcher)를 RTL에서 사용할 수 있도록 확장해 줘요.

```bash showLineNumbers
npm install -D vitest @vitest/ui jsdom @testing-library/react @testing-library/jest-dom @testing-library/user-event @vitejs/plugin-react
# 또는 yarn add -D vitest @vitest/ui jsdom @testing-library/react @testing-library/jest-dom @testing-library/user-event @vitejs/plugin-react
```

*   `vitest`: 테스트 러너<br />
*   `@vitest/ui`: Vitest 웹 UI (선택 사항이지만 편리해요)<br />
*   `jsdom`: DOM 환경 시뮬레이션 (브라우저 환경에서 테스트하는 것처럼)<br />
*   `@testing-library/react`: React 컴포넌트 테스트를 위한 RTL 핵심 라이브러리<br />
*   `@testing-library/jest-dom`: Jest-dom 매처 확장 (예: `toBeInTheDocument()`)
*   `@testing-library/user-event`: 사용자 이벤트를 시뮬레이션 (더 실제와 유사하게)<br />
*   `@vitejs/plugin-react`: Vite가 React 코드를 처리할 수 있도록 해주는 플러그인

### 1️⃣ `vitest.config.ts` 설정
프로젝트 루트에 `vitest.config.ts` 파일을 생성하고 다음과 같이 설정해 주세요. Next.js는 기본적으로 React 17+ JSX 변환을 사용하므로, `@vitejs/plugin-react` 플러그인을 추가해야 해요.

```tsx showLineNumbers
/// <reference types="vitest" />
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom', // 브라우저 환경을 시뮬레이션하기 위해 jsdom 사용
    globals: true, // describe, it, expect 등을 전역으로 사용
    setupFiles: ['./vitest.setup.ts'], // 테스트 실행 전 설정 파일
    css: false, // CSS 파일 import 시 에러 방지
    coverage: {
      provider: 'v8', // 코드 커버리지 리포트 생성
      reporter: ['text', 'json', 'html'],
      exclude: ['node_modules/', '**/e2e/**'],
    },
    // Next.js의 alias 설정과 일치하도록 경로 설정
    alias: {
      '@/components': './components',
      '@/lib': './lib',
      '@/app': './app', // App Router 기준
      // 필요한 다른 alias 추가
    },
  },
});
```

테스트 환경이 준비될 때마다 실행될 설정 파일이에요. 여기서는 `@testing-library/jest-dom`의 확장 매처를 임포트하여 전역으로 사용할 수 있게 해요.

## 🧪 실제 컴포넌트 테스트 예시

이제 실제로 컴포넌트를 만들고 테스트 코드를 작성해 볼까요?

### 0️⃣ 간단한 `Button` 컴포넌트 테스트
사용자에게 메시지를 보여주는 간단한 버튼 컴포넌트를 만들어 볼게요.

`components/Button.tsx`:

```tsx showLineNumbers
import React from 'react';

interface ButtonProps {
  children: React.ReactNode;
  onClick?: () => void;
}

export default function Button({ children, onClick }: ButtonProps) {
  return (
    <button type="button" onClick={onClick}>
      {children}
    </button>
  );
}
```

`components/Button.test.tsx`:

```tsx showLineNumbers
import { render, screen, fireEvent } from '@testing-library/react';
import { describe, it, expect, vi } from 'vitest';
import Button from './Button';

describe('Button', () => {
  it('버튼 텍스트를 올바르게 렌더링해야 해요.', () => {
    render(<Button>클릭하세요</Button>);
    // screen.getByRole을 사용하여 접근성(accessibility)을 고려한 쿼리를 사용하는 것이 좋아요.
    expect(screen.getByRole('button', { name: '클릭하세요' })).toBeInTheDocument();
  });

  it('클릭 이벤트가 올바르게 호출되어야 해요.', () => {
    const handleClick = vi.fn(); // Vitest의 Mock 함수 생성
    render(<Button onClick={handleClick}>클릭하세요</Button>);

    const button = screen.getByRole('button', { name: '클릭하세요' });
    fireEvent.click(button); // 버튼 클릭 이벤트 시뮬레이션

    expect(handleClick).toHaveBeenCalledTimes(1); // Mock 함수가 한 번 호출되었는지 확인
  });

  it('disabled 상태일 때 클릭 이벤트가 호출되지 않아야 해요.', () => {
    const handleClick = vi.fn();
    // disabled prop은 Button 컴포넌트에 정의되어 있지 않으므로, 예시를 위해 추가한다고 가정해요.
    // 실제 코드에서는 Button 컴포넌트에 disabled prop을 추가해야 해요.
    // render(<Button onClick={handleClick} disabled>클릭하세요</Button>);

    // 현재 Button 컴포넌트에는 disabled prop이 없으므로, 해당 테스트는 주석 처리하거나 Button 컴포넌트를 수정해야 해요.
    // 하지만 일반적인 버튼 컴포넌트라면 disabled 속성을 가질 것이므로, 이렇게 테스트할 수 있다는 것을 보여드려요.
    // const button = screen.getByRole('button', { name: '클릭하세요' });
    // fireEvent.click(button);
    // expect(handleClick).not.toHaveBeenCalled();

    // disabled 상태를 테스트하려면, Button 컴포넌트에 disabled prop을 추가하고 아래와 같이 테스트할 수 있어요.
    // 예를 들어, Button 컴포넌트를 다음과 같이 수정한다고 가정해요.
    // function Button({ children, onClick, disabled }: ButtonProps) { return (<button type="button" onClick={onClick} disabled={disabled}>{children}</button>); }
    const DisabledButton = ({ onClick }: { onClick: () => void }) => (
      <button type="button" onClick={onClick} disabled>클M</button>
    );
    render(<DisabledButton onClick={handleClick} />);
    const button = screen.getByRole('button', { name: '클M' });
    fireEvent.click(button);
    expect(handleClick).not.toHaveBeenCalled();
  });
});
```

<Blockquote type="tip">
React Testing Library는 사용자 관점에서 UI를 테스트하는 데 중점을 둬요. 따라서 `screen.getByText`, `screen.getByRole` 등과 같이 사용자가 실제로 보는 텍스트나 역할(role)을 기반으로 요소를 쿼리하는 것을 권장해요. `data-testid`는 최후의 수단으로 사용하는 것이 좋아요.
</Blockquote>

### 1️⃣ `userEvent`를 사용한 상호작용 테스트
`fireEvent`는 DOM 이벤트를 직접 트리거하지만, `userEvent`는 실제 사용자의 상호작용을 더 가깝게 시뮬레이션해요. 예를 들어, `userEvent.click`은 `mousedown`, `mouseup`, `click`과 같은 일련의 이벤트를 발생시켜요. 이는 더 현실적인 테스트를 가능하게 해줘요.

`components/Counter.tsx`:

```tsx showLineNumbers
import React, { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);

  const increment = () => setCount((prev) => prev + 1);
  const decrement = () => setCount((prev) => prev - 1);

  return (
    <div>
      <h1 data-testid="count">Count: {count}</h1>
      <button type="button" onClick={increment}>Increment</button>
      <button type="button" onClick={decrement}>Decrement</button>
    </div>
  );
}
```

`components/Counter.test.tsx`:

```tsx showLineNumbers
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { describe, it, expect } from 'vitest';
import Counter from './Counter';

describe('Counter', () => {
  it('초기 카운트 값을 0으로 렌더링해야 해요.', () => {
    render(<Counter />);
    expect(screen.getByTestId('count')).toHaveTextContent('Count: 0');
  });

  it('Increment 버튼 클릭 시 카운트가 증가해야 해요.', async () => {
    render(<Counter />);
    const incrementButton = screen.getByRole('button', { name: 'Increment' });
    await userEvent.click(incrementButton);
    expect(screen.getByTestId('count')).toHaveTextContent('Count: 1');
  });

  it('Decrement 버튼 클릭 시 카운트가 감소해야 해요.', async () => {
    render(<Counter />);
    const decrementButton = screen.getByRole('button', { name: 'Decrement' });
    await userEvent.click(decrementButton);
    expect(screen.getByTestId('count')).toHaveTextContent('Count: -1');
  });

  it('Increment 후 Decrement 시 카운트가 올바르게 변경되어야 해요.', async () => {
    render(<Counter />);
    const incrementButton = screen.getByRole('button', { name: 'Increment' });
    const decrementButton = screen.getByRole('button', { name: 'Decrement' });

    await userEvent.click(incrementButton);
    await userEvent.click(incrementButton);
    await userEvent.click(decrementButton);

    expect(screen.getByTestId('count')).toHaveTextContent('Count: 1');
  });
});
```

<Blockquote type="info">
`userEvent`를 사용할 때는 대부분의 메서드가 비동기적으로 동작하므로 `await`를 붙여주는 것이 중요해요. 이는 실제 브라우저 환경에서 이벤트가 처리되는 방식을 더 잘 모방하기 위함이에요.
</Blockquote>

### 2️⃣ 모듈 Mocking 예시 (Next.js `useRouter`)
Next.js 애플리케이션에서는 `next/router`나 API 호출과 같은 외부 의존성을 가진 컴포넌트가 많아요. 이러한 의존성을 격리하여 컴포넌트 자체의 로직만 테스트하기 위해 Mocking이 필요해요.

`components/PostCard.tsx`:

```tsx showLineNumbers
'use client';

import { useRouter } from 'next/navigation'; // App Router 기준
import React from 'react';

interface PostCardProps {
  id: string;
  title: string;
  content: string;
}

export default function PostCard({ id, title, content }: PostCardProps) {
  const router = useRouter();

  const handleClick = () => {
    router.push(`/posts/${id}`);
  };

  return (
    <div style={{ border: '1px solid #ccc', padding: '16px', margin: '8px' }}>
      <h3>{title}</h3>
      <p>{content}</p>
      <button type="button" onClick={handleClick}>View Post</button>
    </div>
  );
}
```

`components/PostCard.test.tsx`:

```tsx showLineNumbers
import { render, screen, fireEvent } from '@testing-library/react';
import { describe, it, expect, vi } from 'vitest';
import PostCard from './PostCard';

// next/navigation 모듈을 Mocking해요.
// useRouter 훅이 호출될 때 반환될 값을 정의해요.
vi.mock('next/navigation', () => ({
  useRouter: vi.fn(() => ({
    push: vi.fn(), // router.push 함수를 Mocking해요.
    replace: vi.fn(),
    prefetch: vi.fn(),
    // 필요한 다른 라우터 메서드들도 Mocking할 수 있어요.
  })),
}));

describe('PostCard', () => {
  it('게시글 정보를 올바르게 렌더링해야 해요.', () => {
    render(
      <PostCard
        id="1"
        title="첫 번째 게시글"
        content="이것은 첫 번째 게시글의 내용입니다."
      />
    );

    expect(screen.getByRole('heading', { name: '첫 번째 게시글' })).toBeInTheDocument();
    expect(screen.getByText('이것은 첫 번째 게시글의 내용입니다.')).toBeInTheDocument();
    expect(screen.getByRole('button', { name: 'View Post' })).toBeInTheDocument();
  });

  it('View Post 버튼 클릭 시 올바른 경로로 이동해야 해요.', () => {
    const mockPush = vi.fn();
    // useRouter의 push 함수를 Mocking한 후, 해당 Mock 함수를 가져와요.
    // 이렇게 하면 테스트에서 mockPush가 호출되었는지 여부를 검증할 수 있어요.
    vi.mock('next/navigation', async (importOriginal) => {
      const actual = await importOriginal();
      return {
        ...actual,
        useRouter: vi.fn(() => ({
          push: mockPush,
          replace: vi.fn(),
          prefetch: vi.fn(),
        })),
      };
    });

    render(
      <PostCard
        id="2"
        title="두 번째 게시글"
        content="두 번째 게시글 내용입니다."
      />
    );

    const viewPostButton = screen.getByRole('button', { name: 'View Post' });
    fireEvent.click(viewPostButton);

    expect(mockPush).toHaveBeenCalledTimes(1);
    expect(mockPush).toHaveBeenCalledWith('/posts/2'); // 올바른 경로로 호출되었는지 확인
  });
});
```

<Blockquote type="warning">
`vi.mock`는 파일의 최상단에서 선언되어야 하며, 테스트 파일 전체에 걸쳐 Mocking이 적용돼요. 만약 각 테스트 케이스마다 다른 Mocking이 필요하다면, `vi.doMock` 또는 `vi.unmock`과 같은 기능을 활용하거나, Mocking할 대상을 컴포넌트 외부로 분리하는 것을 고려해 보세요.
</Blockquote>

## 📝 정리

### 0️⃣ 핵심 요약
오늘은 Next.js 프로젝트에서 Vitest와 React Testing Library를 활용하여 UI 컴포넌트를 테스트하는 방법을 살펴보았어요. 주요 내용은 다음과 같아요.<br />

*   **Vitest**: Jest와 유사한 API를 제공하며, 빠른 실행 속도와 Vite 통합이 강점이에요.<br />
*   **React Testing Library (RTL)**: 사용자 관점에서 컴포넌트의 동작을 테스트하는 데 초점을 맞춰요.<br />
*   **설정**: `vitest.config.ts`와 `vitest.setup.ts` 파일을 통해 테스트 환경을 구축해요.<br />
*   **기본 테스트**: `render`, `screen.getByRole`, `expect().toBeInTheDocument()` 등으로 컴포넌트 렌더링을 확인해요.<br />
*   **이벤트 처리**: `fireEvent` 또는 `userEvent`를 사용하여 사용자 상호작용을 시뮬레이션하고, Mock 함수로 콜백 호출 여부를 검증해요.<br />
*   **Mocking**: `vi.mock`를 사용하여 `next/navigation`과 같은 외부 의존성을 격리하고 테스트해요.
