## 🤔 문제/배경

### 0️⃣ JavaScript는 왜 비동기 처리가 필요할까요?

JavaScript는 기본적으로 **싱글 스레드(Single Thread)** 언어예요.<br />
즉, 한 번에 하나의 작업만 처리할 수 있다는 의미인데요. 만약 네트워크 요청이나 파일 I/O처럼 시간이 오래 걸리는 작업을 동기적으로 처리한다면 어떻게 될까요? 해당 작업이 완료될 때까지 다른 모든 작업이 멈추고, 웹 페이지는 먹통이 되어 사용자 경험이 매우 나빠질 거예요.<br />
이러한 문제를 해결하기 위해 JavaScript는 **비동기 처리** 메커니즘을 사용해요. 비동기 처리를 통해 시간이 오래 걸리는 작업을 백그라운드에 맡겨두고, 그동안 다른 작업을 계속 처리할 수 있게 되는 거죠.

### 1️⃣ 비동기 코드, 왜 예상과 다르게 동작할까요?

`setTimeout`, `Promise`, `fetch` 등 비동기 함수를 사용하다 보면, 우리가 예상했던 순서와 다르게 코드가 실행되는 경험을 자주 하실 거예요.<br />
예를 들어, 아래와 같은 코드를 실행했을 때 어떤 결과가 나올 거라고 생각하시나요?

```javascript showLineNumbers
console.log("1. 시작");

setTimeout(() => {
  console.log("2. setTimeout 콜백");
}, 0); // 지연 시간 0ms

Promise.resolve().then(() => {
  console.log("3. Promise 콜백 (마이크로태스크)");
});

console.log("4. 끝");
```

많은 분들이 `1 -> 2 -> 3 -> 4` 또는 `1 -> 4 -> 2 -> 3` 처럼 예상하실 수도 있지만, 실제 결과는 조금 다를 수 있어요.<br />
이처럼 비동기 코드의 실행 흐름을 정확히 예측하기 위해서는 JavaScript 런타임 환경, 특히 **이벤트 루프**의 동작 원리를 명확히 이해해야 해요.

## ⚙️ 이벤트 루프 동작 원리

JavaScript 런타임 환경은 단순히 JavaScript 엔진만으로 구성되어 있지 않아요. 브라우저나 Node.js 같은 환경은 JavaScript 엔진 외에도 다양한 구성 요소들을 포함하고 있으며, 이벤트 루프는 이들 사이의 조율자 역할을 해요.

### 0️⃣ 핵심 구성 요소

이벤트 루프를 이해하기 위해 다음 핵심 구성 요소들을 알아볼게요.

- **콜 스택(Call Stack)**: JavaScript 코드가 실행될 때 함수 호출을 기록하는 스택 구조예요. 가장 위에 있는 함수가 먼저 실행되고 완료되면 스택에서 제거돼요. JavaScript는 싱글 스레드이므로 콜 스택도 하나뿐이에요.
- **힙(Heap)**: 객체나 변수 같은 데이터가 저장되는 메모리 공간이에요.
- **웹 API(Web APIs) / Node.js API**: 브라우저(DOM, `setTimeout`, `fetch`, `localStorage` 등)나 Node.js(파일 시스템, 네트워크 등)에서 제공하는 비동기 기능을 담당하는 환경이에요. JavaScript 엔진의 외부에서 동작하죠.
- **태스크 큐(Task Queue / Callback Queue / Macrotask Queue)**: 웹 API에서 처리된 비동기 작업의 콜백 함수들이 대기하는 큐예요. `setTimeout`, `setInterval`, I/O, UI 렌더링 등이 여기에 해당해요.
- **마이크로태스크 큐(Microtask Queue)**: `Promise.then()`, `Promise.catch()`, `Promise.finally()`, `queueMicrotask`, `MutationObserver` 같은 작업들의 콜백 함수가 대기하는 큐예요. 태스크 큐보다 우선순위가 높아요.
- **이벤트 루프(Event Loop)**: 콜 스택이 비어있는지 지속적으로 확인하고, 비어있다면 마이크로태스크 큐에 있는 콜백을 먼저 가져와 콜 스택으로 푸시해요. 마이크로태스크 큐도 비어있다면, 태스크 큐에서 콜백을 가져와 콜 스택으로 푸시하는 역할을 해요.

<Blockquote type="info">
  정리하자면, JavaScript 엔진은 콜 스택과 힙을 가지고 JavaScript 코드를
  실행해요.
  <br />
  비동기 작업은 웹 API로 넘어가 처리되고, 완료되면 콜백 함수는 태스크 큐 또는
  마이크로태스크 큐로 들어가 대기해요.
  <br />
  이벤트 루프는 콜 스택이 비었을 때 이 큐들을 감시하며 콜백을 콜 스택으로 옮겨
  실행시키는 역할을 하는 거죠.
</Blockquote>

### 1️⃣ 콜 스택과 웹 API의 만남

일반적인 동기 코드는 콜 스택에 쌓이고 순서대로 실행돼요.<br />
하지만 `setTimeout`이나 `fetch` 같은 비동기 함수가 호출되면, 이 함수 자체는 콜 스택에 들어가지만, 실제 비동기 작업(타이머 설정, 네트워크 요청 등)은 **웹 API**로 위임돼요.<br />
웹 API는 해당 작업을 백그라운드에서 처리하고, 작업이 완료되면 미리 등록해 둔 **콜백 함수**를 태스크 큐 또는 마이크로태스크 큐에 넣어줘요.

### 2️⃣ 태스크 큐(Task Queue)와 마이크로태스크 큐(Microtask Queue)의 차이

이 두 큐의 우선순위가 이벤트 루프의 동작에 있어 매우 중요해요.

- **태스크 큐 (Macrotask Queue)**:
  - `setTimeout`, `setInterval`, UI 렌더링, I/O 작업(Node.js의 `fs.readFile` 등) 콜백 등이 여기에 들어가요.
  - 이벤트 루프는 콜 스택이 완전히 비고, **마이크로태스크 큐도 완전히 비었을 때** 비로소 태스크 큐에서 하나의 콜백을 가져와 실행해요.
- **마이크로태스크 큐 (Microtask Queue)**:
  - `Promise.then()`, `Promise.catch()`, `Promise.finally()`, `async/await`(내부적으로 Promise 사용), `queueMicrotask`, `MutationObserver` 콜백 등이 여기에 들어가요.
  - 이벤트 루프는 콜 스택이 비어있으면, **태스크 큐보다 먼저 마이크로태스크 큐에 있는 모든 콜백을 처리해요.** 마이크로태스크 큐가 비어있지 않는 한, 태스크 큐의 콜백은 실행되지 않아요.

이러한 우선순위 때문에 `Promise` 관련 콜백이 `setTimeout` 콜백보다 먼저 실행되는 경우가 많아요.

### 3️⃣ 이벤트 루프의 역할

이벤트 루프는 말 그대로 끊임없이 반복(loop)하면서 다음 세 가지를 확인하고 실행해요.

1.  **콜 스택이 비어있는지 확인해요.** (현재 실행 중인 JavaScript 코드가 없는지)
2.  **마이크로태스크 큐가 비어있지 않다면, 마이크로태스크 큐의 모든 콜백을 콜 스택으로 옮겨 실행해요.**
3.  **마이크로태스크 큐가 비어있다면, 태스크 큐에서 가장 오래된(FIFO) 하나의 콜백을 콜 스택으로 옮겨 실행해요.**
4.  이 과정을 반복해요.

<Blockquote type="warning">
  마이크로태스크 큐의 콜백이 또 다른 마이크로태스크를 생성한다면, 이 새로운
  마이크로태스크도 현재의 이벤트 루프 틱(tick) 내에서 처리돼요. 이 때문에 무한
  루프에 빠지면 UI 렌더링이나 다른 태스크 큐 작업이 영원히 실행되지 않을 수도
  있으니 주의해야 해요.
</Blockquote>

## 🧪 예시로 이해하는 비동기 처리

이제 실제 코드를 통해 이벤트 루프의 동작 방식을 더 자세히 알아볼게요.

### 0️⃣ `setTimeout`과 `Promise`의 실행 순서

아까 위에서 보았던 예시를 다시 한번 살펴볼게요.

```javascript showLineNumbers
console.log("1. 동기 코드 시작"); // ①

setTimeout(() => {
  console.log("4. setTimeout 콜백 (매크로태스크)"); // ④
}, 0);

Promise.resolve().then(() => {
  console.log("3. Promise 콜백 (마이크로태스크)"); // ③
});

console.log("2. 동기 코드 끝"); // ②
```

이 코드의 실행 흐름은 다음과 같아요.

1.  `console.log('1. 동기 코드 시작');` 실행 후 콜 스택에서 제거돼요. (출력: `1. 동기 코드 시작`)
2.  `setTimeout` 함수가 호출되면, 타이머 설정 작업이 웹 API로 넘어가요. 콜백 함수는 0ms 후에 **태스크 큐**에 들어갈 준비를 해요.
3.  `Promise.resolve().then()`이 호출되면, `Promise`는 즉시 해결(resolved)되고 `then` 내부의 콜백 함수는 **마이크로태스크 큐**로 들어가요.
4.  `console.log('2. 동기 코드 끝');` 실행 후 콜 스택에서 제거돼요. (출력: `2. 동기 코드 끝`)
5.  이 시점에서 콜 스택이 비었어요. 이벤트 루프가 작동해요.
6.  이벤트 루프는 먼저 **마이크로태스크 큐**를 확인하고, `Promise` 콜백을 발견해요. 이 콜백을 콜 스택으로 옮겨 실행해요. (출력: `3. Promise 콜백 (마이크로태스크)`)
7.  `Promise` 콜백이 실행 완료되어 콜 스택에서 제거돼요.
8.  마이크로태스크 큐가 비었어요. 이벤트 루프는 이제 **태스크 큐**를 확인하고, `setTimeout` 콜백을 발견해요. 이 콜백을 콜 스택으로 옮겨 실행해요. (출력: `4. setTimeout 콜백 (매크로태스크)`)
9.  `setTimeout` 콜백이 실행 완료되어 콜 스택에서 제거돼요.
10. 콜 스택, 마이크로태스크 큐, 태스크 큐 모두 비어있으므로 모든 작업이 종료돼요.

<Blockquote type="success">
  따라서 예상 출력은 `1. 동기 코드 시작`, `2. 동기 코드 끝`, `3. Promise 콜백
  (마이크로태스크)`, `4. setTimeout 콜백 (매크로태스크)` 순서가 돼요.
</Blockquote>

### 1️⃣ 복잡한 비동기 시나리오 분석

이번에는 `async/await`과 함께 조금 더 복잡한 시나리오를 살펴볼게요.

```javascript showLineNumbers
console.log("A"); // ①

async function asyncFunc() {
  console.log("B"); // ②
  await Promise.resolve();
  console.log("C"); // ⑤ (await 이후는 마이크로태스크)
}

asyncFunc();

setTimeout(() => {
  console.log("D"); // ⑥ (매크로태스크)
}, 0);

Promise.resolve().then(() => {
  console.log("E"); // ④ (마이크로태스크)
});

console.log("F"); // ③
```

이 코드의 실행 흐름을 단계별로 분석해 볼까요?

1.  `console.log('A');` 실행. (출력: `A`)
2.  `asyncFunc()` 호출. 함수 내부로 진입해요.
3.  `console.log('B');` 실행. (출력: `B`)
4.  `await Promise.resolve();`를 만나면, `asyncFunc`의 나머지 부분(`console.log('C');`)은 마이크로태스크 큐로 보내지고, `asyncFunc`는 일시 중단돼요.
5.  `setTimeout` 함수 호출. 콜백은 웹 API로 넘어가고, 0ms 후에 **태스크 큐**로 들어갈 준비를 해요.
6.  `Promise.resolve().then()` 호출. 콜백은 **마이크로태스크 큐**로 들어가요.
7.  `console.log('F');` 실행. (출력: `F`)
8.  콜 스택이 비었어요. 이벤트 루프가 작동해요.
9.  이벤트 루프는 **마이크로태스크 큐**를 확인해요. 두 개의 마이크로태스크가 있어요 (`asyncFunc`의 `console.log('C')` 부분과 `Promise.resolve().then()` 콜백).
10. 먼저 `asyncFunc`의 다음 부분인 `console.log('C');`가 콜 스택으로 옮겨져 실행돼요. (출력: `C`)
11. 다음으로 `Promise.resolve().then()` 콜백인 `console.log('E');`가 콜 스택으로 옮겨져 실행돼요. (출력: `E`)
12. 마이크로태스크 큐가 비었어요.
13. 이벤트 루프는 **태스크 큐**를 확인해요. `setTimeout` 콜백인 `console.log('D');`가 있어요. 이를 콜 스택으로 옮겨 실행돼요. (출력: `D`)
14. 모든 작업이 종료돼요.

<Blockquote type="success">
  최종 출력 순서는 `A`, `B`, `F`, `C`, `E`, `D`가 될 거예요.
</Blockquote>

## 📝 실무 적용 팁

이벤트 루프를 이해하는 것은 단순히 개념을 아는 것을 넘어, 실제 애플리케이션 개발에 큰 도움이 돼요.

### 0️⃣ 비동기 코드 디버깅 전략

복잡한 비동기 코드의 실행 흐름을 파악하기 어려울 때는 개발자 도구의 성능 탭이나 콜 스택 시각화 기능을 활용해 보세요.<br />
`console.log`를 적절히 배치하여 각 콜백이 언제 실행되는지 추적하는 것도 좋은 방법이에요. 특히 마이크로태스크와 매크로태스크의 우선순위를 염두에 두고 로그를 찍어보면 예상치 못한 동작의 원인을 쉽게 찾을 수 있어요.

```javascript showLineNumbers
console.log("Sync 1");

setTimeout(() => {
  console.log("Timeout 1 (Macrotask)");
}, 0);

Promise.resolve().then(() => {
  console.log("Promise 1 (Microtask)");
});

setTimeout(() => {
  console.log("Timeout 2 (Macrotask)");
}, 0);

Promise.resolve().then(() => {
  console.log("Promise 2 (Microtask)");
});

console.log("Sync 2");
```

위 코드의 출력 순서를 예측해보고, 직접 실행해 보면서 이벤트 루프의 동작을 시뮬레이션해 보는 연습을 해보시는 것을 추천해요.<br />
이런 과정을 통해 비동기 코드에 대한 직관을 기를 수 있어요.

### 1️⃣ 성능 최적화를 위한 이벤트 루프 이해

이벤트 루프의 동작 원리를 이해하면 웹 애플리케이션의 성능을 최적화하는 데 큰 도움이 돼요.

- **UI 블로킹 방지**: 시간이 오래 걸리는 계산 작업은 `setTimeout`을 이용해 작은 단위로 쪼개거나, `Web Workers`를 사용하여 메인 스레드와 분리하면 UI가 멈추는 현상을 방지할 수 있어요.
- **효율적인 비동기 패턴 사용**: `Promise`와 `async/await`은 콜백 지옥을 피하고 코드를 더욱 가독성 있게 만들어 줄 뿐만 아니라, 마이크로태스크 큐의 우선순위를 활용하여 중요한 작업을 빠르게 처리할 수 있게 해줘요. 불필요한 `setTimeout(..., 0)` 사용은 자제하고, `Promise`나 `queueMicrotask`를 활용하는 것이 더 효율적일 때가 많아요.
- **렌더링 최적화**: 브라우저의 렌더링은 매크로태스크의 한 종류예요. 따라서 무거운 JavaScript 작업이 마이크로태스크 큐에 너무 많이 쌓이면 렌더링이 지연될 수 있어요. 중요한 UI 업데이트는 다음 이벤트 루프 틱에서 처리되도록 `requestAnimationFrame`을 활용하는 것도 좋은 전략이에요.

## 📝 정리

오늘은 JavaScript의 핵심 비동기 처리 메커니즘인 **이벤트 루프**에 대해 자세히 알아보았어요.<br />
콜 스택, 웹 API, 태스크 큐, 마이크로태스크 큐, 그리고 이 모든 것을 조율하는 이벤트 루프의 역할을 이해하는 것은 견고하고 성능 좋은 JavaScript 애플리케이션을 만드는 데 필수적이에요.

<Blockquote type="info">
  **핵심 요약:**
  <br />
  1. JavaScript는 싱글 스레드 언어이므로 비동기 처리가 필수적이에요.
  <br />
  2. 이벤트 루프는 콜 스택이 비었을 때 마이크로태스크 큐의 콜백을 먼저 처리하고,
  그 다음 태스크 큐의 콜백을 처리해요.
  <br />
  3. `Promise` 관련 콜백은 마이크로태스크 큐에, `setTimeout`, `setInterval` 관련
  콜백은 태스크 큐에 들어가요.
  <br />
  4. 이벤트 루프의 동작 방식을 이해하면 비동기 코드의 실행 순서를 정확히
  예측하고, 성능을 최적화할 수 있어요.
</Blockquote>
